## 2. 主要优点

### 2.1 边界清晰

- 明确网站是**观察者**而非决策者（§3.3 非目标）。
- 区分引擎 `trades` 与 `trade/tools/optimal_trade_points` 事后标注，避免研究层与实盘层混淆——这一点对自动交易系统长期演进非常重要。

### 2.2 与代码库高度对齐

文档引用的数据资产与代码实际一致：

- `decisions.csv` / `trades.csv` / `equity_curve.csv` 由 `TradeLogger` 产出；
- `prod/vX.Y.Z/MANIFEST.json` 可作为版本注册源；
- `market_data/sources/binance_futures.py` 适合补准实时 K 线；
- `trading_system/metrics.py` 可复用收益/回撤计算逻辑。

### 2.3 架构可扩展

`BackendProfile` + `BackendAdapter` 协议（§7.3）是正确抽象：一期 `FileTailAdapter`、二期 `HttpPushAdapter`，前端不感知版本差异（G6）。

### 2.4 里程碑拆分合理

M1（数据接入）→ M2（Dashboard）→ M3（K 线标注）→ M4（准实时）的递进顺序符合依赖关系，优先交付「能看」再追求「能推」。

### 2.5 与 014 文档一脉相承

功能门类（信号/收益/持仓/风险）与 `document/014` 交易系统设计说明书的「可解释、可复盘」目标一致，不是孤立的新需求。

## 3. 关键问题与矛盾（需修订）

### 3.1 【严重】FileTail 与当前 `TradeLogger` 落盘机制不匹配

文档假设（§7.4 步骤 4、§13 验收 5）：

> FileTailAdapter 扫描 CSV 增量；新 backtest 输出后 1 分钟内可查。

**实际情况**：

- `TradeLogger` 仅在 `flush()` 时**整表覆写** CSV，不在每根 bar 增量写盘；
- `flush()` 目前只在回测结束调用一次（`trading_system/backtest/runner.py:59`）。

```225:231:trading_system/logger.py
    def flush(self) -> None:
        self.out_dir.mkdir(parents=True, exist_ok=True)
        self._write_csv(self.out_dir / "decisions.csv", self.decisions)
        self._write_csv(self.out_dir / "orders.csv", self.orders)
        self._write_csv(self.out_dir / "fills.csv", self.fills)
        self._write_csv(self.out_dir / "trades.csv", self.trades)
        self._write_csv(self.out_dir / "equity_curve.csv", self.equity_curve)
```

**影响**：

- 回测**运行中**无法通过 FileTail 做准实时监控；
- 一期实际只能是 **「回测完成后导入/全量刷新」**，而非文档描述的增量 tail。

**建议修订**：

- 一期明确两种模式：
  - **Archive 模式**：回测结束后 `import_backtest.py` 一次性导入（M1 主路径）；
  - **Live 模式**：需后台改造（周期性 `flush()` 或 `MonitorExporter`），归属 M4 前置条件。
- 将 §13 验收 5 改为「回测 `flush()` 完成后 ≤1 分钟可见」，而非「运行中增量可见」。

---

### 3.2 【严重】「实时」目标与一期能力不一致

| 文档位置 | 指标 |
|----------|------|
| G1（§3.2） | 新 K 线/信号/成交 **≤30s** 可见 |
| M4 验收 | 新 bar 后 **≤30s** 更新 |
| §13 验收 5 | FileTail **1 分钟** |
| §3.3 非目标 | 一期以 **backtest/paper** 为主 |

矛盾点：

1. 当前无 **paper/live 运行进程**，只有回测 CLI；
2. FileTail 即使完善，也无法在回测未 `flush` 前推送；
3. G1 的 30s 指标在 M1~M3 阶段**无法验收**。

**建议修订**：

- G1 拆为两级：
  - **G1a（一期）**：已完成回测/仿真结果导入后，Dashboard 与 K 线标注正确；
  - **G1b（M4+）**：纸交易/定时 bar 任务下，事件 ≤30s 推送。
- 总目标中「统一观测与**控制**面板」（§3.1）建议改为「观测面板」，与控制非目标保持一致。

---

### 3.3 【中等】K 线标注价格口径未定义

`BaseConfig.execute_timing = "next_bar_open"`：决策在 bar 收盘，成交在**下一根开盘价**。

文档 §5.3 标注来源写 `trades.csv` / `fills.csv`，但未说明：

- 开仓标记用 `decisions` 的 `price`（收盘价）还是 `fills` 的成交价（下一根 open）？
- 与 `trading_system/backtest/plotting.py` 对齐时以哪张表为准？

**建议**：在 `004_数据模型.md` 中固定标注口径——**推荐以 `fills`/`trades` 成交价为标注价**，`decisions` 仅作信号/意图层叠加（可选虚线）。

---

### 3.4 【中等】`decisions.csv` 列膨胀未给出 ViewModel

`trading_system/logger.py` 中 `record_decision` 已写入 **100+ 列**（trend/crash/segment/bias/teq/participation 等）。

文档 §12 提到「聚合层输出精简 ViewModel」，但 §5.2 未定义 Dashboard 默认展示字段集。

**建议**：M1 前在 `004_数据模型.md` 列出两档字段：

- **Dashboard 核心字段**（约 15~20 个）：`ts, price, p_up/down, edge, conf, action, reason_code, blocked, position_ratio, state, portfolio_equity` 等；
- **明细页扩展字段**：按 regime 分组（trend/crash/bias/teq）。

---

### 3.5 【中等】Paper 模式尚未存在

`BackendProfile.mode` 含 `backtest | paper | live`，但 `trading_system` 目前仅有 `BacktestExecutionEngine`，文档 014 也标明 paper/live 未实现。

**建议**：一期 `mode` 枚举收敛为 `archive | replay`（历史导入/回放），`paper` 标为 M4 前置依赖项，避免验收时对不存在的能力产生预期。

---

### 3.6 【轻微】多版本「切换」vs「对比」边界模糊

- G5：≥2 个 prod 版本可**切换**查看；
- §5.5：版本**对比**列为二期。

建议在 G5 明确：一期只需单版本视图切换，不做并排对比——否则 M5 一周工期偏紧。

### 3.7 【轻微】单标的假设未写入文档

当前 prod 均为 **BTCUSDT 1h**。Dashboard 有标的选择器，但未说明一期是否只支持单标的、多标的何时扩展。

**建议**：一期写死 `BTCUSDT` + `1h`，标的/周期选择器 UI 可预留但 disabled。

## 4. 技术路径评审

### 4.1 技术选型 — 赞同，有一处需尽早拍板

| 选型 | 评审意见 |
|------|----------|
| FastAPI + Python 3.11 | ✅ 与现有栈一致，可直接 import `metrics` / `market_data` |
| React + TS + Vite | ✅ 合理 |
| SQLite 一期 | ✅ 适合单机内网；需注意 **写（FileTail/import）与读（API）并发**，建议 WAL 模式 |
| WebSocket | ✅ M4 再用；M1~M3 可纯 REST 降低复杂度 |
| lightweight-charts vs ECharts | ⚠️ 应在 `002_架构设计.md` 中二选一；K 线+标注+大量 mark 推荐 **lightweight-charts**（性能更好） |

### 4.2 `BackendAdapter` 接口 — 基本完备，建议补充

当前接口缺：

- `import_snapshot(path) -> ImportResult`：一次性导入已完成回测目录（一期必需）；
- `get_config_summary() -> dict`：从 `trading_rule.json` 读杠杆、仓位上限等展示参数；
- `get_reference_metrics() -> dict`：从 `MANIFEST` 指向的 `metrics/*.json` 读 prod 参考值，供 Dashboard「对照基准」展示。

### 4.3 数据流（§7.4）— 建议重绘一期路径

当前一期更现实的路径：

```text
回测结束 → TradeLogger.flush() → import_backtest.py
    → SQLite → REST API → 前端 Dashboard + K 线图

（M4 以后）
定时 bar 任务 → MonitorExporter → HTTP ingest → WS 推送
```

## 6. 风险补充（文档未覆盖）

| 风险 | 说明 | 建议 |
|------|------|------|
| 回测目录膨胀 | `backtest/` 下大量实验输出，FileTail 扫描成本高 | 仅监听 `backends.yaml` 配置的固定 `data_root` |
| CSV 全量覆写 | 每次 flush 重写整个文件，tail 需按 mtime + 全量 diff | 一期用 import 触发器，不做 inotify tail |
| 时区展示 | 引擎用 UTC，用户可能期望 UTC+8 | API 统一 UTC，前端本地化显示 |
| 权益曲线与交易 PnL 口径 | `equity_curve` 归一化从 1.0 起，与绝对 USDT 盈亏不同 | Dashboard 标明「归一化权益」 |
| 前端与后端分仓 | `website/src/frontend` 独立 Node 工程 | `dev.sh` 需同时起 uvicorn + vite proxy |

## 9. 评审结论

**决议：有条件通过，可进入 M1。**

立项文档在战略方向、架构分层、与 wildrose 现有资产集成方面**达到 M0 评审要求**。主要短板不在「要不要做」，而在**一期对「实时」的预期偏高、对 CSV 落盘机制的假设与代码现实不符**。

建议在启动 M1 前花 **1~2 天** 产出修订版 v0.2（或直接进入 `002_架构设计.md` 吸收上述 P0/P1 修订），尤其明确：

1. **一期 MVP = 回测结果监控 + 多版本归档查看 + K 线交易标注**；
2. **准实时推送 = M4，且依赖 `trading_system` 侧 exporter/周期 flush**。

按此收敛后，文档可作为后续 API 规范与实现的可靠基线。
