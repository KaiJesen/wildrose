## 2. 项目背景

### 2.1 现状

wildrose 交易系统（`trading_system/`）已具备较完整的**回测与规则决策**能力：

- 每根 K 线收盘可完成：信号接入 → 趋势/崩盘/分段/偏置过滤 → 规则决策 → 风控与仓位 → 模拟成交 → 日志落盘。
- `TradeLogger` 可输出 `decisions.csv`、`orders.csv`、`fills.csv`、`trades.csv`、`equity_curve.csv` 等复盘数据。
- **当前缺口**：`TradeLogger` 仅在 `flush()` 时整表覆写 CSV；回测结束才 `flush()`，**无运行中增量导出**；也**无常驻 bar 调度进程**。
- `prod/vX.Y.Z/` 已固化多个策略版本（配置、checkpoint、参考指标、代码快照）。
- `market_data/sources/binance_futures.py` 可通过 REST 拉取 **Binance 永续合约准实时 K 线**（`fapi.binance.com`），满足真实行情接入条件。

### 2.2 痛点

| 痛点 | 说明 |
|------|------|
| 无 Web 界面 | 只能通过 CSV/JSON/PNG 查看结果，无法集中浏览 |
| **无真实实时行情视图** | 无法看到交易所当前 K 线与后台决策的同步关系 |
| **无运行态监控** | 后台是否在跑、最新 bar 是否推进、当前持仓/信号不可见 |
| 无事件流 | 决策与成交无推送/订阅机制 |
| 无聚合视图 | 信号、收益、持仓分散在多张表 |
| 无多版本切换 | 多个 prod 版本缺少统一注册与切换入口 |

### 2.3 建设动机

用户核心诉求：**使用真实实时行情，观察后台交易系统正在运行的情况。**

网站需要实现：

1. **真实实时行情**：从 `binance_futures` 持续拉取 BTCUSDT 等标的 K 线，图表随新 bar 更新；
2. **后台运行态监控**：纸交易/仿真进程每根 bar 收盘后产出信号、决策、成交，网站 **≤30s** 内可见；
3. **门类聚合展示**：信号 / 收益 / 持仓 / 风险 / 版本；
4. **K 线蜡烛图 + 交易标注**（开平仓/持仓/杠杆/收益，成交价口径）；
5. **多版本后台对接**，屏蔽 prod 版本差异；
6. **归档复盘**（辅助）：历史回测结果仍可导入对照。

## 3. 项目目标

### 3.1 总目标

建设 wildrose **交易监控网站**，成为交易系统与研究人员之间的**统一实时观测面板**——在**真实市场行情**下展示后台运行态（信号、持仓、成交、收益），网站本身不参与下单决策。

### 3.2 具体目标

| # | 目标 | 可衡量结果 | 阶段 |
|---|------|------------|------|
| **G1** | **真实实时行情** | K 线来自 `binance_futures`；新 1h bar 在交易所收盘后 **≤60s** 内出现在网站图表（REST 轮询）；Dashboard 显示 `latest_bar_ts` | M1~M2 |
| **G2** | **运行态可见** | 纸交易 bar 任务下，decision/fill/持仓变化在 **≤30s** 内经 WebSocket 推送到前端 | M2~M3 |
| G3 | 周期统计 | 时间窗口（今日 / 7 天 / 30 天 / 自定义）；一期标的 **BTCUSDT**、周期 **1h** | M2 |
| G4 | 门类聚合 | 首页「信号 / 收益 / 持仓 / 风险 / 版本」五大类卡片与趋势 | M2 |
| G5 | K 线标注 | 蜡烛图上叠加开平仓（成交价）、当前持仓、杠杆、单笔收益 | M3 |
| G6 | 多版本切换 | ≥2 个 prod 版本可单视图切换（一期不做并排对比） | M2~M4 |
| G7 | 归档复盘（辅助） | 回测 `flush()` 后导入，≤1 分钟可查历史 trades/decisions | M1 |
| G8 | 可扩展 | 新增后台版本不改前端，仅增适配器与配置 | 全程 |

### 3.3 非目标（本期不做）

- **不替代**交易决策引擎（网站是观察者）
- **不实现**交易所真实账户下单与资金划转（运行态为**纸交易/仿真成交**，行情为真实市场数据）
- **不实现**用户权限与多租户（一期单用户/内网）
- **不实现**移动端原生 App
- **不实现**多版本并排对比（二期）
- **不展示**事后最优交易点（`trade/tools/optimal_trade_points`）为实盘信号

### 3.4 一期范围界定（MVP）

**一期 MVP = 真实实时行情 + 后台运行态监控 + Dashboard + K 线标注**：

| 能力 | 一期 | 说明 |
|------|------|------|
| 真实实时 K 线 | ✅ 必需 | `binance_futures` REST 轮询 + 本地缓存 |
| 纸交易 bar 任务 | ✅ 必需 | `trading_system` 侧 bar 调度器 + 每 bar `on_bar_close` |
| 运行态事件推送 | ✅ 必需 | `MonitorExporter` → 网站 ingestion → WebSocket |
| Dashboard / K 线标注 | ✅ 必需 | |
| 归档导入 | ✅ 辅助 | 历史回测对照、冷启动回填 |
| 多标的 / 多周期 | ❌ 二期 | 一期固定 BTCUSDT / 1h |

**「真实实时」定义（一期）**：

- **行情**：交易所公开市场数据（Binance USDT-M 永续 REST），非合成、非仅 Vision 历史归档；
- **运行**：后台常驻进程按 1h 对齐执行引擎，非仅事后导入 CSV；
- **延迟**：行情 ≤60s（受 REST 轮询与 bar 周期约束）；运行态事件 ≤30s。

## 5. 功能需求

### 5.1 监控总览（Dashboard）

```text
┌─────────────────────────────────────────────────────────────┐
│  [LIVE ●] 版本 [v1.0.0 ▼]  BTCUSDT  1h  最新 bar: 12:00 UTC  │
├──────────────┬──────────────┬──────────────┬─────────────────┤
│  信号概览     │  收益概览     │  持仓概览     │  风险 / 运行态   │
│  p_up/down   │  总收益(归一) │  方向/仓位比  │  最大回撤        │
│  edge/conf   │  今日收益     │  杠杆(配置)  │  最新决策时间     │
│  BLOCK 占比  │  胜率         │  未实现盈亏   │  bar 任务状态    │
├──────────────┴──────────────┴──────────────┴─────────────────┤
│  权益曲线                         最近 N 笔成交（实时追加）       │
├─────────────────────────────────────────────────────────────┤
│  真实实时 K 线（binance_futures）+ 运行态开平仓标注           │
└─────────────────────────────────────────────────────────────┘
```

**运行态指示**：页面须区分 `live`（实时运行中）与 `archive`（历史导入快照），避免混淆。

### 5.2 真实实时行情服务

| 项 | 要求 |
|----|------|
| 数据源 | `market_data.sources.binance_futures.BinanceFuturesKlineProvider` |
| 标的/周期 | 一期 BTCUSDT / 1h |
| 拉取策略 | APScheduler：每 **30~60s** 轮询最近 N 根 K 线；检测新 bar 入库 |
| 存储 | SQLite `ohlcv_bars` 表；内存缓存最近 500 根供图表 |
| 推送 | 新 bar 写入后 WebSocket `event_type=bar_close` 通知前端 |
| 冷启动 | 首次启动用 `binance_vision` 或 futures 拉取近 30 天历史回填 |
| 与引擎对齐 | bar 任务以**同一数据源**的已收盘 K 线触发 `on_bar_close` |

### 5.3 后台运行态监控

| 项 | 要求 |
|----|------|
| 运行进程 | `runtime/bar_runner.py`（或 `trading_system/live/bar_scheduler.py`）常驻 |
| 触发 | 每根 1h K 线收盘 → 拉取 OHLCV → 模型信号 → `TradingEngine.on_bar_close` |
| 导出 | `MonitorExporter`：每 bar 将 decision/fill/trade/equity POST 到网站 `/api/ingest` |
| 健康 | `health` 含：`runner_alive`、`latest_bar_ts`、`last_decision_ts`、`lag_seconds` |
| 落盘 | 可选每 bar 增量写 SQLite；CSV `flush()` 仍可按日/按 bar 周期保留 |

### 5.4 按周期统计与 ViewModel

（与 v0.2 相同，数据来源在 live 模式下为**运行态事件流**而非仅 CSV 导入。）

Dashboard 核心字段：`ts, price, p_up, p_down, p_flat, p_risk, edge, conf, pred_cum_ret_5, action, reason_code, blocked, blocked_reason, position_ratio, state, portfolio_equity`。

### 5.5 K 线图表与交易标注

- 主图 OHLCV：**真实实时行情**（`binance_futures`）；
- 开平仓标注：**成交价**（`fills`/`trades`），与 v0.2 口径一致；
- 当前持仓：运行态下每 bar 更新未实现盈亏；
- 图表库：**lightweight-charts**，WebSocket 增量 append 新 bar。

### 5.6 多版本后台对接

```text
Backend Profile
├── id, display_name, manifest_path
├── symbol          # BTCUSDT
├── interval        # 1h
├── mode            # live | archive | replay
├── adapter_type    # live_push | archive_import
├── runner_config   # bar 任务参数（live 模式）
└── data_root       # 归档/日志目录（archive 模式）
```

| mode | 说明 | 优先级 |
|------|------|--------|
| **`live`** | 真实行情 + bar 任务 + 事件推送 | **一期主路径** |
| `archive` | 回测 `flush()` 后一次性导入 | 辅助 / 冷启动对照 |
| `replay` | 只读已导入快照 | 辅助 |

### 5.7 辅助功能

**一期**：决策明细（BLOCK 筛选）、运行态健康面板、归档导入对照。

**二期**：版本并排对比、告警、多标的/多周期、导出。

## 7. 技术路径

### 7.1 总体架构

```text
┌──────────────────────────────────────────────────────────────┐
│              前端 (website/src/frontend)                      │
│   LIVE 指示 │ Dashboard │ 实时K线 │ WS 订阅                    │
└────────────────────────────┬─────────────────────────────────┘
                             │ REST + WebSocket
┌────────────────────────────▼─────────────────────────────────┐
│              网站后端 (website/src/backend)                   │
│   MarketDataService │ Ingest API │ 聚合 │ WS Hub │ 版本注册  │
└──────────────┬─────────────────────────────┬─────────────────┘
               │                             │
               │ binance_futures REST        │ POST /api/ingest
               ▼                             ▼
        [交易所真实行情]              ┌──────────────────────────┐
                                      │ runtime/bar_runner.py     │
                                      │  bar调度 → TradingEngine  │
                                      │  → MonitorExporter        │
                                      └──────────────────────────┘
```

### 7.2 技术选型

| 层级 | 技术 |
|------|------|
| 前端 | React + TypeScript + Vite + **lightweight-charts** |
| 后端 | FastAPI + Python 3.11 |
| 实时行情 | **`binance_futures` REST 轮询**（一期）；二期可评估 WS |
| 运行态推送 | **WebSocket** + HTTP ingest |
| 调度 | **APScheduler**（行情轮询 + bar 对齐检查） |
| 存储 | SQLite（WAL） |
| 运行环境 | **`website/runtime/`**：venv、配置、bar_runner、启动脚本 |
| 时区 | API UTC；前端默认 UTC+8 |

### 7.3 后台适配器

```python
class BackendAdapter(Protocol):
    profile: BackendProfile

    def health(self) -> HealthStatus: ...           # runner_alive, latest_bar_ts, lag
    def import_snapshot(self, data_root: Path) -> ImportResult: ...  # archive 辅助
    def ingest_events(self, events: list[TradingEvent]) -> None: ... # live 主路径
    def get_config_summary(self) -> dict: ...
    def get_reference_metrics(self) -> dict: ...
    def get_ohlcv(self, start, end) -> pd.DataFrame: ...  # 来自 MarketDataService
    def get_trades(self, start, end) -> list[TradeRecord]: ...
    def get_equity_curve(self, start, end) -> list[EquityPoint]: ...
    def get_metrics(self, period) -> dict: ...
```

- **`LivePushAdapter`（一期主路径）**：消费 ingest + 读运行态 SQLite；
- **`ArchiveImportAdapter`（辅助）**：历史回测导入。

### 7.4 数据流

#### 一期主路径（Live + 真实实时行情）

```text
1. MarketDataService 每 30~60s 从 binance_futures 拉 BTCUSDT 1h
2. 检测到新 bar → 写入 ohlcv_bars → WS 推送 bar_close
3. bar_runner 在同一 bar 收盘后：
     拉 K 线 → ModelSignalProvider → TradingEngine.on_bar_close
     → MonitorExporter POST decision/fill/trade/equity
4. 网站 ingest → SQLite → WS 推送运行态事件
5. 前端：实时 K 线 append + Dashboard/持仓/标注更新
```

#### 辅助路径（Archive）

```text
回测结束 flush() → import_backtest → 只读对照（mode=archive）
```

## 12. 风险与应对

| 风险 | 应对 |
|------|------|
| Binance REST 限频 / 网络 | 退避重试；拉取窗口最小化；内网可配代理 |
| bar 时间与交易所不对齐 | 以交易所 K 线 `close_time` 为准，runner 延迟 5~15s 再触发 |
| `TradeLogger` 仅 flush 写 CSV | live 模式走 **MonitorExporter → ingest**，不依赖 CSV tail |
| 网站与 runner 同机部署 | 一期默认同机；ingest 走 localhost |
| 行情真实但成交是仿真 | UI 标明「纸交易 / 仿真成交」 |
| SQLite 并发 | WAL 模式 |
| 一期无鉴权 | 内网/VPN |

---

## 13. 验收标准（一期 MVP）

1. K 线来自 **`binance_futures` 真实行情**；新 1h bar ≤**60s** 出现在网站。
2. `bar_runner` 常驻运行，健康 API 返回 `runner_alive=true` 与 `latest_bar_ts`。
3. 每 bar 收盘后，信号/决策/持仓变化 ≤**30s** 经 WebSocket 更新前端。
4. Dashboard 五大门类在 **live** 模式下数据正确；页面有 **LIVE** 状态指示。
5. K 线开平仓标注使用成交价；当前持仓有明确标记。
6. 可切换 ≥2 个 prod 版本（live 或 archive）；可导入历史回测做对照。
7. `website/runtime/start.sh` 一键启动行情服务 + API + bar_runner + 前端。
8. 文档：001 v0.3、002 架构、003 API、004 数据模型。

## 15. 结论

交易监控网站的核心价值是：在**真实实时市场行情**下，观测 wildrose 交易系统**正在运行**的信号、持仓、成交与收益。

一期 MVP 以 **`binance_futures` 真实行情 + `bar_runner` 纸交易任务 + WebSocket 运行态推送** 为主路径；归档导入降为辅助对照能力。网站与 `runtime/bar_runner` 协同交付，不可仅靠事后 CSV 导入满足监控诉求。
