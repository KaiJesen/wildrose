## 2. 项目背景

### 2.1 现状

wildrose 交易系统（`trading_system/`）已具备较完整的**回测与规则决策**能力：

- 每根 K 线收盘可完成：信号接入 → 趋势/崩盘/分段/偏置过滤 → 规则决策 → 风控与仓位 → 模拟成交 → 日志落盘。
- `TradeLogger` 可输出 `decisions.csv`、`orders.csv`、`fills.csv`、`trades.csv`、`equity_curve.csv` 等复盘数据。
- `prod/vX.Y.Z/` 已固化多个策略版本（配置、checkpoint、参考指标、代码快照）。
- `market_data/` 可拉取 Binance 永续合约历史与准实时 K 线；`trade/tools/` 可生成事后最优交易点标注图。

### 2.2 痛点

当前系统**缺少面向运维与研究的统一监控入口**：

| 痛点 | 说明 |
|------|------|
| 无 Web 界面 | 只能通过 CSV/JSON/PNG 文件查看结果，无法实时浏览 |
| 无事件流 | 决策与成交仅在 `flush()` 时批量写盘，无推送/订阅机制 |
| 无聚合视图 | 信号、收益、持仓分散在多张表，缺少按周期/门类汇总 |
| 无多版本对比 | `prod/v0.0.0` ~ `v1.1.0` 等版本并存，缺少统一注册与切换 |
| 回测与监控割裂 | K 线、交易点、权益曲线需手工拼接图表 |

### 2.3 建设动机

在交易系统后台**基本可用**的前提下，需要一套**网站**实现：

1. **实时/准实时监控**交易运行状态；
2. **按周期统计**信号、收益、持仓等核心指标；
3. **门类聚合展示**，一眼看清系统健康度；
4. **K 线蜡烛图 + 交易标注**（开仓/平仓/持仓/杠杆/收益）；
5. **对接多个版本**的后台程序，屏蔽版本差异。

## 3. 项目目标

### 3.1 总目标

建设 wildrose **交易监控网站**，成为交易系统与研究人员之间的**统一观测与控制面板**。

### 3.2 具体目标

| # | 目标 | 可衡量结果 |
|---|------|------------|
| G1 | 实时可见 | 后台产生的新 K 线、信号、成交可在网站 **≤30s** 内可见（准实时，受 K 线周期约束） |
| G2 | 周期统计 | 支持 1h / 4h / 1d 等周期切换，展示区间内的信号分布、收益、持仓变化 |
| G3 | 门类聚合 | 首页按「信号 / 收益 / 持仓 / 风险 / 版本」五大类聚合卡片与趋势 |
| G4 | K 线标注 | 蜡烛图上叠加历史与当前开仓点、平仓点、持仓量、杠杆、单笔/累计收益 |
| G5 | 多版本对接 | 至少同时注册 **2 个** prod 版本后台，可在网站切换查看 |
| G6 | 可扩展 | 新增后台版本时，**不改前端**，仅增加适配器与配置 |

### 3.3 非目标（本期不做）

- **不替代**交易决策引擎本身（网站是观察者，不是决策者）
- **不实现**完整实盘下单与交易所账户对接（可预留接口，一期以回测/仿真/纸交易为主）
- **不实现**用户权限与多租户（一期单用户/内网部署）
- **不实现**移动端原生 App（响应式 Web 即可）
- **不展示**事后最优交易点（`trade/tools/optimal_trade_points`）为实盘信号——仅可作为研究对比层，需明确标注「事后标注」

## 4. 用户与使用场景

| 角色 | 场景 |
|------|------|
| 策略研究员 | 查看当前信号分布、模型 edge/conf、被 BLOCK 的原因，对比不同版本 |
| 运维/开发 | 监控后台进程是否存活、最新 bar 是否推进、日志是否落盘 |
| 项目经理 | 查看区间收益、回撤、胜率、交易笔数，对照 prod 参考指标 |
| 复盘分析 | 在 K 线图上对照开平仓点，检查规则是否按预期触发 |

## 5. 功能需求

### 5.1 监控总览（Dashboard）

首页按**门类**聚合展示，建议五大板块：

```text
┌─────────────────────────────────────────────────────────────┐
│  版本选择器  [v1.0.0 ▼]  标的 [BTCUSDT]  周期 [1h ▼]       │
├──────────────┬──────────────┬──────────────┬─────────────────┤
│  信号概览     │  收益概览     │  持仓概览     │  风险概览        │
│  p_up/down   │  总收益       │  方向/仓位比  │  最大回撤        │
│  edge/conf   │  今日/本周    │  杠杆 20x    │  风控拦截次数     │
│  BLOCK 占比  │  胜率/盈亏比  │  未实现盈亏   │  保证金使用率     │
├──────────────┴──────────────┴──────────────┴─────────────────┤
│  权益曲线（可缩放）                    最近 N 笔交易列表       │
├─────────────────────────────────────────────────────────────┤
│  K 线蜡烛图 + 开平仓标注 + 持仓区间着色                      │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 按周期获取与统计

| 统计维度 | 指标示例 | 数据来源 |
|----------|----------|----------|
| **信号** | `p_up/p_down/p_flat/p_risk`、`edge`、`conf`、`pred_cum_ret_5`、BLOCK 次数与原因 TopN | `decisions` |
| **收益** | 总收益、区间收益、年化、超额收益、胜率、盈亏比、平均每笔持仓 K 线数 | `trades` + `equity_curve` + `metrics` |
| **持仓** | 当前方向、仓位比例、入场价、持仓 K 线数、未实现盈亏、保证金估算 | `decisions` 最新行 + `portfolio` 状态 |
| **成交** | 开/平仓时间、价格、手续费、滑点、成交原因 | `fills` / `trades` |
| **风险** | 最大回撤、风控拦截、仓位上限违规、冷却状态 | `decisions` + `metrics` |

**周期切换**：支持按用户选择的时间窗口（今日 / 近 7 天 / 近 30 天 / 自定义）与 K 线周期（1h / 4h / 1d）聚合。

### 5.3 K 线图表与交易标注

| 图层 | 内容 |
|------|------|
| 主图 | OHLCV 蜡烛图（绿涨红跌，加密惯例） |
| 开仓标记 | 做多 ▲（蓝）/ 做空 ▼（橙），附方向与开仓价 |
| 平仓标记 | ✕ 标记，附净收益百分比 |
| 持仓区间 | 开仓至平仓（或当前）之间虚线连接/背景着色 |
| 持仓信息 | 悬浮提示：仓位比例、杠杆（固定 20x）、未实现/已实现盈亏 |
| 副图（可选） | 成交量柱、权益曲线叠加 |

标注数据来自引擎 `trades.csv` / `fills.csv`，**不是**事后最优标注器。

### 5.4 多版本后台对接

网站需能同时对接多个后台实例，每个实例对应一个 **Backend Profile**：

```text
Backend Profile
├── id              # 如 "prod-v1.0.0"
├── display_name    # "v021 trend-bias full_bias"
├── manifest_path   # prod/v1.0.0/MANIFEST.json
├── symbol          # BTCUSDT
├── interval        # 1h
├── mode            # backtest | paper | live（一期以 backtest/paper 为主）
├── adapter_type    # file_tail | http_push | grpc（见 §7.3）
└── data_root       # 日志/状态文件根目录
```

切换版本时，Dashboard、统计、K 线图同步刷新为该实例数据。

### 5.5 辅助功能（二期可裁剪）

- 决策明细表：可筛选 `action`、`reason_code`、`blocked`
- 版本对比：同区间两版本收益/回撤并排
- 告警配置：回撤超阈值、bar 停滞、进程离线（预留）
- 导出：区间 CSV / 截图

## 7. 技术路径

### 7.1 总体架构

采用 **三层架构**，网站与交易后台解耦：

```text
┌──────────────────────────────────────────────────────────────┐
│                     前端 (website/src/frontend)               │
│   Dashboard │ K线图表 │ 统计面板 │ 版本切换                     │
└────────────────────────────┬─────────────────────────────────┘
                             │ REST / WebSocket
┌────────────────────────────▼─────────────────────────────────┐
│                     网站后端 (website/src/backend)              │
│   API Gateway │ 聚合服务 │ 行情服务 │ 版本注册 │ WS Hub         │
└────────────────────────────┬─────────────────────────────────┘
                             │ BackendAdapter 接口
┌────────────────────────────▼─────────────────────────────────┐
│              后台适配层 (website/src/adapters)                  │
│   FileTailAdapter │ HttpPushAdapter │ (future) GrpcAdapter   │
└────────────────────────────┬─────────────────────────────────┘
                             │
        ┌────────────────────┼────────────────────┐
        ▼                    ▼                    ▼
  prod/v1.0.0           prod/v1.1.0         研发 configs/
  回测/仿真进程          纸交易进程           实验实例
```

### 7.2 技术选型（建议）

| 层级 | 技术 | 理由 |
|------|------|------|
| 前端 | **React + TypeScript + Vite** | 组件生态成熟；图表可用 lightweight-charts 或 ECharts |
| 后端 | **FastAPI + Python 3.11** | 与现有 `trading_system` / `market_data` 同栈，便于复用 |
| 实时推送 | **WebSocket**（FastAPI 内置） | bar/decision/fill 增量推送 |
| 存储 | **SQLite**（一期）→ PostgreSQL/TimescaleDB（二期） | 一期零运维；二期支持长历史 |
| 任务/调度 | **APScheduler** 或独立 worker | 定时拉 K 线、扫描日志增量 |
| 部署 | Docker Compose（website + 可选 DB） | 内网单机即可 |

### 7.3 后台适配器设计

所有后台程序通过统一接口与网站交互：

```python
class BackendAdapter(Protocol):
    """交易后台适配器：屏蔽版本与传输差异。"""

    profile: BackendProfile

    def health(self) -> HealthStatus: ...
    def latest_bar(self) -> BarSnapshot | None: ...
    def stream_events(self) -> Iterator[TradingEvent]: ...  # decision/fill/trade
    def get_ohlcv(self, start, end) -> pd.DataFrame: ...
    def get_trades(self, start, end) -> list[TradeRecord]: ...
    def get_equity_curve(self, start, end) -> list[EquityPoint]: ...
    def get_metrics(self, period) -> dict: ...
```

**一期实现 `FileTailAdapter`**：

- 监听 `backtest/` 或指定 `out_dir` 下 CSV 增量；
- 解析 `MANIFEST.json` 自动注册 prod 版本；
- 无需改动现有 `TradeLogger` 即可对接。

**二期实现 `HttpPushAdapter`**：

- 交易后台在 `on_bar_close()` 后 POST 事件到网站 ingestion API；
- 适合纸交易/准实盘，延迟更低。

### 7.4 数据流（一期）

```text
1. 用户打开网站 → 选择 Backend Profile（如 prod-v1.0.0）
2. 后端加载 MANIFEST + 配置 → 展示版本元信息与参数摘要
3. 行情服务从 binance_futures 拉取最近 N 根 K 线 → 入库/缓存
4. FileTailAdapter 扫描 decisions/trades/equity CSV → 写入 SQLite
5. 聚合服务按选定周期计算信号/收益/持仓统计 → API 返回
6. 前端请求 REST 渲染 Dashboard；WebSocket 推送新 bar/新成交
7. K 线组件请求 OHLCV + trades → 叠加开平仓标注
```

### 7.5 与交易后台的封装边界

| 归属 | 职责 |
|------|------|
| **交易后台**（现有 `trading_system`） | 决策、成交、写日志；可选增加 `MonitorExporter` 薄封装 |
| **网站适配层**（新建 `website/src/adapters`） | 读取/接收后台输出，转换为统一 `TradingEvent` |
| **网站后端**（新建 `website/src/backend`） | 存储、聚合、API、推送 |
| **网站前端**（新建 `website/src/frontend`） | 展示、交互 |

建议在 `trading_system/` 增加可选模块 `monitor_exporter.py`（二期）：

- 在 `TradeLogger.record_*` 时同步 emit 事件；
- 不破坏现有 CSV 落盘逻辑；
- 由网站适配层消费，而非让网站直接 import 引擎。

## 9. 核心数据模型（概要）

### 9.2 展示用聚合表（逻辑视图）

| `v_trade_marks` | entry_ts、exit_ts、side、entry_price、exit_price、net_pnl、leverage |

详细字段映射见后续 `004_数据模型.md`（将对照 `decisions.csv` / `trades.csv` 列定义）。

## 12. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 后台仅 CSV 落盘、无实时流 | 监控延迟 | 一期 FileTail；二期 MonitorExporter + HTTP push |
| `decisions.csv` 列过多（100+） | 前端难以展示 | 聚合层输出精简 ViewModel，明细页按需展开 |
| 多版本 schema 漂移 | 适配器破碎 | 以 `TradingEvent` 统一载荷；版本差异在 adapter 内消化 |
| K 线数据源延迟（Vision 1~2 天） | 实时图不准 | 监控用 `binance_futures` REST；Vision 仅历史回填 |
| 事后最优标注与实盘混淆 | 误导决策 | UI 明确分区；实盘标注仅来自引擎 trades |
| 一期无鉴权 | 内网安全风险 | 部署于内网/VPN；二期加 Token |

## 13. 验收标准（一期）

1. 网站可注册并切换 **≥2 个** prod 版本后台。
2. Dashboard 五大门类（信号/收益/持仓/风险/版本）数据正确，支持 **今日 / 7 天 / 30 天** 切换。
3. K 线蜡烛图正确展示 OHLCV，并标注**全部历史**开平仓点；当前持仓有明确标记。
4. 标注信息包含：方向、价格、杠杆（20x）、单笔净收益；悬浮可查看详情。
5. 新 backtest 输出 CSV 后，网站在 **1 分钟内** 可查到新交易（FileTail）。
6. 文档齐全：`doc/001` 立项、`doc/002` 架构、`doc/003` API（可在 M1 后补齐）。
