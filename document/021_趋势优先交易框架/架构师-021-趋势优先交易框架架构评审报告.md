## 2. 总体评价

021 的核心方向是正确的：把当前分散在 `TrendSignalProvider`、`TrendSegmentEngine`、`SlowUptrendDetector`、`CrashRegimeDetector`、旧 `TrendRegimeFilter` 中的趋势判断，统一收敛为 `TrendBiasContext`，再让 `RuleEngine`、`PositionSizer`、`RiskManager` 消费这个统一偏置层。

这个方向能解决当前系统的核心结构问题：

```text
短信号先触发，趋势模块再修正
```

升级为：

```text
趋势背景先裁决，短信号再解释与执行
```

但当前 021 文档仍处于“架构意图清楚、工程契约不足”的状态。若直接实现，容易出现：

1. 新增 `TrendBiasContext` 后，旧趋势规则仍继续生效，导致重复 block / 重复放宽；
2. 偏置系数语义不统一，阈值被错误放大或缩小；
3. 趋势加仓与 20x / 20% 仓位上限之间缺少可验证风险预算；
4. 多个趋势模块冲突时，`TrendBiasBuilder` 缺少确定性裁决表；
5. 回测收益变好或变差后，难以判断是趋势偏置有效，还是旧规则副作用。

因此，本评审建议：**021 可以立项实施，但必须先补齐统一字段契约、冲突优先级、偏置数学公式、风险预算接口和验收 gates。**

## 3. 主要问题与建议

### 问题一：`TrendBiasContext` 字段命名与语义尚未完全统一

**严重级别：高**

不同 021 文档中字段命名存在不一致：

| 文档表达 | 另一处表达 |
|----------|------------|
| `long_open_bias` | `open_bias_long` |
| `long_size_bias` | `size_bias_long` |
| `allow_counter_add_long` | `allow_long_add` / `block_counter_long` |
| `alignment_score` | `alignment_score_long` / `alignment_score_short` |

这会导致程序员实现时产生两套字段，或者在 `rules.py`、`sizing.py` 中出现适配层混乱。

**建议修正：**

021 应先冻结一个唯一 schema，建议采用“方向在前、用途在后”的字段名：

```python
@dataclass(frozen=True)
class TrendBiasContext:
    macro_direction: BiasDirection
    leg_direction: BiasDirection
    micro_direction: BiasDirection

    alignment_score_long: int
    alignment_score_short: int
    counter_level_long: CounterTrendLevel
    counter_level_short: CounterTrendLevel

    open_bias_long: float
    open_bias_short: float
    size_bias_long: float
    size_bias_short: float
    hold_bias_long: float
    hold_bias_short: float
    exit_bias_long: float
    exit_bias_short: float

    allow_open_long: bool
    allow_open_short: bool
    allow_add_long: bool
    allow_add_short: bool
    force_exit_long: bool
    force_exit_short: bool

    active_leg_id: int | None
    active_leg_type: str
    sub_phase: str
    leg_progress_ratio: float

    source_confidence: float
    reason_codes: list[str]
```

字段冻结后，所有 021 文档与代码都按该命名重写，避免后续工程漂移。

---

### 问题二：偏置系数缺少明确数学含义，容易被反向使用

**严重级别：高**

设计中写到：

```text
long_open_bias > 1 放宽多头开仓，<1 收紧
```

但当前 `RuleEngine` 中阈值大多是：

```text
edge >= threshold
p_up >= threshold
pred_cum_ret_5 >= threshold
p_risk <= threshold
```

对于“越大越好”的阈值，放宽应当降低 threshold；对于“越小越好”的风险阈值，放宽可能是提高上限。若只给 `bias > 1`，程序员可能写出相反效果。

**建议修正：**

021 必须规定公式，而不是只规定直觉：

```text
effective_long_edge_threshold = base_long_edge_threshold / max(open_bias_long, eps)
effective_long_prob_threshold = base_long_prob_threshold / max(open_bias_long, eps)
effective_long_risk_max       = base_risk_max * risk_tolerance_bias_long

target_position_ratio =
    base_position_ratio
    * signal_strength_scale
    * size_bias_side
    * risk_budget_scale
```

并将 `open_bias` 与 `risk_tolerance_bias` 分开。否则“顺势放宽开仓”可能不小心同时放宽风险，这在 20x 杠杆系统中风险过高。

---

### 问题三：趋势模块冲突时缺少确定性裁决表

**严重级别：高**

021 指出当前趋势模块很多、优先级分散，但 `TrendBiasBuilder` 目前只是说“汇总”：

```text
TrendSignalProvider
TrendSegmentEngine
SlowUptrendDetector
CrashRegimeDetector
TrendRegimeFilter
```

还没有明确当它们冲突时谁覆盖谁。例如：

| 场景 | 可能冲突 |
|------|----------|
| `TrendSegmentEngine` = `SLOW_UP_LEG`，但 `TrendSignal` = `REVERSAL_RISK` | 慢涨持有还是准备退出 |
| `CrashContext.is_crash = true`，但短周期模型 `p_up` 高 | crash 优先还是模型优先 |
| `SegmentContext.sub_phase = PULLBACK`，但 `best_point_exit` 高 | 正常回撤还是退出 |
| macro UP，leg DOWN，micro DOWN_ACCELERATION | 逆大趋势空头是 medium 还是 hard block |

**建议修正：**

新增 `TrendBiasBuilder` 裁决优先级表：

| 优先级 | 来源 | 用途 |
|--------|------|------|
| P0 | `RiskManager` / 熔断 / 强平风险 | 永远最高，直接阻断或退出 |
| P1 | `CrashContext` 强确认 | 可强制空头偏置，但仍受风险预算限制 |
| P2 | `SegmentContext.active_leg` | 主交易腿方向，决定 `leg_direction` |
| P3 | `TrendSignal` | 微观方向、阶段、破坏信号 |
| P4 | `SlowTrendContext` | 多头慢趋势增强器，只修正 long bias |
| P5 | 旧 `TrendContext` | 兼容兜底，不能覆盖 P1-P4 |
| P6 | `BestPointSignal` | 只作为 entry/exit 投票，不直接改主趋势方向 |

同时要求每次裁决写入 `reason_codes`，例如：

```text
SEGMENT_DOWN_CONFIRMED
CRASH_OVERRIDES_MODEL_UP
SLOW_UP_BOOST_LONG_SIZE
BEST_POINT_EXIT_HALF_VOTE_ONLY
```

### 问题四：`TrendBiasContext` 与旧规则并存期间可能重复生效

**严重级别：高**

当前 `RuleEngine.decide()` 已经直接接收：

```python
trend_context
crash_context
trend_signal
best_point_signal
slow_context
segment_context
```

021 建议再增加：

```python
trend_bias
```

如果 Phase B 只是“新增 bias 参数”，但旧逻辑继续读原始上下文，容易出现：

1. `trend_bias` 已经 block long，旧 `trend_context` 再 block 一次；
2. `slow_context` 已经提高 long bias，旧 slow-up 分支又绕过主流程开多；
3. `crash_context` 通过 bias 提升 short，旧 crash 分支又直接 `OPEN_SHORT_CRASH`；
4. 回测结果变化无法归因。

**建议修正：**

021 需要明确迁移期间的“单一决策所有权”：

```text
Phase A：只记录 bias，不参与决策；
Phase B：bias 只接管开仓允许性与阈值，旧趋势 block 关闭或降为诊断；
Phase C：bias 接管持仓 / 加仓 / 退出，旧 slow/crash/segment 直接动作改为 bias 修正；
Phase D：RuleEngine 不再直接消费趋势模块原始字段，只保留日志诊断。
```

并在配置中增加：

```json
{
  "trend_bias": {
    "enabled": true,
    "decision_scope": "observe|open_only|open_size|full",
    "disable_legacy_trend_rules": false
  }
}
```

没有该开关，021 很难做干净 A/B。

---

### 问题五：风险预算与趋势加仓的接口仍不够硬

**严重级别：高**

021 提到趋势仓要与风险预算打通，但目前仍偏原则描述：

```text
加仓后保证金最坏损失
最大仓位 20%
固定 20 倍杠杆
```

在 20x 杠杆下，趋势偏置一旦提高 size，风险不应只靠 `max_position_ratio` 兜底，还需要明确新增仓位后的最坏损失估计。

**建议修正：**

`RiskManager` 需要新增一个显式风险预算对象：

```python
@dataclass(frozen=True)
class RiskBudget:
    current_position_ratio: float
    max_position_ratio: float
    current_margin_ratio: float
    remaining_position_ratio: float
    worst_case_loss_ratio: float
    remaining_loss_budget_ratio: float
    allow_add: bool
    reason: str
```

`PositionSizer` 只能在 `RiskBudget.allow_add=True` 且 `remaining_position_ratio > 0` 时扩大仓位：

```text
target_position_ratio =
    min(
        base * signal_scale * trend_size_bias,
        risk_budget.remaining_position_ratio,
        cfg.base.max_position_ratio
    )
```

同时验收中必须记录：

```text
max_position_ratio_observed
max_margin_loss_ratio_observed
trend_add_rejected_by_risk_count
trend_add_allowed_count
```

### 问题六：`macro_direction` 的来源过弱，可能引入伪精度

**严重级别：中**

文档建议：

```text
macro_direction 可由 TrendSignalProvider 的方向 + 更长窗口特征推导
```

但当前 `TrendSignalProvider` 本质上仍是逐 bar 趋势模块。若直接拿它派生 macro，会让 macro、leg、micro 三层高度同源，表面上有三层趋势，实际只是同一个信号的不同窗口版本。

**建议修正：**

在没有稳定 Regime 模块前：

1. `macro_direction` 默认允许为 `NONE`；
2. `alignment_score` 不应因 macro 缺失而误判；
3. 大趋势分数权重低于 `leg_direction`；
4. 若使用长窗口 EMA/segment 生成 macro，必须在日志写 `MACRO_DERIVED_WEAK`。

建议权重：

```text
leg_direction:   ±2
macro_direction: ±1
micro_direction: ±1
```

这是当前文档已有倾向，但需要强制写进实现规范。

---

### 问题七：`CrashContext` 与统一框架的关系需要保留“紧急通道”

**严重级别：中**

021 说 crash 不应再独立定义完整持仓逻辑，而应通过 `TrendBiasContext` 输出偏置。方向正确，但要注意：crash 是极端风险场景，不能被普通 bias 流程完全稀释。

例如模型强多、macro UP、但价格发生模型盲区崩跌时，`CrashContext` 应能强制：

```text
block_long = true
short_open_bias 明显上调
long_exit_bias 明显下调
```

但是否开 crash short 仍应交给风险预算。

**建议修正：**

把 crash 定义为：

```text
P1 紧急偏置源，不直接绕过风险，但可以覆盖普通趋势方向。
```

不要让 crash 退化成普通 `short_size_bias_up`，否则会削弱 015 项目为模型失效设计的保护能力。

---

### 问题八：缺少“交易动作优先级”总表

**严重级别：中**

021 定义了 open、add、hold、exit 的偏置，但没有定义同一根 bar 上多个动作同时成立时如何排序。例如：

```text
ADD_TREND_CONTINUATION
REDUCE_TREND_EXHAUSTION
CLOSE_RISK
OPEN_REVERSE
HOLD_RUNNER
```

若没有动作优先级，`RuleEngine` 重构后仍可能被条件顺序主导。

**建议修正：**

新增统一动作优先级：

```text
P0 风险强制退出 / 熔断
P1 趋势破坏强退出 / 反向腿确认
P2 减仓保护利润 / exhaustion
P3 加仓 / upgrade
P4 新开仓
P5 继续持有
P6 观望
```

并明确：

```text
同一 bar 内，P0/P1 可以覆盖任何趋势偏置；
P3 加仓必须晚于 P0/P1/P2 检查；
新开仓不得绕过 trend_bias.allow_open_side 与 RiskBudget。
```

---

### 问题九：验收指标仍偏功能验收，缺少架构验收与回归 gates

**严重级别：中**

模块改造清单中的验收建议包括“字段能稳定输出、A/B 对比”，但还不够可执行。021 是架构改造，不能只看收益率。

**建议补充 gates：**

| 类别 | 指标 | 建议 |
|------|------|------|
| 架构 | `legacy_trend_direct_read_count` | Phase D 应显著下降 |
| 可解释 | `bias_reason_codes_coverage` | 每个非 HOLD 动作必须有 bias reason |
| 趋势覆盖 | `confirmed_leg_coverage_ratio` | 不低于 020 基线 |
| 持仓 | `avg_trend_hold_bars` | 不低于 020 基线 |
| 逆势控制 | `hard_counter_open_count` | 应接近 0 或仅试探仓 |
| 风险 | `max_position_ratio_observed` | 永远 <= 20% |
| 风险 | `trend_add_rejected_by_risk_count` | 必须可统计 |
| 回归 | `max_drawdown` | 不得显著劣化 |
| 稳定 | `trade_count` | 不应因 bias 重复生效暴增 |

## 4. 建议补充的设计产物

在进入代码实现前，建议 021 增加 4 个补充文档或附录。

### 4.1 `TrendBiasContext` 最终字段契约

内容包括：

1. 字段名；
2. 类型；
3. 默认值；
4. 来源模块；
5. 是否允许缺失；
6. 是否进入日志；
7. 是否影响交易。

### 4.2 `TrendBiasBuilder` 冲突裁决表

内容包括：

1. 多模块冲突优先级；
2. crash / slow_up / segment / trend_signal 的覆盖关系；
3. reason code 命名规范；
4. 特殊场景样例。

### 4.3 偏置数学公式说明

内容包括：

1. `open_bias` 如何修正阈值；
2. `size_bias` 如何进入 `PositionSizer`；
3. `hold_bias` 如何修正最小持仓；
4. `exit_bias` 如何修正退出票门槛；
5. bias 不得直接放宽硬风险边界。

### 4.4 021 A/B 回测验收表

至少包含：

```text
v020 baseline
v021_observe_bias_only
v021_open_bias
v021_open_size_bias
v021_full_bias
```

每一阶段都能独立回滚。

## 5. 建议实施顺序

### Phase 0：契约冻结

先完成：

1. `TrendBiasContext` 最终字段；
2. `TrendBiasConfig`；
3. `TrendBiasBuilder` 优先级表；
4. 偏置公式；
5. 日志字段。

不建议在契约冻结前改 `rules.py` 主流程。

### Phase A：观察模式

实现：

```text
engine 生成 trend_bias；
logger 记录 trend_bias；
rules 不消费 trend_bias。
```

目标：

```text
确认 bias 输出是否稳定，尤其是 6 月初下跌、慢涨长段、震荡段。
```

### Phase B：开仓 gate 接管

只让 `trend_bias` 影响：

```text
allow_open_long / allow_open_short
open_bias_long / open_bias_short
```

旧趋势 block 应关闭或降级为诊断，避免重复生效。

### Phase C：仓位接管

让 `PositionSizer` 使用：

```text
size_bias_long / size_bias_short
RiskBudget
```

严格检查 20% 总仓位与最坏损失。

### Phase D：持仓 / 加仓 / 退出接管

逐步接管：

```text
allow_add_side
hold_bias_side
exit_bias_side
force_exit_side
```

并清理旧 `RuleEngine` 直接读取 trend/segment/slow/crash 的局部特判。

## 6. 最终评审结论

021 的架构方向 **建议通过**，原因是它抓住了当前系统最关键的问题：趋势模块已经很多，但缺少统一裁决层；短信号仍是主驱动，趋势更多像补丁。

但本评审不建议直接进入“大改 `RuleEngine` 主流程”的实现阶段。应先补齐以下前置条件：

1. 冻结 `TrendBiasContext` 字段命名与类型；
2. 明确偏置系数的数学公式；
3. 给出 `TrendBiasBuilder` 冲突裁决优先级；
4. 定义 `RiskBudget`，确保趋势加仓不突破风险边界；
5. 设置 `trend_bias.decision_scope`，支持 observe / open_only / full 分阶段回测；
6. 增加架构验收指标，避免只看收益率。

如果以上补充完成，021 可以成为当前交易系统从：

```text
短信号主导 + 趋势补丁
```

升级为：

```text
趋势背景主导 + 短信号执行细节
```

的合理架构转折点。
