## 3. 阻塞问题

### 问题一：`exit_bias` 公式与语义相反

**严重级别：阻塞**

`软件设计师-021-偏置数学公式说明.md` 中写明：

```text
exit_bias > 1：需要更多票才退出（顺势 runner）
exit_bias < 1：更快退出
```

但当前公式是：

```text
effective_exit_votes_required =
    ceil(base_exit_votes / max(exit_bias_side, eps))
```

这会产生相反效果：

```text
base_exit_votes = 3
exit_bias = 1.5
ceil(3 / 1.5) = 2
```

也就是说，`exit_bias > 1` 反而降低退出门槛，使顺势 runner 更容易退出。

**必须修正：**

建议改为：

```text
effective_exit_votes_required =
    ceil(base_exit_votes * max(exit_bias_side, eps))
```

或者保留除法公式，但将字段改名为：

```text
exit_speed_bias
```

其中 `>1` 表示退出更快。考虑当前所有文档都把 `exit_bias > 1` 解释为“更难退出”，建议采用乘法公式。

**影响范围：**

- `软件设计师-021-偏置数学公式说明.md` §7；
- `软件设计师-021-TrendBiasContext字段与伪代码说明.md` 中的 `exit_bias` 伪代码；
- 所有涉及 `exit_bias_long / exit_bias_short` 的验收解释。

---

### 问题二：`RiskBudget` 只定义 `allow_add`，但被用于新开仓 gate

**严重级别：阻塞**

当前 `RiskBudget` 定义为：

```python
class RiskBudget:
    current_position_ratio: float
    max_position_ratio: float
    current_margin_ratio: float
    remaining_position_ratio: float
    worst_case_loss_ratio: float
    remaining_loss_budget_ratio: float
    allow_add: bool
    reason: str
```

但同一文档又规定：

```text
P4 新开仓不得绕过 allow_open_* + RiskBudget
```

这会产生语义不完整：

- 空仓开新仓不是 add；
- 反手开仓也不是普通 add；
- 加仓、开仓、反手、扩大仓位的风险判断不同；
- 使用 `allow_add` 来 gate `OPEN_LONG` / `OPEN_SHORT` 会让命名和行为不一致。

**必须修正：**

建议把 `RiskBudget` 扩展为动作级字段：

```python
@dataclass(frozen=True)
class RiskBudget:
    current_position_ratio: float
    max_position_ratio: float
    current_margin_ratio: float
    remaining_position_ratio: float
    worst_case_loss_ratio: float
    remaining_loss_budget_ratio: float

    allow_open_long: bool
    allow_open_short: bool
    allow_add_long: bool
    allow_add_short: bool
    allow_reverse: bool

    open_reject_reason_long: str
    open_reject_reason_short: str
    add_reject_reason_long: str
    add_reject_reason_short: str
```

或者拆成：

```text
RiskBudget
RiskPermission
```

但无论采用哪种形式，Phase 0 的契约必须能回答：

```text
能不能新开？
能不能加仓？
能不能反手？
还能开多大？
被拒绝的原因是什么？
```

## 4. 高优先级修正建议

### 问题三：P0 风险层在 Builder 与消费阶段之间边界仍有轻微冲突

**严重级别：高**

冲突裁决表中将 P0 写为：

```text
RiskManager / 熔断 / 强平风险 → force_exit_*、禁止一切加仓
```

但 Builder 推荐流程又写：

```text
Step 8: P0 由 RuleEngine/RiskManager 在消费阶段最终裁定 force_exit
```

这两个表述容易导致实现歧义：

1. `TrendBiasBuilder` 是否接收 `RiskEvent` / `RiskBudget`？
2. `force_exit_*` 是 bias 的输出，还是 RiskManager 的最终覆盖？
3. 如果 Builder 先写 `force_exit=false`，RiskManager 后面改为强退，日志以谁为准？

**建议修正：**

明确 P0 不属于 `TrendBiasBuilder` 的趋势裁决输入，而属于最终动作裁决层：

```text
TrendBiasBuilder 输出趋势偏置；
RiskManager 输出 RiskBudget / RiskEvent；
RuleEngineActionResolver 按 P0–P6 动作优先级合并二者。
```

字段上建议区分：

```text
trend_bias.force_exit_long     # 趋势侧强退建议，如 LEG_END / crash 极端
risk_event.force_exit          # 风险侧强退命令，最终优先级最高
```

不要让 `TrendBiasBuilder` 伪装成风险管理器。

---

### 问题四：强逆势 hard block 与 `open_bias == 0` 的关系需要写成硬不变量

**严重级别：高**

字段契约中 `counter_level_* == HARD_BLOCK` 可由 `allow_open_* = false` 表达；公式文档也写了硬边界：

```text
counter_level_* == HARD_BLOCK 时 open_bias_* == 0 或 allow_open_* == false
```

这里的“或”会给实现留下空间。如果 `counter_level_long == HARD_BLOCK`，但 `allow_open_long=True` 且 `open_bias_long=0`，某些代码可能只检查 `allow_open_long` 而漏掉 `open_bias`，造成强逆势开仓。

**建议修正为硬不变量：**

```text
counter_level_side == HARD_BLOCK
  => allow_open_side = false
  => open_bias_side = 0
  => size_bias_side = 0
  => allow_add_side = false
```

若允许极小试探仓，必须独立配置：

```text
allow_hard_counter_probe = true
hard_counter_probe_ratio <= 0.01
hard_counter_probe_max_hold_bars <= 2
```

不能混在普通开仓 bias 中。

## 5. 中优先级问题

### 问题五：旧字段别名仍少量残留在非契约文档中

**严重级别：中**

复查发现 `软件设计师-021-模块改造清单.md` 中仍有旧式表达：

```text
long_hold_bias_up
long_add_bias_up
bias_block_counter_long
bias_block_counter_short
```

这些不是冻结 schema 中的字段，容易误导实现和日志字段设计。

**建议修正：**

统一改为：

```text
hold_bias_long
allow_add_long
counter_level_long / allow_open_long
counter_level_short / allow_open_short
```

日志字段也建议直接记录冻结 schema，不再引入 `bias_*` 前缀别名：

```text
trend_bias.open_bias_long
trend_bias.counter_level_long
trend_bias.reason_codes
```

### 问题六：A/B 验收中的日期场景写成 `2024-06 初下跌`

**严重级别：中**

`软件设计师-021-A-B回测验收表.md` 的人工验收清单写：

```text

```

但 018/021 上下文一直讨论的是“6 月初持续下跌”，当前项目材料没有明确这是 2024、2025 还是 2026。若年份写错，回测师会选错验证区间。

```text

```

或直接写具体数据集路径与时间：

```text
backtest/... 中 6 月初下跌窗口：YYYY-MM-DD HH:MM ~ YYYY-MM-DD HH:MM
```

### 问题七：`trend_add_rejected_by_risk_count > 0` 不能作为无条件 gate

**严重级别：中**

A/B 表中 Phase C 要求：

```text
trend_add_rejected_by_risk_count 可统计且 > 0（证明 RiskBudget 生效）
```

但某些回测窗口可能没有任何加仓候选，或者 RiskBudget 足够宽松，拒绝次数为 0 并不代表 RiskBudget 未生效。

**建议修正：**

拆成两个指标：

```text
trend_add_candidate_count
trend_add_risk_evaluated_count
trend_add_rejected_by_risk_count
```

验收规则改为：

```text
若 trend_add_candidate_count > 0，
则 trend_add_risk_evaluated_count == trend_add_candidate_count；
rejected_by_risk_count 可为 0，但必须能输出 reason 分布。
```

---

### 问题八：`source_confidence` 进入冻结字段，但缺少计算定义

**严重级别：中**

字段契约定义：

```text
source_confidence: float [0, 1]
```

但当前没有说明如何计算。没有定义时，该字段容易变成固定值 `0.5`，失去诊断意义。

**建议修正：**

Phase 0 若不使用该字段，建议先标为诊断保留：

```text
source_confidence = 1.0 if active confirmed segment exists
source_confidence = 0.7 if trend_signal confirmed only
source_confidence = 0.5 if fallback / macro derived weak
source_confidence = 0.3 if conflicting sources unresolved
```

或者暂时移出冻结必需字段，避免实现时伪造置信度。

## 7. 复审结论

本次软件工程师补充后的 021 立项材料，已经从“方向性设计”推进到“可进入工程契约冻结”的状态。相比上一版，关键架构风险大幅降低。

2. `RiskBudget` 对新开仓 / 加仓 / 反手的权限表达不完整；

因此复审结论为：


## 8. Phase 0.1 修订清单

必须修订：

1. 将 `exit_bias` 公式改为 `ceil(base_exit_votes * exit_bias_side)`，或重命名字段；
2. 扩展 `RiskBudget`，至少区分 `allow_open_*`、`allow_add_*`、`allow_reverse`；
3. 明确 P0 风险层不属于普通 `TrendBiasBuilder` 趋势裁决，而是最终动作合并层；
4. 将 `HARD_BLOCK` 写成强不变量：`allow_open=false`、`open_bias=0`、`size_bias=0`、`allow_add=false`；
5. 清理非契约文档中的旧字段别名；
6. 修正 A/B 人工验收中的 6 月样本年份或改为具体回测窗口；
7. 调整 `trend_add_rejected_by_risk_count > 0` 为条件型验收；
8. 补充或移除 `source_confidence` 计算规则。

完成以上修订后，021 可以进入：

```text
Phase 0：trend_bias.py 数据结构 + Builder 骨架 + 配置 schema + 日志字段
```

并继续坚持原定迁移原则：

```text
observe → open_only → open_size → full
```
