本文给出 021 期建议采用的“趋势优先”交易框架草图。其目标不是替代现有短信号模型，而是为现有系统增加一个更高层的统一趋势调度层，使开仓、加仓、持仓和退出更系统地服从大趋势背景。

---

## 2. 核心原则

021 框架遵循以下原则：

1. **先趋势，后信号**
2. **先偏置，后执行**
3. **顺逆势按层级分层偏置，而不是一刀切**
4. **趋势腿优先于单 bar 波动**
5. **趋势仓与风险预算统一管理**

可压缩为一句话：

```text
短周期模型负责发现机会，
趋势主导层负责决定这些机会值不值得做、该做多大、该拿多久。
```

---

## 3. 旧框架与新框架对比

### 3.1 当前框架（简化）

```text
市场状态模型信号
  -> RuleEngine 基础开仓
  -> 趋势模块阻断 / 升级 / 延长 / 退出
  -> 风控校验
```

问题：

- 短周期信号是主驱动力
- 趋势更多像“修正器”
- 趋势模块之间缺少统一优先级

### 3.2 021 目标框架

```text
市场状态模型信号
趋势模块集合（trend / trend_signal / slow_up / crash / segment）
        ↓
统一趋势偏置层 TrendBiasContext
        ↓
RuleEngine 读取趋势偏置 + 短周期信号
        ↓
PositionSizer / RiskManager
        ↓
ExecutionEngine
```

重点变化：

1. 趋势先收敛成一个统一对象
2. RuleEngine 不再直接硬编码消费多个趋势模块原始结论
3. 趋势不再只用来 block，而是连续影响全链路决策

## 4. 建议新增对象：`TrendBiasContext`

### 4.1 设计目标

把当前分散在多个模块中的趋势判断整合成一个统一的交易偏置对象。

### 4.2 建议字段（冻结版摘要）

完整契约见 `软件设计师-021-TrendBiasContext最终字段契约.md`。命名规则：**方向在前、用途在后**。

```python
@dataclass(frozen=True)
class TrendBiasContext:
    macro_direction: BiasDirection
    leg_direction: BiasDirection
    micro_direction: BiasDirection

    alignment_score_long: int
    alignment_score_short: int
    counter_level_long: CounterTrendLevel
    counter_level_short: CounterTrendLevel

    open_bias_long: float          # >1 放宽多头开仓门槛（见偏置公式 §3）
    open_bias_short: float
    size_bias_long: float
    size_bias_short: float
    hold_bias_long: float
    hold_bias_short: float
    exit_bias_long: float
    exit_bias_short: float
    risk_tolerance_bias_long: float   # 与 open_bias 分离
    risk_tolerance_bias_short: float

    allow_open_long: bool
    allow_open_short: bool
    allow_add_long: bool
    allow_add_short: bool
    force_exit_long: bool
    force_exit_short: bool

    active_leg_id: int | None
    active_leg_type: str
    sub_phase: str
    leg_progress_ratio: float

    regime_strength: str
    regime_phase: str
    is_confirmed: bool
    is_trend_breaking: bool

    source_confidence: float
    reason_codes: list[str]
```

### 4.3 来源

此对象由 `TrendBiasBuilder` 按 **P0–P6 优先级** 汇总（见 `TrendBiasBuilder冲突裁决表.md`）：

- `TrendSegmentEngine`（P2）
- `TrendSignalProvider`（P3）
- `SlowUptrendDetector`（P4，仅 long 修正）
- `CrashRegimeDetector`（P1 紧急通道）
- 旧 `TrendRegimeFilter`（P5 兜底）
- `BestPointSignal`（P6，仅投票，不改方向）

## 5. 新版决策顺序

021 建议把每根 K 线上的交易决策统一改为以下顺序：

### Step 1：更新上下文

读取：

- 市场状态模型信号
- 价格、真实波动幅度
- 趋势上下文
- 趋势区间上下文
- crash / slow_up / best_point 上下文

### Step 2：构建 `TrendBiasContext`

根据当前趋势方向、阶段、腿结构、是否确认、是否破坏，以及大趋势 / 中趋势 / 小趋势是否一致，形成统一偏置。

### Step 3：解释短信号

不直接问“能不能开仓”，而是问：

```text
在当前趋势背景下，
这个短信号支持哪一边？
支持强度有多大？
是否只是逆趋势噪声？
```

### Step 4：生成目标动作

由 RuleEngine 基于：

- `TrendBiasContext`
- 短周期模型信号
- 当前持仓状态

给出：

- `OPEN`
- `ADD`
- `REDUCE`
- `HOLD`
- `CLOSE`
- `REVERSE`
- `BLOCK`

### Step 5：做风险校验

RiskManager 校验：

- 固定 20 倍杠杆
- 最大仓位 20%
- 加仓后保证金最坏损失
- 日回撤 / 连亏 / 熔断

### Step 6：执行与记录

ExecutionEngine 下发模拟成交，TradeLogger 记录：

- 当前趋势偏置
- 当前腿 ID
- 动作原因
- 仓位变化

## 6. 趋势如何连续影响交易

## 6.1 开仓偏置

当前系统主要是“满足阈值就开，不满足就不开”。021 建议改成“阈值受趋势偏置影响”。

### 大趋势、中趋势、小趋势高度同向

- 放宽同向开仓阈值
- 提高同向仓位与持有偏置
- 允许顺势加仓
- 对反向交易提高门槛
- 仅在强反向共振时才直接 block

### 大趋势与中趋势同向，但小趋势反向

- 允许小级别逆向交易
- 但需更高入场门槛
- 更小仓位
- 更短持有
- 默认禁止逆向加仓

### 大趋势与中趋势冲突

- 不简单视为“必须 block”
- 允许高质量逆向或反转早期交易
- 但仓位显著下调
- 退出更快
- 加仓更严格

### 强反向共振

- 直接阻断或近似阻断重逆势开仓
- 若保留试探仓，也必须极小、极短、不可加仓

---

## 6.2 仓位偏置

趋势不仅应决定“能不能开”，还应决定“开多大”。

### 建议逻辑

初始仓位 = 基础仓位 × 趋势方向偏置 × 趋势强度偏置 × 风险折扣

示例：

- 高一致性顺势：`1.2x ~ 1.5x` 基础仓位
- 普通顺势：`1.0x ~ 1.2x`
- 轻度逆向：`0.4x ~ 0.7x`
- 中度逆向：`0.2x ~ 0.5x`
- 强逆势：`0x` 或极小试探仓

在固定 20 倍杠杆、最大 20% 仓位约束下，趋势偏置只影响仓位分配，不改变总风控边界。

---

## 6.3 加仓偏置

加仓不应只看盈利和 `ADD_TREND_CONTINUATION`，应显式由趋势偏置层控制。

### 允许加仓

需同时满足：

1. 当前腿已确认
2. 趋势方向与持仓方向一致，或至少不是中度 / 强逆势
3. 趋势阶段为 `CONTINUATION` 或 `ACCELERATION`
4. 趋势未破坏
5. 当前风险未升高
6. 剩余仓位与剩余保证金风险预算允许

### 禁止加仓

任一条件成立：

- `REVERSAL_RISK`
- `EXHAUSTION`
- `LEG_END`
- 中度或强逆势持仓
- 已达最大加仓次数

---

## 6.4 持仓时长偏置

021 应把“持多久”也趋势化。

### 顺趋势持仓

- 提高最小持仓 bar 数
- 回撤中允许 `HOLD_TREND_PULLBACK`
- runner 模式允许更高退出确认门槛

### 轻度或中度逆向持仓

- 缩短最小持仓 bar 数
- 更早进入 `EXIT_PENDING`
- 更少的退出投票即可平仓
- 不允许把逆向 pullback 当成长期 runner

### 震荡持仓

- 不鼓励延长
- 以小利润、小止损、快速出清为主

---

## 6.5 退出偏置

退出也不应完全对称。

### 顺趋势退出

允许：

- 需要更多退出投票
- pullback 不立即视为趋势失败
- `best_point_exit` 只贡献一部分退出票

### 逆向退出

建议：

- 价格对当前持仓不利时更快减仓
- 模型累计收益翻反向时更早平仓
- 较少投票即可退出

---

## 7. 趋势腿与生命周期绑定草图

建议在 021 中把持仓生命周期更强地绑定到腿上下文：

| 生命周期 | 进入条件 | 允许动作 | 退出条件 |
|----------|----------|----------|----------|
| `PROBATION` | 新开仓 | 小仓位试探、观察 | 趋势确认 / 快速失败 |
| `TREND` | 趋势已确认 | 延长持仓 | 腿破坏 / 退出票足够 |
| `ADDABLE_TREND` | `ACCELERATION` 且盈利达到门槛 | 允许加仓 | 加速结束 / 风险升高 |
| `RUNNER` | 趋势盈利足够大 | 保护利润、低频退出 | 趋势衰竭 / 明确反转 |
| `EXIT_PENDING` | `EXHAUSTION` / `REVERSAL_RISK` / `LEG_END` | 只许减仓或平仓 | 最终退出 |

这样“当前是哪条腿、腿走到哪里、仓位该处于什么阶段”会形成一套完整映射。

---

## 8. 多空对称化草图

021 不建议继续把空头和多头做成大量不同分支，而应统一在趋势偏置层上表达：

### 统一项

- `direction`
- `phase`
- `is_confirmed`
- `allow_add_long` / `allow_add_short`
- `hold_bias`
- `size_bias`
- `exit_bias`

### 特化项

- `CrashContext`：空头特化
- `SlowUptrendContext`：多头特化

但这些特化项应只修改偏置，不直接绕过统一交易框架。

参数集中配置，并增加 **迁移开关**（架构评审问题四）：

```json
{
  "trend_bias": {
    "enabled": true,
    "decision_scope": "observe",
    "disable_legacy_trend_rules": false,
    "long_open_relax": 1.2,
    "light_counter_tighten": 0.75,
    "hard_counter_block": 0.0
  }
}
```

`decision_scope`：`observe` | `open_only` | `open_size` | `full`。分阶段说明见 `A-B回测验收表.md`。

偏置接入公式见 `偏置数学公式说明.md`，不散落隐式行为。

## 10. 021 框架的预期收益

如果该框架落地，理论上应解决的不是某一个具体回测数字，而是以下结构性问题：

1. 趋势已识别但不开仓
2. 趋势已确认却拿不住
3. 顺趋势和逆趋势的处理差异不够系统
4. 趋势模块太多但没有统一主导层
5. 趋势仓与加仓和风控之间关系不够清晰

## 11. 实施阶段（与架构评审对齐）

```text
Phase 0  契约冻结 — 不改 rules.py 主流程
Phase A  observe — 只记录 bias
Phase B  open_only — 开仓 gate + 阈值；关闭旧 trend block
Phase C  open_size — size_bias + RiskBudget
Phase D  full — 全链路；RuleEngine 仅消费 trend_bias
```

验收 gates 见 `软件设计师-021-A-B回测验收表.md`。

## 12. 结论

021 期推荐框架不是“再堆一个新模块”，而是：

```text
在现有模型、趋势、分段、风险模块之上，
新增一个统一趋势偏置层，
让整个交易系统从“短信号主导”
升级为“趋势背景主导，短信号负责执行细节”。
```

这将成为后续继续优化收益率、持仓覆盖率和趋势段利用率的基础。
