# 新结构分支稳定性验证

## 1. 文档说明

| 项 | 内容 |
|----|------|
| 角色 | 项目经理 |
| 流水号 | 009 |
| 项目名称 | 新结构分支稳定性验证 |
| 依据 | `document/008/项目进度纪要.md` |
| 适用范围 | 市场状态模型线 · 0062e 新结构分支 |

本文从项目管理角度定义 009 阶段要做什么、谁来做、何时算完成。技术指标与训练配置由架构师、训练师在各自文档中细化，本文不展开。

---

## 2. 阶段背景

008 阶段结束后，项目处于 **双基线并存** 状态：

- **0058**：usable 主基线，已发布 `v4.0-market-state-usable`
- **0062e**：新结构分支候选，已发布 `v4.1-cum-return-branch`，但仅为 **有条件通过**

项目经理关切的问题不是「指标再高多少」，而是：

1. 0062e 的成果是否 **可重复**，还是单次实验偶然？
2. 两套基线如何 **分工、命名、对外说明**，避免团队和用户混淆？
3. 下一步资源应投入 **验证稳定性**，还是继续 **追指标** 或 **扩结构**？

009 阶段的定位：**先验稳，再进阶。**

---

## 3. 阶段总目标

在 009 阶段结束时，项目应达成以下管理目标：

1. **结论可信**：对新结构分支（0062e 路线）是否值得继续投入，给出有实验支撑的明确判断，而非依赖单次结果。
2. **规则清晰**：usable 主基线与新结构分支的 **验收口径、使用场景、发布边界** 在文档与流程上对齐，减少角色间扯皮。
3. **路线可控**：明确「验稳之后」是否进入收益指标恢复类工作；未验稳前 **不替换主基线、不扩大模型结构**。
4. **交付可追溯**：本阶段产出可归档、可复盘，便于写入 `document/009/项目进度纪要.md` 及对外同步。

---

## 4. 阶段划分

### 阶段一：多轮复现验证（必做）

**目的**：用不同随机种子重复同一训练方案，观察新结构分支表现是否一致。

**负责角色**：训练师（执行）· 架构师（对照 008 任务书验收）

**主要交付**：
- 0063 系列实验报告（`reports/0063*`）
- 训练师实验复盘摘要（`document/009/0063_multi_seed_stability/`）

**阶段完成标志**：多轮实验跑完，训练师提交汇总材料；架构师完成是否符合「稳定性」的初审意见。

---

### 阶段二：阶段复审与路线裁定（必做）

**目的**：基于阶段一结果，由架构师给出技术结论，项目经理据此做 **发布与资源** 裁定。

**负责角色**：架构师（技术结论）· 项目经理（路线与版本裁定）

**主要交付**：
- `document/009/项目进度纪要.md`（本阶段终审记录）
- 是否维持双基线、是否启动下一阶段、是否打新版本的 **书面决定**

**阶段完成标志**：009 项目进度纪要定稿；团队对「0062e 是否升级为稳定分支候选」有统一说法。

---

### 阶段三：流程与文档对齐（必做，可与阶段一并行）

**目的**：把 008 以来「两套验收、两套基线」的共识固化到工程与文档，降低协作成本。

**负责角色**：软件设计师 / 工程师（实现）· 项目经理（验收对齐结果）

**主要交付**：
- 训练与报告流程能区分 **usable 线** 与 **新结构分支线**
- `document/README.md`、项目日记、关键路径说明保持同步

**阶段完成标志**：新成员能仅凭文档理解「当前该用哪条基线、本阶段在验证什么」。

---

### 阶段四：收益指标恢复探索（条件执行）

**目的**：仅在阶段一、二判定 **新结构分支稳定** 后，再投入资源尝试改善 008 遗留的 step 级收益排序短板。

**前置条件**：阶段二裁定「可进入进阶」；未经裁定 **不得启动**。

**负责角色**：架构师（出任务书）· 训练师（0064 系列）· 软件设计师（如需小改训练流程）

**主要交付**：
- 0064 系列实验（若启动）
- 架构师对「是否值得合并进分支基线」的后续意见

**阶段完成标志**：有明确 accept / reject / 暂缓 结论，或项目经理决定整体延后。

## 7. 完成判据（项目管理口径）

009 阶段视为 **完成**，须同时满足：

1. 多轮复现实验已完成，且有训练师书面汇总。
2. `document/009/项目进度纪要.md` 已发布，包含：稳定性结论、基线策略、是否进入 0064。
3. 双轨验收与文档入口已对齐，团队无「该用哪版模型」的歧义。
4. 版本策略已明确：是否维持仅 `v4.0` + `v4.1`，或新增 `v4.2` 类标签，由阶段二纪要写明。

009 阶段 **不自动** 将 0062e 升格为全项目唯一主基线；是否替换 0058 属于阶段二的管理裁定，而非本计划预设结果。

## 8. 风险与约束

| 风险 | 应对 |
|------|------|
| 单 seed 偶然被误判为突破 | 阶段一强制多轮复现 |
| 训练师与架构师对「通过」理解不一致 | 阶段三固化双轨流程；指标细节只在架构师文档 |
| 资源被吸到追指标、加结构 | 009 明确：未验稳不做结构扩展、不替换主基线 |
| 对外说法混乱（两版模型） | 进度纪要中写清使用场景：主基线 vs 分支候选 |
| 0064 过早启动 | 阶段四写死前置条件，项目经理把关 |

---

## 9. 本阶段不做的事

- 不扩大模型主干或新增复杂结构
- 不承诺 009 结束时全面超越 0058
- 不在项目经理文档中规定具体指标阈值（见架构师 / 训练师文档）
- 不将 checkpoints 纳入 git（策略不变）
