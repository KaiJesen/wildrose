## 2. 总体结论

026 的立项动因 **成立**、方向 **正确**、路径 **克制**。024/025 在规则+模型联合工程上已将 `c1_pw20` 这个固定 checkpoint 的 Pareto 前沿探索殆尽，026 回到训练侧是唯一的正确出口。

v1.0 方案采纳了架构师优化方案矩阵中的 **F1 组合**，并正确地以 **C3 cross-attn 参与头** 作为 Phase 0 最高杠杆入口。阶段分层（C3 → C1+D1 → A1/B1 → 全链路）的递进逻辑清晰，门禁设计合理。

但在 **5 处细节** 上仍存在可能导致错误决策或工程浪费的风险：

1. **C3 快速晋级条件过于宽松**：`part_auc ≥ 0.62` 即跳过 C1+D1 可能留性能余量在桌上。
2. **A1 vs B1 决策规则缺失**：Phase 2 两条路径的具体选型条件未给出。
3. **cum_IC 退化 10% 的冲击未量化**：放宽至 10% 后需 gating 而非仅报告。
4. **通道条件复测 §5.7 可能重新打开 025**：自动触发条件需要架构师审批闸门。
5. **Phase 0 B0 复现失败处理缺失**：025 有详细降级路径，026 需继承。

因此评审结论为：

```text
026 可以立项，建议有条件批准；
Phase 0（B0 复现 + C3 训练）为强制性硬门禁，未过不得进入 Phase 1；
v1.1 修订 5 处细节后进入实施；
若 C3 快速晋级触发，仍需完成 C1+D1 对照臂（可降为 P2 单臂验证）；
prod v1.1.0 维持不变直至 Phase 3 探索双门 + Phase 4 投产线双门 PASS。
```

  C3 valid part_auc ∈ [0.60, 0.62) → 必须执行 C1+D1 + A1（Phase 2，A1 优选）；

**评审意见：有条件通过。**

### 3.3 cum_IC 退化放宽至 10% — **通过方向，需硬门禁**

024 constrained 训练使用 5% 门，026 放宽至 10%。这在业务上合理：025 已证明 c1_pw20 的 cum_IC 约束 **过度压制** 参与头表达空间。

但 10% 退化意味着：若 0062e `cum_return_ic = 0.06`，允许降至 0.054。legacy `edge = p_up − p_down` 仍被标准 OPEN_SIGNAL / reverse / continue 消费——10% 退化可能导致这些路径的边际收益变化。

**修订建议**（非阻塞，Phase 0~1 内执行）：

```text
Phase 0~1 所有 checkpoint 在 valid 上监控并写入 cum_return_ic；
若退化 >8% 且 <10%：训练继续但写入 REPORT 高亮；
若退化 >10%：立即回退上一步权重，不得进入下一阶段；
若退化 >10% 且 C3 单独 accuses → C3 参与头增加 0.05·L_legacy 蒸馏项。
```

**评审意见：通过。建议在 train_v026_phase0_c3.py 中实现 early-stop IC 门。**

## 4. 架构合理性评审

### 4.1 C3 参与头 cross-attn — **通过，架构可行性高**

经代码审查确认：

- `HorizonReturnHead`（L231-264）已实现 **learned query + `nn.MultiheadAttention` + layer_norm + output MLP** 模式
- 输入 `pooled [B,D]` + `tokens [B,S,D]` + `seg_mask` 均在 `MarketStateHead.forward()` 中可用
- 当前 `participation_logit_long` 仅消费 `pooled`（L208），升级为 cross-attn 仅需约 40 行新增 + 替换调用点

```text
当前：participation_logit_long(pooled)           → [B]
升级：ParticipationAttn(query, tokens, seg_mask)  → logit [B]
      where query = learned_param + pooled
      cross-attn over frozen trunk tokens
```

trunk 冻结 → 参与头 Q/K/V 投影仅需学习"从 bar 序列中挑选参与信号"的注意力模式。`kaiming` 类初始化可复用 `HorizonReturnHead.query` 的 `torch.randn(..., 0.02)`。

**与 HorizonReturnHead 的关键差异（需在实现中注意）**：

| 项 | HorizonReturnHead | ParticipationAttn（C3） |
|----|-------------------|--------------------------|
| query shape | `[1, horizon, D]` | `[1, 1, D]` |
| output | `step_out [B,horizon]` + `cum_out [B]` | 单个 logit `[B]` |
| mask | `~seg_mask` | 同 |
| trainable | Q/K/V 投影 + output MLP | 同（trunk 冻结） |

**架构评审意见：通过。C3 是现有模式的直接简化版，风险极低。**
