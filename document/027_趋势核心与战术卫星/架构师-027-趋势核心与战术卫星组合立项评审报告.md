## 2. 总体结论

027 的立项动因 **成立**、方向 **正确**、在 023→026 序列中是 **结构性创新的必要拐点**。026 以三种不同训练方式均卡在 coverage 26.67%，已排除模型信号问题的残余。027 将瓶颈归因于 **单仓位状态机的结构性限制**，并主张以 **双 Slot 共存 + 独立规则 + 账户冲突裁决** 取代第四次调参——这是 **正确的诊断和正确的出口**。

但 v1.0 方案存在 **四类必须在实施前明确的风险和缺口**：

1. **§11 五项目前尚无架构师裁定**：Core 是否启用 slow_up（025 A3a 失败）、max_drawdown 放宽至 5%、卫星是否允许反向——这些不是细节，是架构决策，决定工程设计范围。
2. **引擎改造量被低估**：当前 `TradingEngine` 硬编码单 `PortfolioState` 与单一 `position`，拆分为双 Slot 意味着决策循环、风险管理、净额结算均需重写——非「加个 dict」级改动。
3. **冲突裁决是新领域**：021 的冲突裁决表解决的是 **同一策略内趋势模块之间的优先级**，027 的冲突表涉及 **两个独立策略的核心持仓冲突**——这是全新的裁决维度，对归因要求更高。
4. **收益预期需要有效边界量化**：§4.3 给出的收益预期区间（+9~11% / DD −3~−5%）缺乏 B0 对照下的分解归因（Core 单独能做什么、Satellite 增量贡献多少）。

因此评审结论为：

```text
027 可以立项，建议有条件批准；
§11 五项前置裁定必须在 Phase 0 开始前由架构师书面给出；
Phase 0「双 Slot 均 FLAT ≡ B0」等价证明是硬门禁——这是整个 027 工程中最重要的单点验证；
架构师支持 DAD（否决跨 Slot 反向开仓）为默认立场；
不建议为 027 降低探索 coverage 或 return 门槛。
```

## 4. 架构合理性评审

### 4.1 Dual-Slot 架构 — **通过，结构性创新正确**

当前架构硬伤审查（`engine.py` L35-76, `portfolio.py` L10-15）：

```text
PortfolioState.position: PositionState  ← 单一持仓（LONG|SHORT|FLAT）
TradingEngine:
  self.portfolio = PortfolioState()      ← 单实例
  决策循环：每 bar 产出一个 TradingAction → apply 到单个 position
```

**这个单一 `position` 字段是 024/025/026 所有 coverage 问题的根**。趋势持仓时无法开战术仓；平仓后 regime 门阻止再入；slow_up 500 bar WATCH 无落仓位。

027 的 dual-Slot 设计：

```text
AccountEquity
├── slot_core: CorePortfolio + CoreRuleEngine + CoreConfig
└── slot_satellite: SatellitePortfolio + SatelliteRuleEngine + SatelliteConfig
    ↓ 账户级 netting + 冲突裁决 + 总风险
```

**这是正确的解**。不是「多加一个仓位」，而是 **策略引擎层面分离两个独立决策实体**。§5.1 的 budget 划分（Core 55~65% / Satellite 25~35%）合理。

**架构评审意见：通过。方向是解决 coverage 的结构性解（而非模型/规则的第十次调参）。**

---

### 4.2 冲突裁决表 — **方向正确，但需从 021 升级**

方案 §5.4 给出 P0~P5 五级冲突裁决。然而 021 的冲突裁决表（`软件设计师-021-TrendBiasBuilder冲突裁决表.md`）解决的是 **同一策略内趋势模块**（crash/slow_up/trend 方向、bias、alignment）之间的优先级——而非 **两个独立持仓方向** 之间的冲突。

027 的冲突是跨持仓的：

```text
Core LONG + Satellite SHORT 同 bar — 同一账户内的对冲仓位
```

这不是「哪个 bias 更强」，而是「是否允许同一账户内同时持多空」。这一问题在 021 从未被讨论。

**架构师默认立场（建议采纳为 027 裁定）**：

```text
DAD（Directional Alignment Default）：禁止跨 Slot 反向开仓。
Core LONG → Satellite 仅可 LONG 或 FLAT；
Core SHORT → Satellite 仅可 SHORT 或 FLAT；
仅当 Core FLAT 时 Satellite 可双向。

例外（须架构师额外书面批准）：
  - Satellite 对冲仓位 ≤ Core 仓位的 20%（净多/净空）
  - Core 进入 hold_mode=TREND 时此例外自动关闭
```

这比方案中的 P1/P2「禁止 Sat 开反向」更 **强**：它不在 bar 级别判断，而是 **账户层面** 禁止净多净空并存。理由：

1. BTCUSDT 1h 样本仅为 ~55 天，方向对冲在这个数据量级下极难验证；
2. 025 损失 023 phase3 的教训：规则过于复杂 → 归因不清；
3. 若 DAD 被回测推翻，实证证据会比理论辩论更有说服力。

**前置裁定建议**：架构师需在 §11 回复中明确接受或拒绝 DAD。若接受，冲突裁决表可大幅简化（P1/P2 移除，P3 仅保留同向/FLAT 分支）。若拒绝，须给出跨 Slot 反向的最大净头寸比。

**架构评审意见：方向正确，但需从 021 单策略冲突升级为 027 跨策略冲突裁决。**

### 4.3 引擎改造复杂度 — **通过，Phase 0 等价证明是关键安全阀**

方案 §6.1 P0-1 要求：「双 Slot 架构空实现，两 Slot 均 FLAT ≡ B0」。

这个等价证明是 **整个 027 项目中最重要的单体工程验证**：

```text

则引擎改造未引入净值计算错误；
在此基础上叠加任何 Core/Satellite 增量 → 归因可信。
```

等价要求需要在 Phase 0 出口审查以下一致性：

| 检查项 | 容差 |
|--------|------|
| `equity_curve` 每 bar | ≤ 1e-8 |
| `trade_count` | = B0 |
| `total_return` | = B0（浮点容差） |
| `max_drawdown` | = B0 |
| `decisions.csv` reason_code | = B0（无新增/缺失） |

**架构评审意见：通过。等价证明是 Phase 0 的最高优先出口条件。**

## 5. 技术可行性评审

### 5.1 引擎改造范围 — **可行，规模大但可控**

| 模块 | 当前 | 027 | 风险 |
|------|------|-----|------|
| `portfolio.py` | `PortfolioState` 单 `position: PositionState` | `CorePortfolio` + `SatellitePortfolio` → `AccountEquity` | 中（净值核算） |
| `engine.py` | 单 `TradingEngine` 决策循环 | `DualSlotEngine` 或引擎内两次 `step` | 中高（决策顺序） |
