## 3. 下一步可执行清单（优先级排序）

### 3.1 【立即执行】prod 日常监控（0 成本，持续）

```
- 每周跑一次 prod_health_monitor.sh，关注：
  · trade_count 相对于基线的漂移（当前 10 vs 8，一个月后若有 12+ 需警惕）
  · risk_rule_violations（当前 0，任何 >0 立即调查）
  · max_drawdown 是否放大（当前 −0.85%，若超过 −1.5% 需检查）

- 每次监控时同步跑未成交归因，记录：
  · WATCH_SLOW bar 占比趋势
  · TEQ reject 是否出现
```

**命令**：

```bash
bash scripts/prod_health_monitor.sh
```

### 3.2 【本周】collect live 数据（1~2 小时）

```
- 收集 prod v1.1.0 上线至今的所有 live trade 记录
- 与同一时段 test backtest 对照：
  · live vs backtest 的 buy/sell match rate
  · live fill price vs signal bar close 的滑点分布
  · live 覆盖的行情模式（crash/trend/slow_up 各占比）
```

目的：建立 live↔backtest 可信度基线。如果 live 连续 2 周 match rate > 90%，再考虑研究版本上线。

### 3.3 【本周】trade_count 漂移专项排查（~30 分钟）

当前 prod test 的 trade_count=10 vs baseline=8（+25%）。虽然可能是样本噪声（55 天窗口波动），但需要确认：

```
- 额外的 2 笔交易来自哪个 channel？
  · 如果来自 crash（正常——6 月下跌增加 crash 触发）
  · 如果来自 TEQ fast leg（正常——模型信号扩展）
  · 如果来自未知 channel → 排查规则

- 额外的 2 笔是否盈利？
  · 若盈利 → 降低关注度
  · 若亏损 → 排查是否误开
```

### 3.4 【1~2 周后】slow_up 专用规则设计（仅当假设清晰时）

**前置条件**：以下问题有明确答案后，再启动 028 立项

```
问题 1：slow_up 的可参与标准应该是什么？
  当前：依赖 leg 确认（leg < 2 根不触发） → 慢涨永远不触发
  候选：价格在 channel 中位以上 + RSI 50~70 + 波动率压缩
  关键：这个标准在 valid 上有正 return 吗？（先不做回测，靠历史行情目视判断）

问题 2：如果 slow_up 开仓，止损和退出逻辑是什么？
  slow_up 不是腿型确认的强趋势 → 不能用 TEQ 的 hold 逻辑
  需要用独立的风险管理参数（max_hold_bars、stop_loss_pct）

问题 3：slow_up 规则的 coverage 预期是多少？
  当前 slow_up 469 bar，如果 10% 能开仓 = 47 根 ≈ 2~3 笔交易
  不能指望 slow_up 显著提升 coverage（最多 +2~3pp）
```

**只有当以上三个问题都有合理答案时，才启动 028 的 Phase 0 探索**。在此之前，不如保持 prod 运行、积累更多 live 数据。
