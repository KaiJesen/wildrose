# BTC 合约预测模型结构改造指导

## 1. 文档目标

本文档面向软件工程师，指导对当前 BTC 合约预测模型进行结构化改造。改造范围仅限模型、数据标签、训练与评估代码，不涉及下单、仓位、资金管理或任何交易执行逻辑。

目标不是推倒现有实现，而是在当前 `transformer_kit/` 的基础上，将模型从“单一未来收益或方向预测”升级为“多任务市场状态预测模型”。

## 2. 当前模型现状

当前代码中主要有两条模型路径：

1. `KlinePatternPredictor`
   - 位置：`transformer_kit/pattern_model.py`
   - 结构：自动切分 `AutoSegmentVQEncoder` + `CausalTransformer` + pooling + future head。
   - 当前输出：未来 `pred_horizon` 步的 log return 或方向相关辅助头。
   - 优点：已有形态 token、VQ code、break logits、raw context 旁路和 horizon-aware head。
   - 问题：输出头和损失已经积累较多实验参数，但任务边界不够清晰，方向、幅度、路径形态、风险状态还没有统一成明确的数据契约。

2. `CPCEncoder + DirectionDecoder`
   - 位置：`transformer_kit/cpc.py`
   - 结构：CPC 自监督预训练编码器 + 轻量方向解码器。
   - 当前输出：未来 `H` 步方向 logits。
   - 优点：结构更轻，最近实验中 `test_step_dir` 约 0.52、`test_cum_dir` 约 0.57，说明存在少量可学习信号。
   - 问题：只覆盖方向预测，缺少收益幅度、波动率、风险状态等任务，难以判断模型是否真的理解市场状态。

## 3. 改造目标架构

建议目标结构如下：

```text
OHLCV / public futures data
        ↓
causal feature builder
        ↓
PatternSequenceDataset
        ↓
shared sequence encoder
        ↓
multi-task market state heads
        ↓
MarketStateOutput
```

模型输出不应是交易动作，而应是市场状态预测：

```text
direction_logits      : [B, H, C] 或 [B, H]
return_pred           : [B, H]
volatility_pred       : [B, H] 或 [B, 1]
risk_logits           : [B, H, R] 或 [B, R]
auxiliary_outputs     : VQ、break、code supervision 等已有辅助信息
```

其中：

- `direction_logits` 用于预测未来每个 horizon 的方向或三分类状态。
- `return_pred` 用于预测未来 log return。
- `volatility_pred` 用于预测未来 realized volatility。
- `risk_logits` 用于预测未来是否进入高波动、插针、趋势加速等风险状态。
- `auxiliary_outputs` 保留当前 VQ、break logits、segment codes 等诊断信息。

## 4. 核心设计原则

1. 保留已有因果约束。
   - `CausalTransformer` 的 causal mask 继续保留。
   - 特征工程必须只使用当前及历史数据。
   - 时间切分继续使用 train / valid / test 的顺序切分，不允许随机全局切分。

2. 将标签生成从训练循环中抽离。
   - 目前方向标签多处以 `(future[..., 0] > 0)` 临时生成。
   - 建议统一由 dataset 或 label builder 生成，避免不同实验脚本标签口径不一致。

3. 将输出结构显式化。
   - 不再让模型只返回 `pred` 和松散的 `aux` 字典。
   - 建议引入 dataclass，例如 `MarketStateOutput`，统一字段和 shape。

4. 优先小模型验证。
   - 默认参数建议保持 `d_model=128`、`n_heads=4`、`n_layers=2`。
   - 在验证集没有稳定提升前，不扩大模型。

5. 强基线约束。
   - 任何新增 head 必须至少对比方向 baseline、LightGBM 或当前 CPC direction baseline。
   - 如果多任务模型没有提升 cumulative direction、IC、volatility MAE 或 calibration，不应合并为默认实验路径。

## 5. 建议新增或修改的模块

### 5.1 新增 `transformer_kit/labels.py`

职责：集中生成未来市场状态标签。

建议包含：

```python
@dataclass(frozen=True)
class MarketStateTargets:
    future_log_ret: torch.Tensor
    direction_label: torch.Tensor
    volatility: torch.Tensor
    risk_label: torch.Tensor
    move_label: torch.Tensor | None = None
```

建议函数：

```python
def build_market_state_targets(
    future_raw_log_ret: np.ndarray,
    *,
    direction_threshold: float,
    risk_quantile_threshold: float,
) -> MarketStateTargets:
    ...
```

标签口径：

- `future_log_ret`: 原始尺度未来 log return。
- `direction_label`: 建议第一版使用三分类，`down / flat / up`。
- `volatility`: 未来窗口 realized volatility。
- `risk_label`: 未来窗口绝对波动、high-low range 或累计回撤超过阈值。

注意：阈值必须只由训练集统计得到，再应用到 valid / test。

### 5.2 修改 `transformer_kit/segment_dataset.py`

当前 `PatternSequenceDataset` 返回：

```python
{
    "ctx_bars": ...,
    "ctx_lengths": ...,
    "future_bars": ...,
    "future_raw_log_ret": ...,
}
```

建议扩展为：

```python
{
    "ctx_bars": ...,
    "ctx_lengths": ...,
    "future_bars": ...,
    "targets": MarketStateTargets 或展开后的 target 字段
}
```

如果暂时不想引入嵌套 dataclass，可以先展开为：

```python
"target_return": torch.Tensor
"target_direction": torch.Tensor
"target_volatility": torch.Tensor
"target_risk": torch.Tensor
```

验收要求：

- 所有 target 的时间窗口必须从 `context_end` 开始，到 `future_end` 结束。
- 所有标准化统计只允许使用 `context_end - 1` 及之前的数据。
- valid / test 不能重新拟合阈值或归一化参数。

### 5.3 修改 `transformer_kit/pattern_model.py`

建议将 `KlinePatternPredictor` 的输出头重构为多任务 head。

当前主要 head：

- `future_head`
- `horizon_head`
- `direction_head`
- `code_supervision_head`

建议目标：

```python
class MarketStateHead(nn.Module):
    def __init__(self, d_model: int, horizon: int, direction_classes: int = 3):
        ...

    def forward(self, pooled, tokens=None, seg_mask=None) -> MarketStateOutput:
        ...
```

建议输出：

```python
@dataclass
class MarketStateOutput:
    return_pred: torch.Tensor
    direction_logits: torch.Tensor
    volatility_pred: torch.Tensor
    risk_logits: torch.Tensor
    aux: dict[str, torch.Tensor]
```

实现策略：

1. 保留 `AutoSegmentVQEncoder`、`CausalTransformer`、`SegmentAttentionPool`。
2. 将 `future_head` 更名或封装为 `return_head`。
3. 新增 `volatility_head`。
4. 新增 `risk_head`。
5. 将 `direction_head` 从累计方向辅助头升级为正式 horizon-level direction head。
6. `code_supervision_head` 暂时保留为辅助，不作为主输出。

第一版不要删除旧参数，可以通过配置开关保持兼容：

```python
use_market_state_head: bool = False
```

但新实验脚本应默认走 market state head。

### 5.4 修改 `transformer_kit/training.py`

建议新增独立损失函数：

```python
def market_state_loss(
    output: MarketStateOutput,
    target: MarketStateTargets,
    *,
    return_weight: float = 0.4,
    direction_weight: float = 0.4,
    volatility_weight: float = 0.15,
    risk_weight: float = 0.05,
) -> tuple[torch.Tensor, dict[str, float]]:
    ...
```

推荐第一版损失：

- return: Huber loss，优先于 MSE，降低极端波动影响。
- direction: cross entropy，三分类时比二分类更稳。
- volatility: log-volatility 上的 Huber loss 或 MSE。
- risk: binary cross entropy 或 focal loss。

不要继续把所有实验损失堆进 `stage3_prediction_loss`。该函数可以保留给旧实验，新多任务路径应使用新的 `market_state_loss`，降低维护复杂度。

### 5.5 新增实验脚本

建议新增：

```text
examples/train_market_state_model.py
```

职责：

1. 加载真实 BTCUSDT 数据。
2. 构造 `PatternSequenceDataset`。
3. 初始化多任务模型。
4. 训练 `market_state_loss`。
5. 输出 metrics 到 `reports/<run_id>/metrics.json`。

该脚本必须遵守项目规则：

- 默认真实数据：`binance_vision / BTCUSDT / 1h / 365d`。
- 不默认使用 synthetic。
- synthetic 只允许用于单元调试。

## 6. 推荐实施顺序

### 阶段 1：整理标签与指标

目标：不改模型主干，只统一标签生成和评估指标。

任务：

1. 新增 `transformer_kit/labels.py`。
2. 修改 `PatternSequenceDataset`，输出统一 target 字段。
3. 新增 market state metrics：
   - `direction_acc`
   - `direction_macro_f1`
   - `cum_direction_acc`
   - `return_ic`
   - `return_mae`
   - `volatility_mae`
   - `risk_auc` 或 `risk_f1`

验收：

- 旧实验仍能跑。
- 新 target 字段在 train / valid / test 上 shape 一致。
- 标签阈值来源可追踪，不能在 valid / test 上重新拟合。

### 阶段 2：新增多任务 head

目标：在 `KlinePatternPredictor` 上新增 market state 输出。

任务：

1. 增加 `MarketStateOutput`。
2. 增加 `MarketStateHead`。
3. 在 `KlinePatternPredictor.forward()` 中通过配置选择旧输出或新输出。
4. 编写最小单元测试，验证 forward shape。

验收：

- 输入 `[B, T, F]` 时输出：
  - `return_pred: [B, H]`
  - `direction_logits: [B, H, 3]`
  - `volatility_pred: [B, H]` 或 `[B, 1]`
  - `risk_logits: [B, H]` 或 `[B, 1]`
- 旧训练脚本不受影响。

### 阶段 3：新增多任务训练路径

目标：建立独立的 market state 训练脚本和 loss。

任务：

1. 新增 `market_state_loss`。
2. 新增 `evaluate_market_state`。
3. 新增 `examples/train_market_state_model.py`。
4. 输出完整 JSON metrics。

验收：

- 能在 CPU 上完成小规模真实数据 smoke run。
- 训练日志中分别展示每个任务的 loss 与指标。
- 至少和当前方向 baseline 对比。

### 阶段 4：对比 CPC 路径

目标：判断是否保留 CPC 作为预训练编码器。

任务：

1. 保留 `CPCEncoder` 作为可选 encoder baseline。
2. 给 `DirectionDecoder` 增加 market state decoder 版本，或将 CPC encoder 接到同一个 `MarketStateHead`。
3. 对比：
   - 无预训练 Transformer
   - CPC 预训练 + finetune
   - VQ segment encoder + market state head

验收：

- 至少跑同一数据区间、同一 horizon、同一指标集。
- 如果 CPC 只提升方向但伤害收益或波动率预测，不能作为默认主干。

## 7. 默认模型配置建议

第一版建议配置：

```text
symbol: BTCUSDT
interval: 1h
days: 365
context_bars: 128
pred_horizon: 5
d_model: 128
n_heads: 4
n_layers: 2
dropout: 0.1
direction_classes: 3
loss:
  return_weight: 0.4
  direction_weight: 0.4
  volatility_weight: 0.15
  risk_weight: 0.05
```

不要一开始增加到 `d_model=256+` 或更深层数。当前报告显示方向信号仍弱，扩大参数更可能放大过拟合。

## 8. 泄漏风险检查清单

每次改动必须检查：

- rolling z-score 是否只使用 trailing window。
- direction / risk 阈值是否只从 train split 估计。
- future target 是否严格从 `context_end` 之后开始。
- 多 horizon 标签是否没有包含输入窗口最后一根 bar。
- valid / test 是否没有参与 scaler、quantile、class weight 的拟合。
- 多标的预训练时，decode symbol 的 valid / test 是否没有被用于预训练采样。

## 9. 最小验收标准

合并前至少满足：

1. 单元级 shape 测试通过。
2. 真实 BTCUSDT 数据 smoke run 通过。
3. `metrics.json` 包含各任务指标。
4. 与当前 CPC direction baseline 有同口径对比。
5. 文档中记录本次实验的标签阈值、horizon、数据区间和 split。

推荐通过标准不是单个 accuracy 提升，而是多个指标不相互矛盾：

- `cum_direction_acc` 不低于当前 baseline。
- `return_ic` 为正且在 valid / test 同方向。
- `volatility_mae` 优于 naive rolling volatility baseline。
- `risk_f1` 或 `risk_auc` 高于类别频率 baseline。

## 10. 不建议做的修改

暂不建议：

- 直接删除 VQ / CPC 旧路径。
- 将交易动作作为标签。
- 使用随机切分评估时间序列。
- 只追求 `all5_dir`，忽略逐 horizon 稳定性。
- 继续在 `stage3_prediction_loss` 中叠加更多实验参数。
- 在没有强基线对比前扩大模型规模。

## 11. 工程结论

当前模型已经具备较好的研究组件，但结构上需要从“实验参数累积”转为“明确的多任务市场状态建模”。推荐的工程路线是：

```text
先统一标签和指标
→ 再新增 MarketStateHead
→ 再新增独立 market_state_loss
→ 最后比较 VQ 主干、CPC 主干和轻量 Transformer 主干
```

这样可以最大程度复用现有代码，同时降低模型输出、损失函数和实验脚本之间的耦合。
