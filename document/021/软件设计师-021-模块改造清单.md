## 2. 改造总原则

021 期不建议大规模推翻已有模块，而应采用：

```text
新增统一偏置层
-> 收拢当前分散规则
-> 逐步替换 RuleEngine 内部的直接耦合判断
```

因此改造策略应是：

1. **新增对象优先**
2. **兼容旧逻辑**
3. **按链路分层替换**
4. **每阶段都能回测验证**

## 3. 新增模块建议

## 3.1 `trading_system/trend_bias.py`

### 目标

新增统一趋势偏置层，汇总趋势模块结论并对交易模块输出连续偏置与顺逆势分层抑制结果。

### 建议新增内容

- `TrendBiasContext`（schema 见最终字段契约）
- `TrendBiasBuilder`（优先级见冲突裁决表）
- `alignment_score_long` / `alignment_score_short`
- `counter_level_long` / `counter_level_short`
- `open_bias_*` / `size_bias_*` / `hold_bias_*` / `exit_bias_*`
- `risk_tolerance_bias_*`（与 open_bias 分离）

### 原因

当前 `RuleEngine` 直接消费：

- `TrendContext`
- `TrendSignal`
- `SlowTrendContext`
- `SegmentContext`
- `CrashContext`

耦合过高，优先级分散，需要一个统一中间层。

---

## 3.2 可选：`trading_system/bias_context.py`

若不想把偏置逻辑全部放在一个文件中，可拆成：

- `trend_bias.py`
- `bias_context.py`

其中：

- `TrendBiasContext` 只定义数据结构
- `TrendBiasBuilder` 负责构建

## 4. 现有模块改造清单

## 4.1 `trading_system/engine.py`

### 当前职责

- 拉通所有上下文
- 调用 RuleEngine / RiskManager / PositionSizer / ExecutionEngine
- 维护组合状态和持仓生命周期

### 建议改造

1. 新增 `self.trend_bias_builder`
2. 在每根 bar 上生成 `TrendBiasContext`
3. 将 `TrendBiasContext` 作为显式参数传入 `RuleEngine`
4. 在 logger 里记录（对齐冻结字段）：
   - `open_bias_long` / `open_bias_short`
   - `size_bias_long` / `size_bias_short`
   - `counter_level_long` / `counter_level_short`
   - `allow_open_long` / `allow_open_short`
   - `reason_codes`
   - `decision_scope`

### 目标

让 `engine.py` 成为：

```text
上下文收集器 + 偏置构建器 + 生命周期协调器
```

而不是只做多模块的直接拼装。

---

## 4.2 `trading_system/rules.py`

### 当前问题

- 开仓逻辑大量直接依赖 `signal`
- 趋势影响主要靠 block / upgrade / hold 的局部逻辑
- 多个趋势模块结论直接混杂在单个 `decide()` 里

### 建议改造

#### 第一阶段

保留原有逻辑，但增加新参数：

- `trend_bias: TrendBiasContext | None`

并先用于：

1. 开仓阈值修正
2. 逆向级别分层抑制（而非一刀切阻断）
3. 初始仓位偏置
4. 趋势持仓时长偏置

#### 第二阶段

把以下逻辑从“直接读 trend / seg / slow / crash”改为“先读 trend_bias”：

- `BLOCK_COUNTER_TREND_LONG`
- `BLOCK_COUNTER_TREND_SHORT`
- `UPGRADE_TO_TREND_LONG`
- `UPGRADE_TO_TREND_SHORT`
- `ADD_TREND_CONTINUATION`
- `HOLD_TREND_CONTINUATION`
- `CLOSE_TREND_EXIT_CONFIRMED`

#### 第三阶段

重构 `decide()` 主流程：

```text
先读 bias
-> 再解释 signal
-> 再判断 open / hold / add / reduce / close
```

而不是“先看短信号，再让趋势修正”。

---

## 4.3 `trading_system/sizing.py`

### 当前职责

- 根据信号强弱、风控配置计算目标仓位

### 建议改造

增加趋势偏置输入：

- `trend_size_bias`
- `counter_trend_penalty`
- `trend_add_scale`

建议把仓位计算改为：

```text
base_size
  * signal_strength_scale
  * trend_size_bias
  * counter_trend_scale
  * risk_penalty
```

### 目标

让“顺趋势开大一点、轻逆向少开一点、中逆向显著缩小、强逆势近似禁止”成为统一机制，而不是若干特例。

---

## 4.4 `trading_system/risk.py`

### 当前职责

- 风险边界、熔断、冷却、亏损限制

### 建议改造

1. 新增 `RiskBudget` dataclass（见偏置数学公式说明 §5，含 `allow_open_*` / `allow_add_*` / `allow_reverse`）
2. 新增 `RiskEvent`（P0：`force_exit` 等）
3. 输出 `remaining_position_ratio`、`worst_case_loss_ratio` 及分动作 reject reason
4. 验收记录：`trend_add_candidate_count`、`trend_add_risk_evaluated_count`、`trend_add_rejected_by_risk_count`

### 目标

让风险管理回答：

```text
不是“规则想不想加仓”，
而是“规则想加仓且风险也允许加仓”。
```

---

## 4.5 `trading_system/trend_signal.py`

### 当前职责

- 输出逐 bar 趋势方向、强度、阶段、确认、破坏

### 建议改造

不建议 021 大改内部算法，主要建议：

1. 明确哪些字段供 `TrendBiasBuilder` 使用
2. 明确哪些字段仍只用于诊断
3. 减少 RuleEngine 直接访问其细碎字段

### 目标

将其定位稳定为：

```text
趋势微观状态提供者
```

而不是直接在规则层承担全部趋势决策。

---

## 4.6 `trading_system/trend_segment.py`

### 当前职责

- 提供趋势腿、子阶段、是否避免逆势、是否可升级、是否腿结束等区间上下文

### 建议改造

1. 明确 `SegmentContext` 中哪些字段会进入 `TrendBiasContext`
2. 强化：
   - `should_avoid_counter`
   - `should_hold_trend`
   - `leg_progress_ratio`
   - `sub_phase`
3. 让腿内阶段更直接映射到生命周期：
   - `PULLBACK`
   - `ACCELERATION`
   - `EXHAUSTION`
   - `LEG_END`

### 目标

让 `TrendSegmentEngine` 成为趋势偏置层中最重要的“区间锚点”。

---

## 4.7 `trading_system/slow_trend.py`

### 当前职责

- 识别稳定慢涨行情

### 建议改造

1. 保留算法不大改
2. 但不再让它直接绕过主框架
3. 让它通过 `TrendBiasContext` 输出：
   - `slow_up_active = true`
   - `hold_bias_long` 上调
   - `allow_add_long` 上调（或保持 true）

### 目标

从“慢涨特化分支”改为“多头趋势偏置增强器”。

---

## 4.8 `trading_system/crash.py`

### 当前职责

- 识别模型失效下的崩跌

### 建议改造

1. 保留 crash 作为空头特化场景
2. 但它不应再独立定义完整持仓逻辑
3. 建议通过 `TrendBiasContext` 输出 P1 紧急偏置（见冲突裁决表 §5）：
   - `crash_short_active=true`
   - `allow_open_long=false`
   - `open_bias_short` 上调
   - **不**独立绕过 RiskBudget 开空

### 目标

将 crash 从“独立状态机”逐步收敛到“趋势偏置极端特化器”。

---

## 4.9 `trading_system/logger.py`

### 建议改造

新增记录字段（对齐冻结 schema，不用 `bias_*` 别名）：

```text
trend_bias.open_bias_long
trend_bias.size_bias_long
trend_bias.hold_bias_long
trend_bias.counter_level_long
trend_bias.allow_open_long
trend_bias.reason_codes
risk_budget.allow_open_long
risk_budget.allow_add_long
risk_event.force_exit
```

### 目标

后续回测复盘时，能回答：

```text
这笔单为什么被压小？
为什么被持有更久？
为什么这次逆势单被拦掉？
```

## 4.10 `trading_system/config.py`

### 建议改造

新增配置组，例如：

```text
TrendBiasConfig
```

```text
TrendBiasConfig:
  enabled: bool
  decision_scope: observe | open_only | open_size | full
  disable_legacy_trend_rules: bool
  long_open_relax / short_open_relax
  light_counter_tighten / medium_counter_tighten / hard_counter_block
  trend_size_boost / light_counter_size_penalty / medium_counter_size_penalty
  trend_hold_extend_bars / counter_hold_shorten_bars
  trend_exit_vote_bonus / counter_exit_vote_penalty
```

### 目标

把趋势偏置从“散落在多个模块中的隐式行为”变成“集中配置、可回测调优的显式行为”。

## 6. 可延后改造项

以下内容不建议作为 021 第一阶段的必须项：

1. 重训新的市场状态模型
2. 新增更多 best_point 标签版本

原因：

021 的主任务是**框架收敛**，不是再扩展输入源。

## 7. 分阶段实施建议

与架构师评审一致；详细 gates 见 `软件设计师-021-A-B回测验收表.md`。

## Phase 0：契约冻结

- 字段契约、冲突表、偏置公式入仓
- **不改 `rules.py` 主流程**

## Phase A：观察模式（`decision_scope=observe`）

目标：

- 新增 `TrendBiasContext` + `TrendBiasBuilder`
- `engine.py` 生成 bias；logger 记录
- `rules.py` **不消费** bias
- 回测与 v020 交易结果完全一致

## Phase B：开仓 gate（`open_only`，`disable_legacy_trend_rules=true`）

目标：

- bias 接管 `allow_open_*`、`open_bias_*`
- 旧 trend block 关闭或降为诊断

## Phase C：仓位（`open_size`）

目标：

- `size_bias_*` + `RiskBudget`
- 验收 `max_position_ratio_observed <= 20%`

## Phase D：全链路（`full`）

目标：

- `allow_add_*`、`hold_bias_*`、`exit_bias_*`、`force_exit_*`
- `legacy_trend_direct_read_count` 显著下降

---

## 8. 验收建议

完整 gates 见 `软件设计师-021-A-B回测验收表.md`。摘要：

| 类别 | 关键指标 |
|------|----------|
| 架构 | `legacy_trend_direct_read_count`（Phase D 下降） |
| 可解释 | `bias_reason_codes_coverage` |
| 逆势 | `hard_counter_open_count` ≈ 0 |
| 风险 | `max_position_ratio_observed` ≤ 20% |
| 风险 | `trend_add_candidate_count` / `trend_add_risk_evaluated_count` | 有加仓候选时须全量评估 |
| 风险 | `trend_add_rejected_by_risk_count` | 可统计；可为 0，须有 reason 分布 |
| 回归 | `max_drawdown` 不显著劣化；`trade_count` 不因重复生效暴增 |

分版本回测：`v020_baseline` → `v021_observe` → `v021_open_bias` → `v021_open_size_bias` → `v021_full_bias`。

---

## 9. 结论

021 期的工程重点不是“新增更多功能”，而是：

```text
让已有功能变得层级清晰、职责清晰、优先级清晰。
```

具体来说：

- `trend_signal.py`、`trend_segment.py`、`slow_trend.py`、`crash.py` 继续保留
- 021 新增 `trend_bias.py` 作为统一调度层
- `engine.py`、`rules.py`、`sizing.py`、`risk.py` 按新偏置层逐步重构

这样后续继续优化收益率、长趋势覆盖率和回撤控制，才会有更稳定的工程基础。
