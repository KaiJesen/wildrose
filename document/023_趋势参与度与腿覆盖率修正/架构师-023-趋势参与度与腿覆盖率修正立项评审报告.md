## 1. 评审说明

| 项 | 内容 |

| 流水号 | 023 |
| 评审对象 | `软件设计师-023-趋势参与度与腿覆盖率修正立项方案.md` |
| 评审维度 | 业务可行性、架构合理性、技术可行性 |
| 参考材料 | `document/019_慢速上涨趋势捕捉修正/`、`document/020_趋势区间分段框架/`、`document/021_趋势优先交易框架/`、`document/022_趋势模块质量与性能修正/`、`backtest/v022_trade_points_plot/`、`backtest/v022_system_test/` |


## 2. 总体结论

023 的立项方向 **业务上成立、架构上必要、技术上基本可行**。

022 已经证明趋势质量修正可以在不伤害风控的前提下小幅提升收益，但 v022 的核心矛盾仍然存在：

| 指标 | v022 表现 | 业务含义 |
|------|-----------|----------|
| `trade_count = 8` | 1308 bar 仅 8 次开仓 | 系统过度保守 |
| `leg_coverage_ratio = 17.3%` | 44 条确认腿大多未参与 | 趋势识别未转化为持仓 |
| `long_trend_capture_ratio = 0%` | 无多头趋势仓 | 慢涨行情完全漏掉 |
| `WATCH_SLOW_UPTREND = 534` | 检测慢涨但不开仓 | 019 能力未进入交易链路 |
| `trend_upgrade_count = 0` | crash / probation 未升级 | 018/021 生命周期目标未落地 |

因此，023 从“模块质量”转向“趋势参与度与腿覆盖率”是合理的下一阶段。

但当前立项材料仍有三个会直接影响实施正确性的阻塞问题：

1. **验收口径存在收益 / 回撤符号与硬门槛不一致，可能允许业务退化通过。**
2. **`leg_coverage_ratio` 等核心 KPI 未定义精确计算口径，容易被小仓 probe 或短暂重叠刷高。**
3. **新增 `OPEN_*_TREND_QUALIFIED` 直接落在 `rules.py`，可能绕开 021/022 已冻结的趋势偏置、Chop Guard 与风险预算边界。**

评审结论：

```text
023 可以立项；
但进入 Phase 1 实施前，应先完成一次小幅立项修订；
修订后可进入 Phase 0 诊断与基线构建。
```

## 3. 业务可行性评审

### 3.1 业务问题定位准确

023 把问题定义为：

```text
不是看不见趋势，而是趋势已识别后没有转化为可持续持仓。
```

这个判断与 v022 trade points 数据一致。尤其是：

- 慢涨检测 534 bar，但 `slow_up_open_count=0`；
- `missed_confirmed_trend_bars=248`，说明确认趋势下仍大量空仓；
- 收益集中于少数 crash / short trend 交易，策略脆弱；
- 多头趋势捕捉为 0%，说明系统仍有“下跌补救强、多头趋势弱”的结构偏差。

**评审意见：通过。**

---

### 3.2 北极星指标方向正确，但需防止“覆盖率刷分”

方案把硬 KPI 从单一收益扩展为：

```text
leg_coverage_ratio
slow_up_open_count
long/short_trend_capture_ratio
missed_confirmed_trend_bars
trade_count
```

这是业务上正确的，因为 023 的目标不是只提高胜率，而是提高行情参与度。

但当前 `leg_coverage_ratio` 没有定义：

1. 一笔 1 bar、0.03 仓位的 probe 是否算覆盖整条腿？
2. 覆盖率按“腿数量”还是“腿内 bar 数”统计？
3. 多笔短进短出是否算同一条腿覆盖？
4. 亏损参与是否与盈利参与同权？
5. 只覆盖腿尾部是否与覆盖腿中段同权？

如果不定义，系统可能通过增加小仓试探来提高覆盖率，但业务收益没有改善。

**必须修正：**

建议将覆盖指标拆成三层：

| 指标 | 建议定义 |
|------|----------|
| `leg_count_coverage_ratio` | 有持仓重叠的 confirmed leg 数 / confirmed leg 总数 |
| `leg_bar_coverage_ratio` | 持仓 bar 与 confirmed leg bar 的交集 / confirmed leg 总 bar |
| `leg_pnl_capture_ratio` | 系统在 confirmed leg 内获得的净收益 / 理想腿收益或基准腿收益 |

并设置最小有效覆盖：

```text
effective_leg_covered =
  overlap_bars >= min(3, leg_bars * 0.15)
  AND avg_position_ratio >= min_effective_position_ratio
```

---

### 3.3 收益与回撤门槛存在不一致

材料中同时出现：

| 位置 | 表述 |
|------|------|
| 北极星 Phase 1 | `total_return ≥ 9.31%（不退化）` |
| Phase 4 硬门槛 | `test: total_return ≥ v022 × 0.95` |

这两者冲突：后者允许收益从 9.31% 降到约 8.84%，但前者要求不退化。

回撤也存在符号歧义：

```text
max_drawdown（test） ≤ -1.5% / ≤ -2.0%
```

若 `max_drawdown` 用负数表示，`≤ -2.0%` 实际表示“更差或等于 -2%”，与“不超过 2% 回撤”的业务语义相反。Phase 4 又写 `max_drawdown ≥ -2.0%`，口径不一致。

**必须修正：**

建议统一为：

```text
total_return_phase1 >= v022_total_return
total_return_phase2 >= 12% 或 excess_return 相对 v022 提升 >= 20%
max_drawdown >= -2.0%          # 若沿用负数口径
abs_max_drawdown <= 2.0%       # 若新增正数口径
```

同时将 `total_return ≥ v022 × 0.95` 降级为“诊断容忍线”，不得作为最终硬门槛。

---

### 3.4 参与度提升有业务价值，但必须绑定风险预算

023 准备提升开仓率、趋势资格开仓、慢涨 probe、再入场与加仓。这些动作符合“吃完整趋势腿”的业务目标，但也会改变策略风险形态：

- 交易次数从 8 提升到 15~25；
- 开仓阈值下调；
- 慢涨低波动行情允许 probe；
- 同 regime 再入场从禁止变为允许；
- participation floor 会在长时间空仓后放松门槛。

这些设计必须以 `RiskBudget` 为硬前置，否则容易从“低参与”直接滑向“过度交易”。

**评审意见：有条件通过。**

## 4. 架构合理性评审

### 4.1 从 022 到 023 的层级切换合理

022 的定位是：

```text
趋势模块质量与性能修正
```

023 的定位是：

```text
趋势识别结果如何进入开仓、持仓、加仓、再入场生命周期
```

这个分层合理，避免继续在 022 中混入交易执行层调参，也符合 021 “趋势背景主导、短信号执行细节”的架构方向。

**评审意见：通过。**

---

### 4.2 `OPEN_*_TREND_QUALIFIED` 不应成为绕过 TrendBias 的新分支

材料建议新增：

```text
OPEN_LONG_TREND_QUALIFIED
OPEN_SHORT_TREND_QUALIFIED
```

业务上这是必要能力，但架构上不能让它退化为 `rules.py` 中的新特判。021/022 已经明确：

```text
趋势模块 → TrendBiasContext / reason_codes / RiskBudget → RuleEngine
```

如果 023 直接在 `rules.py` 中读取 `trend_signal + segment` 并开仓，会重新制造 021 之前的问题：

- 多模块直接耦合；
- 与 `counter_level_* == HARD_BLOCK` 冲突；
- 与 Chop Guard / false leg 过滤重复或绕开；
- 与 `risk_budget.allow_open_*` 边界不一致；
- 回测难以归因。

**必须修正：**

建议把趋势资格开仓表达为统一的 entry intent：

```text
TrendBiasBuilder / TrendEntryQualifier
  -> trend_entry_qualified_long / short
  -> reason_codes: TREND_QUALIFIED_LONG_FAST_UP, TREND_QUALIFIED_SHORT_FAST_DOWN

RuleEngine
  -> 读取 trend_bias + entry_qualification
  -> 再检查 signal threshold + risk_budget.allow_open_*
```

若不想扩展冻结 `TrendBiasContext` 字段，可将资格结果作为 **非冻结诊断/决策辅助对象**：

```python
TrendEntryQualification(
    allow_trend_entry_long: bool,
    allow_trend_entry_short: bool,
    side: str,
    reason_codes: list[str],
)
```

但禁止 `rules.py` 重新直接拼接所有趋势模块。

---

### 4.3 Participation Floor 架构风险较高

方案提出：

```text
连续 48 bar 无开仓，且滚动 leg_coverage_ratio 低于 20%
→ 临时下调 open_edge_threshold 15%
```

这个机制能避免长期空仓，但它本质上是“为了参与而参与”的全局压力阀。如果没有强约束，容易在低机会环境中触发错误交易。

**建议修正为 P2/可选项，不作为 Phase 3 默认必开能力。**

若保留，需要硬约束：

```text
仅当存在 confirmed leg 或 stable slow_uptrend 时可触发；
不得覆盖 HARD_BLOCK / CHOP_HARD；
不得放宽 p_risk / max_position_ratio / RiskBudget；
每 7d 最多触发 N 次；
只允许 probe 仓，不允许 full 仓；
必须记录 RELAX_PARTICIPATION_FLOOR 并单独统计 PnL。
```

---

### 4.4 `RiskManager` 职责边界需要调整表述

材料在 Phase 2 中写：

```text
RiskManager 读取 segment_context.leg_progress_ratio 与 trend_signal.phase
```

从 021/022 架构看，`RiskManager` 应主要负责硬风险预算，不应成为趋势生命周期裁决器。`leg_progress_ratio` 和 `trend_signal.phase` 更适合由：

```text
TrendBiasBuilder / RuleEngineActionResolver
```

转化为：

```text
hold_bias
exit_bias
time_exit_permission
```

然后 `RiskManager` 只检查这些动作是否突破仓位、保证金、回撤、最坏损失边界。

**建议修正：**

将“分 hold_mode 时间退出、leg_progress 联动”从 `trading_system/risk.py` 主责中移出，改为：

| 模块 | 职责 |
|------|------|
| `trend_bias.py` / lifecycle helper | 计算 hold/exit 偏置 |
| `rules.py` / ActionResolver | 决定 TIME_EXIT 是否被允许 |
| `risk.py` | 检查仓位、保证金、回撤、强制退出 |

---

### 4.5 与 022 的依赖关系需要改成显式 gate

材料写：

```text
023 假设 Phase 1 模块 gates 已 PASS，专注参与度
```

但 023 又计划使用：

- Chop Guard；
- confirmed trend；
- `SLOW_UP_LEG`；
- false leg 过滤；
- `counter_level_*`。

如果 022 的模块质量 gate 没有固定版本、配置 hash 和评估报告，023 的参与率改动会混入趋势模块变化，导致归因不清。

**建议增加前置 gate：**

```text
023 Phase 0 只能基于已冻结 v022 推荐配置；
记录 trend_quality_config_hash；
记录 TrendSignal / Segment / SlowUp detector 版本；
若 022 模块后续变化，023 A/B baseline 必须重跑。
```

## 5. 技术可行性评审

### 5.1 Phase 0 诊断工具可行，且应先做

`eval_participation.py`、`participation_overlay`、023 baseline config 的交付路径清晰，风险低，建议作为首个 PR。

但 Phase 0 必须补充：

| 项 | 要求 |
|----|------|
| 数据窗口 | 固定 valid/test 起止时间、bar 数、交易成本配置 |
| 配置 hash | v022 baseline 与 v023 baseline 都需记录 |
| 指标定义 | coverage / capture / missed bars 需有公式 |
| 输出文件 | `REPORT_023_BASELINE.md` + metrics json |

**评审意见：通过，需补充公式。**

---

### 5.2 Phase 1 慢涨与趋势资格开仓可行，但需先小仓观察

启用 `slow_up_position.enabled=true` 是低成本改动，且有 019 设计基础。趋势资格开仓也符合 023 目标。

建议落地顺序：

```text
Phase 1a: 仅 slow_up watch→probe，小仓，valid 观察
Phase 1b: crash→trend upgrade
Phase 1c: trend-qualified entry
```

不要在一个 PR 同时打开三条新开仓路径，否则出现收益或回撤变化时无法归因。

---

### 5.3 Phase 2 加仓与再入场技术可行，但回撤风险高

趋势延续加仓与同 regime 再入场能提升 capture，但也是最容易放大回撤的部分。

建议增加硬限制：

```text
max_reentry_per_leg
max_add_count_per_leg
min_bars_after_exit
no_reentry_after_exhaustion
no_add_after_leg_progress_gt_0.6
```

并分开统计：

```text
reentry_candidate_count
reentry_allowed_count
reentry_pnl
trend_add_candidate_count
trend_add_allowed_count
trend_add_pnl
```

---

### 5.4 Phase 3 动态阈值应晚于 Phase 1/2

Regime-aware threshold 与 OOS 分布对齐是合理方向，但它会改变模型信号解释方式，应放在慢涨通道、趋势开仓、持仓生命周期之后。

评审同意当前 Phase 3 排序。

但 `Participation Floor` 建议降级为实验项，不能作为推荐 config 的默认开启项，除非 A/B 能证明其单独贡献为正且不增加假腿交易。

---

## 6. 必须修订项

### 阻塞项一：统一收益 / 回撤硬门槛

修订要求：

1. 删除或降级 `total_return ≥ v022 × 0.95`；
2. 统一 `max_drawdown` 负数口径；
3. 明确 Phase 1、Phase 2、最终推荐 config 的收益底线。

建议写法：

```text
Phase 1: total_return >= v022_total_return * 0.95（仅探索线）
推荐 config: total_return >= v022_total_return
软目标: total_return >= 12% 或 excess_return 相对 v022 提升 >= 20%
max_drawdown >= -2.0%
```

---

### 阻塞项二：冻结核心参与度指标公式

修订要求：

至少定义：

```text
leg_count_coverage_ratio
leg_bar_coverage_ratio
leg_pnl_capture_ratio
slow_up_watch_to_open_ratio
missed_confirmed_trend_bars
effective_leg_covered
```

并明确：

- 是否按 valid/test 分开统计；
- 是否过滤 `CHOP_HARD`；
- probe 仓是否计入 coverage；
- 多笔交易覆盖同一腿如何去重；
- 亏损覆盖是否单列统计。

---

### 阻塞项三：趋势资格开仓必须走统一偏置 / 风险预算链路

修订要求：

`OPEN_*_TREND_QUALIFIED` 不得直接绕过：

```text
counter_level_*
HARD_BLOCK
CHOP_HARD
risk_budget.allow_open_*
max_position_ratio
reason_codes
```

建议新增 `TrendEntryQualification` 或等价辅助对象，而不是在 `rules.py` 中重新直接读取多个趋势模块。

---

## 7. 高优先级修订建议

1. **Participation Floor 降级为可选实验项**，默认关闭；通过独立 A/B 后再进入推荐配置。
2. **明确 022 前置 gate**：v022 推荐配置、趋势模块版本、eval 报告与配置 hash 必须固定。
3. **将 `RiskManager` 的趋势生命周期职责移回 bias / rules 层**，RiskManager 保持硬风险边界。
4. **慢涨开仓需增加假突破指标**：`slow_up_false_entry_count`、`slow_up_probe_pnl`、`slow_up_stop_count`。
5. **再入场与加仓必须按 leg 去重与限次**：否则 trade_count 达标可能来自同一腿内频繁噪声交易。
6. **Phase 1 拆小 PR**：slow_up、crash upgrade、trend-qualified entry 分开打开，避免归因混乱。

## 8. 建议新增验收表

| 类别 | 指标 | 最低要求 |
|------|------|----------|
| 业务 | `total_return` | 推荐 config 不低于 v022 |
| 风险 | `max_drawdown` | `>= -2.0%` |
| 参与 | `leg_bar_coverage_ratio` | Phase 1 ≥ 25%，最终 ≥ 35% |
| 参与 | `leg_count_coverage_ratio` | Phase 1 ≥ 35%，最终 ≥ 50% |
| 慢涨 | `slow_up_watch_to_open_ratio` | Phase 1 > 0，最终需有正向 PnL |
| 多头 | `long_trend_capture_ratio` | Phase 1 ≥ 15%，最终 ≥ 30% |
| 空头 | `short_trend_capture_ratio` | Phase 1 ≥ 45%，最终 ≥ 55% |
| 质量 | `false_leg_entry_count` | 不高于 v022 基线显著水平 |
| 风险 | `hard_counter_open_count` | 不得上升 |
| 归因 | 每个新通道 PnL | slow_up / trend_qualified / crash_upgrade / reentry / add 分开统计 |

## 9. 最终评审结论

023 项目 **建议批准立项，但需先完成立项修订后再进入实施**。

批准理由：

1. 业务痛点明确：v022 低参与、低腿覆盖、慢涨零参与已经被数据证实；
2. 项目方向与 018~022 的演进一致：从识别质量转向持仓生命周期与参与度；
3. 技术改造点明确：慢涨通道、趋势资格开仓、crash 升级、腿对齐持有、再入场、动态阈值均可分阶段实现；
4. 若指标定义与风险边界补齐，023 有机会把 v022 的“高精度低参与”升级为“趋势腿驱动的中等参与”。

暂不建议直接进入 Phase 1。建议先完成：

```text
Phase 0.1：立项修订
  - 统一收益 / 回撤门槛
  - 冻结参与度指标公式
  - 明确 trend-qualified entry 的架构链路
  - 固定 022 前置 gate 与配置 hash

Phase 0：诊断与基线
  - eval_participation.py
  - participation_overlay
  - REPORT_023_BASELINE.md
```

完成 Phase 0.1 后，023 可进入 Phase 0；Phase 0 指标与 v022 复现一致后，再分 PR 打开 slow_up、crash upgrade、trend-qualified entry。
