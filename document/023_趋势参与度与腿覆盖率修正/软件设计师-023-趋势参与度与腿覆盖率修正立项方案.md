## 4. 根因分析

### 4.1 架构层：「识别」与「参与」脱节

```text
趋势模块（signal / slow_up / segment / crash）→ 大量正向诊断输出
RuleEngine 开仓门 → 仍以短周期模型 edge/p_up 为主闸门
持仓生命周期 → 升级/再入场/腿对齐持有 多数关闭或未接线
```

系统已具备 016~021 的「趋势大脑」，但 **执行层默认策略仍是「极度选择性短线」**：只有强模型信号或 crash 盲跌才开仓，确认趋势本身不构成开仓资格。

### 4.2 慢涨长行情：检测启用、交易关闭（P0）

**直接原因**：`configs/trading_rule_v022_trend_quality_0062e.json` 中

```json
"slow_up_position": { "enabled": false, ... }
```

**行为链**：

1. `SlowUptrendDetector` 在 534 bar 输出 `is_slow_uptrend=1`
2. `_can_open_slow_up()` 首行检查 `slow_up_position.enabled` → 恒 false
3. `rules.py` 落入 `WATCH_SLOW_UPTREND` 分支 → **只观察不开仓**
4. 其中 157 bar 已达 `is_stable_slow_uptrend=1`，仍无法 `OPEN_LONG_SLOW_TREND`

这是 **019 已设计但未在 v022 生产配置中启用的功能开关问题**，不是检测器失效。

### 4.3 中快行情：开仓门槛与持有上限双重压缩（P0）

**开仓侧**（空仓 `HOLD_NO_ENTRY` 645 bar 统计）：

| 条件 | 阈值 | 空仓实际（p90） | 影响 |
|------|------|-----------------|------|
| edge | ≥ 0.04 | 0.033 | 绝大多数 bar 不满足 |
| p_up / p_down | ≥ 0.34 | ~0.33 | 概率头偏弱 |
| pred_cum_ret_5 | > 0（多头） | 均值 0.09 但常与 edge 不同步 | 多头更难共振 |
| risk_ok | p_risk ≤ 0.45 | 均值 0.42 | 边缘 bar 被挡 |

通过双阈值的空仓 bar：多头 **4**、空头 **2**（全样本仅 8 次开仓与之吻合）。

**持有侧**：

- `rule.max_hold_bars = 8`：普通仓时间退出过短（案例：2026-05-24 多单 +1.7% 在 8 bar 被 `CLOSE_MAX_HOLD_BARS` 截断）
- `trend_position.allow_crash_trend_upgrade = false`：3 笔 crash 空无一升级，`HOLD_CRASH_TREND_CONFIRMING` 40 bar 空转
- `same_regime_once = true`（crash_short）：同 regime 禁止再入，错过延续腿
- 确认趋势空仓 253 bar 中：139 `HOLD_NO_ENTRY`、109 `WATCH_SLOW_UPTREND`、仅 3 次开仓尝试

### 4.4 分段与偏置：守门过严、激励不足（P1）

- `trend_segment.counter_trend_block = true`：逆势开仓被挡（合理），但 **顺势开仓没有对应 relax**
- `TrendBias` 已计算 `slow_up_size_boost / hold_boost`，却因 `slow_up_position.enabled=false` 无法作用于开仓
- `trend_add_*` 全为 0：趋势延续不加仓，快行情「吃不全」
- 确认趋势下 `trend_direction=DOWN` 有 145 bar 空仓，仅 2 次 `OPEN_SHORT_CRASH`；模型空单门槛过高是主因

### 4.5 模型与任务失配（P2，023 内仅定义接口）

0062e 市场状态模型优化目标是 **短周期状态分类**，不是 **趋势腿 ROI 最大化**：

- 空仓 edge 分布集中在 0±0.02，与 `open_edge_threshold=0.04` 系统性失配
- 慢涨时 `p_up≈0.31` 长期低于 `open_prob_threshold=0.34`
- best-point 在 v022 配置中未启用门控，**不是当前主因**；但 017 标签偏短，对未来 exit 扩展仍是风险

### 4.6 根因树（压缩）

```text
收益率不满意
├── 参与率极低（8 trades / 1308 bars）
│   ├── 慢涨：slow_up_position.enabled=false → 534 bar 仅 WATCH
│   ├── 标准开仓：edge/p_up 阈值高于模型 OOS 分布
│   └── 确认趋势空仓：无「趋势资格开仓」通道
├── 腿覆盖率低（17%）
│   ├── 开仓少 → 先天覆盖不足
│   ├── crash 不升级 trend（allow_crash_trend_upgrade=false）
│   ├── max_hold_bars=8 截断中周期
│   └── 无 same_regime 再入场 / 趋势加仓
└── 收益集中度高
    └── 依赖单笔 crash + 单笔 trend short，策略脆弱
```

### 5.2 设计原则

1. **趋势资格优先于模型强信号**：确认腿 / 稳定慢涨 / 确认 crash 应能触发「小仓 probe → 升级 full」路径
2. **分 regime 阈值，而非全局一套 edge**：慢涨、趋势延续、crash 使用不同开仓契约
3. **持有跟腿走，不跟 bar 走**：`hold_bias` / `time_exit_permission` 与 segment `leg_progress` / `trend_phase` 联动（由 bias / rules 裁决，非 RiskManager）
4. **参与度与收益联合验收**：禁止再以「少交易、低回撤」单独通过
5. **022 模块输出为输入，不返工**：Chop Guard、false confirm 等仍在 022 后续 phase 消化；023 基于冻结 v022 配置（§5.5）
6. **所有新通道以 RiskBudget 为硬前置**：`allow_open_*` / `allow_add_*` 未通过则不得开仓或加仓，Participation Floor 不得绕过

### 5.3 参与度指标公式（冻结）

以下公式在 `eval_participation.py` 与 `backtest/runner.py` 中**同构实现**；valid / test **分开统计**。

**腿集合 L**：`is_leg_confirmed=1` 且 `leg_type ∉ {TRANSITION_LEG, RANGE_LEG}`；默认**排除** `CHOP_HARD` 所在 bar 上的腿（与 022 Chop Guard 对齐）。

**腿方向 `leg_direction`**：由 `leg_type` 派生——`FAST_UP_LEG` / `SLOW_UP_LEG` → `UP`；`FAST_DOWN_LEG` / `SLOW_DOWN_LEG` → `DOWN`。

**重叠 bar 拆分（复审澄清 C1）**：coverage **只统计方向一致**的持仓，避免逆势仓刷高参与度。

```text
aligned_overlap_bars(leg) =
  |{bar ∈ leg_bars : position_side 与 leg_direction 同向}|

counter_overlap_bars(leg) =
  |{bar ∈ leg_bars : position_side 与 leg_direction 反向}|

overlap_bars(leg) = aligned_overlap_bars(leg)   # 下文 coverage 均指 aligned
```

`counter_overlap_bars > 0` 时记入 `counter_leg_participation_count`（按 leg 去重），**不计入** `leg_count_coverage_ratio` / `leg_bar_coverage_ratio`；须与 `HARD_BLOCK`、probe 规则、`reason_codes` 联动解释。

**单腿有效覆盖 `effective_leg_covered(leg)`**（防刷分）：

```text
avg_pos_ratio = aligned 重叠区间内 avg_position_ratio 均值

effective_leg_covered =
  aligned_overlap_bars >= min(3, leg_bars * 0.15)
  AND avg_pos_ratio >= min_effective_position_ratio   # 默认 0.02
```

probe 仓**可计入** aligned coverage，但必须满足上述有效覆盖条件；1 bar、0.01 仓位试探**不计**。

**三层覆盖率**

| 指标 | 公式 |
|------|------|
| `leg_count_coverage_ratio` | `\|{leg ∈ L : effective_leg_covered}\| / \|L\|` |
| `leg_bar_coverage_ratio` | `Σ aligned_overlap_bars(leg) / Σ leg_bars(leg)` |
| `leg_pnl_capture_ratio` | `Σ captured_pnl_in_leg / Σ ideal_leg_pnl`（见下） |

**`leg_pnl_capture_ratio` 分母（复审澄清 C2）**：分母必须为**正值机会收益**，避免下跌腿 raw return 为负导致 ratio 符号反转。

```text
ideal_leg_pnl(leg) = abs(close_end - close_start) / close_start

captured_pnl_in_leg = 该 leg 区间内 aligned 持仓的 net_pnl 之和（同向）

若 ideal_leg_pnl < small_move_threshold（默认 0.3%）：
  该腿不计入 leg_pnl_capture_ratio 分母，记入 small_move_leg_count
```

**同腿多笔交易**：按 `leg_id` 去重，coverage 只计一次；PnL 累加。

**亏损覆盖**：`leg_loss_coverage_count` 单独统计（effective 覆盖但 leg 内 net_pnl < 0），与盈利覆盖分列。

**逆势参与（审计）**

| 指标 | 公式 |
|------|------|
| `counter_leg_participation_count` | `\|{leg ∈ L : counter_overlap_bars > 0}\|` |
| `counter_overlap_bar_ratio` | `Σ counter_overlap_bars / Σ leg_bars` |

**慢涨专项**

```text
slow_up_watch_to_open_ratio =
  slow_up_open_count / max(watch_slow_uptrend_count, 1)

slow_up_false_entry_count =
  entry_was_slow_up=1 且 leg 内 net_pnl < 0 且 bars_held < min_hold_bars 的笔数
```

**归因 PnL（强制分列）**：`slow_up_pnl`、`trend_qualified_pnl`、`crash_upgrade_pnl`、`reentry_pnl`、`trend_add_pnl`。

### 6.4 Phase 1：分 PR 打开参与通道（P0，2~3 周）

> **禁止**在一个 PR 同时打开 1a / 1b / 1c，避免收益或回撤变化无法归因。

#### 6.4.1 Phase 1a：慢涨 watch → probe（首个 PR）

**配置**

```json
"slow_up_position": { "enabled": true, "position_ratio": 0.04, "stable_position_ratio": 0.06, ... }
```

**规则**（`rules.py`，仍经 `_bias_open_gate` + `risk_budget.allow_open_long`）：

1. `WATCH_SLOW_UPTREND` 持续 ≥ `watch_min_bars`（6）且 `is_stable_slow_uptrend` → `OPEN_LONG_SLOW_TREND`（probe 仓）
2. 放宽 `model_opp_edge_min` / `model_opp_cum_ret_min`（见 v1.0 §6.2.1）
3. `SLOW_UP_LEG` 确认后 `upgrade_min_bars` 6→4（仅慢涨通道）

**1a 出口**：valid 观察 `slow_up_false_entry_count` 可控；test `slow_up_open_count ≥ 1`；`total_return` 不低于探索线。

#### 6.4.2 Phase 1b：crash → trend 升级（第二个 PR）

```json
"trend_position": { "allow_crash_trend_upgrade": true, ... }
```

- `HOLD_CRASH_TREND_CONFIRMING` + `crash_upgrade_profit_atr` 或 confirmed 持续 N bar → `lifecycle=RUNNER`
- crash 仓持有上限用 `crash_short.max_hold_bars`；升级后切换 `trend_position.max_trend_hold_bars`

**1b 出口**：`trend_upgrade_count > 0`；`crash_upgrade_pnl` 单独为正或贡献可解释。

#### 6.4.3 Phase 1c：趋势资格开仓（第三个 PR）

**架构约束（阻塞项 B3）**：不得在 `rules.py` 内直接拼接 `trend_signal + segment + slow_up`。须经统一资格层：

```text
TrendEntryQualifier（新模块，或 TrendBiasBuilder 子步骤）
  输入：trend_signal, segment_context, slow_context, crash_context, chop_guard_state
  输出：TrendEntryQualification（非冻结辅助对象）

TrendEntryQualification:
  allow_trend_entry_long: bool
  allow_trend_entry_short: bool
  reason_codes: list[str]   # 如 TREND_QUALIFIED_LONG_FAST_UP
  entry_tier: PROBE | FULL
  relax_edge_mult: float
  relax_prob_delta: float

RuleEngine:
  读取 trend_bias + TrendEntryQualification + risk_budget
  → 再检查 signal 阈值（含 relax）→ OPEN_*_TREND_QUALIFIED
```

**禁止绕过**：`counter_level_* == HARD_BLOCK`、`CHOP_HARD`、`risk_budget.allow_open_* == false`、`max_position_ratio`。

**契约（初始值，经 Qualifier 输出后由 RuleEngine 执行）**

| 字段 | 多头 | 空头 |
|------|------|------|
| Qualifier 前提 | UP + confirmed + FAST_UP/SLOW_UP leg | DOWN + confirmed + FAST_DOWN leg |
| edge（relax 后） | ≥ 0.02 | ≤ -0.02 |
| p_up / p_down | ≥ 0.30 | ≥ 0.30 |
| 初始仓位 | `trend_qualified_position_ratio` 0.03~0.05 | 同左 |

**偏置联动**：`TrendBias.open_bias_*` 对 TREND_QUALIFIED 再乘 0.85；Qualifier 不写入冻结 `TrendBiasContext` 字段。

**1c 出口**：`leg_count_coverage_ratio ≥ 35%`（test）；`hard_counter_open_count` 不高于 v022；`counter_leg_participation_count` 须可解释（见 §5.3）。

### 6.5 Phase 2：持有延长与腿对齐（P0，2 周）

#### 6.5.1 分模式持有上限（职责划分）

| 模块 | 职责 |
|------|------|
| `trend_bias.py` / lifecycle helper | 据 `leg_progress_ratio`、`trend_phase` 计算 `hold_bias`、`exit_bias`、`time_exit_permission` |
| `rules.py` / ActionResolver | 据偏置决定是否发出 TIME_EXIT / REDUCE |
| `risk.py` | **仅**检查仓位、保证金、回撤、最坏损失是否突破硬边界 |

| hold_mode | 023 建议 max（经 hold_bias 缩放） |
|-----------|-----------------------------------|
| NORMAL | 8 |
| CRASH / PROBATION | 24~36 |
| TREND / RUNNER | 48~72 |
| SLOW_UP | 48~96 |

```text
leg_progress < 0.5 且 phase ∈ {CONTINUATION, ACCELERATION}
  → hold_bias *= 1.2，time_exit_permission = false
leg_progress > 0.75 或 phase == EXHAUSTION
  → time_exit_permission = true
```

#### 6.5.2 趋势延续加仓（leg 限次）

- 条件：`hold_mode=TREND`，`profit_atr ≥ add_profit_atr`，`trend_is_confirmed`，`leg_progress < 0.6`
- **硬限制**：`no_add_after_leg_progress_gt_0.6`；`max_add_count_per_leg`；`risk_budget.allow_add_*`
- 分列统计：`trend_add_candidate_count`、`trend_add_allowed_count`、`trend_add_pnl`

#### 6.5.3 同 regime 再入场（leg 限次）

```json
"crash_short": { "same_regime_once": false, "same_regime_reentry_cooldown_bars": 6 }
```

`trend_reentry` 配置段：

```text
max_reentry_per_leg
min_bars_after_exit
no_reentry_after_exhaustion
仅 probe 仓再入，须 risk_budget.allow_open_*
```

分列统计：`reentry_candidate_count`、`reentry_allowed_count`、`reentry_pnl`。

## 12. 结论

v022 证明了「趋势模块 + 偏置层」可以 **不伤害风控地略微提升收益**，但 **没有解决交易系统的核心瓶颈：参与率与腿覆盖率**。

023 v1.2 在架构师复审后明确：

1. **aligned / counter 分层 coverage** + 正值 `ideal_leg_pnl` 分母（§5.3）
2. **收益 / 回撤门槛统一**；推荐 config 不得劣于 v022（§5.1、§13）
3. **趋势资格开仓经 TrendEntryQualification**，延续 021/022 偏置架构（§6.4.3）；`teq_*` 日志可复盘（§5.4）
4. **Phase 1 分 PR**：1a slow_up → 1b crash upgrade → 1c trend-qualified
5. **Participation Floor 默认关闭**，仅实验分支

**立项复审通过，可进入 Phase 0**（`eval_participation.py` 为首个交付）。

按本方案执行后，系统应从「高精度低参与」转向「趋势腿驱动的中等参与」，在冻结 022 质量成果的前提下，显著提升腿覆盖与总收益。
