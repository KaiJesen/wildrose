| 现象 | 规则层表现 | 模型层假设 |
|------|-----------|-----------|

## 4. 根因分析（模型侧）

### 4.1 时间尺度错配（P0）

当前 `MarketStateHead` 默认 `pred_horizon=5`（5×1h），交易规则却消费：

```text
TrendSegment 腿：min_leg_bars=12，典型 FAST_* 20~40 bar，SLOW_UP 48~96 bar
TrendEntryQualifier：要求 confirmed leg + 放宽后 edge/prob
```

模型优化的是 **未来 5 bar 方向/收益**，规则需要的是 **腿内前 1/3~2/3 段的可参与性与方向一致性**。5 bar 信号在腿中段往往已衰减或反转，导致「趋势模块说 UP、模型说 flat/弱多」。

### 4.2 训练目标与业务 KPI 未对齐（P0）

`market_state_loss` 主权重为 step return + 3-class direction（008/0062e 配方）。023 业务 KPI 为：

```text
leg_count_coverage_ratio、leg_pnl_capture_ratio、aligned overlap
```

训练阶段 **未出现「腿」或「参与」概念**，valid 上 cum_return_ic 可达标，但 **confirmed leg 空仓 bar 的 edge 分位数不抬升**。

### 4.3 方向头未条件于趋势上下文（P1）

推理时 `edge = p_up - p_down` 为 **全局 3-class softmax**，未区分：

```text
CHOP / RANGE vs TREND leg
FAST_UP vs SLOW_UP
model_blind_crash vs 常规模型
```

023 Phase 3 在规则层用 regime 乘子补救，已证伪「标准开仓也放宽」路径；需在 **模型输出层** 提供 **条件参与分数**，而非仅靠规则缩放阈值。

### 4.4 监督信号与 017 best-point 未形成闭环（P1）

017 best-point 在 observe 回测中总收益较好，但与 **腿覆盖** 未联合验收。024 需定义：**腿对齐 entry zone** 标签，使模型在 edge 弱时仍输出可解释的 `p_entry_aligned`。

### 4.5 008 结构性 trade-off 未在 023 中解决（P2）

0062e：`cum_return_ic` 高、`return_ic` 低。crash 与累计方向受益，**腿内逐步参与**依赖 step 方向质量。024 需在 **不丢弃 cum 分支** 前提下，增加 **腿对齐 step 监督**。

## 5. 目标与范围

### 5.1 总目标

在 **冻结 023 phase1c 规则栈** 上，替换/增强模型信号，使 test split 同时逼近：

| 类别 | 指标 | 024 目标 |

- 不在 024 默认开启 Participation Floor（023 P2 实验项）

## 6. 解决方案

### 6.1 总体架构

```text
离线标签（Phase 0）
  K 线 + TrendSegmentEngine + TrendSignalProvider
    → LegParticipationLabel（per bar）
    → MultiHorizonReturnLabel（5/12/24/48 bar）

训练（Phase 1~2）
  PatternPredictor + MarketStateHead 扩展
    → return_pred[5]（保留）
    → return_pred_long[12,24,48]（新增 optional heads）
    → participation_logits（align_long / align_short）
    → loss = L_ms + λ_leg·L_leg + λ_part·L_part + λ_hz·L_long_hz

推理（Phase 3）
  ModelSignalProvider
    → TradingSignal 扩展字段（participation_edge_long/short, hz_edge）
    → 或由校准层映射为 effective edge / p_up 供 RuleEngine

验收（Phase 4）
  eval_model_participation.py → 全链路 v023 phase1c A/B
```

### 6.2 WP1：腿对齐离线标签（P0，1 周）

**交付**：`examples/build_leg_participation_labels.py` → `data/labels/leg_participation_{split}.parquet`

**每 bar 字段（最小集）**：

| 字段 | 说明 |
|------|------|
| `leg_id` / `leg_type` | 来自 segment |
| `is_leg_confirmed` | 腿是否确认 |
| `align_direction` | UP / DOWN / NONE |
| `leg_progress_ratio` | 0~1 |
| `ideal_participate_long` |  hindsight：多头持仓至 leg 结束净收益>成本 且 MAE 可控 |
| `ideal_participate_short` | 对称 |
| `forward_leg_roi_12/24/48` | 腿内多视野 ROI（用于长周期头） |

**标签原则**：

- 仅 **aligned** 腿生成正样本；counter 腿为 hard negative
- `EXHAUSTION` / `LEG_END` 后窗口标为 non-participate
- 与 §023.5.3 `effective_leg_covered` 定义对齐，避免标签与验收口径不一致

### 6.3 WP2：多视野预测头与损失（P0，2~3 周）

**结构改动**（`transformer_kit/pattern_model.py`）：

```text
MarketStateHead 扩展：
  return_pred          # 现有 H=5，保留
  return_pred_12/24/48 # 新增 Huber，可选共享 trunk
  participation_logit_long / participation_logit_short  # 二分类
```

**损失**（`transformer_kit/training.py`）：

```text
L_total = L_market_state_legacy          # 0062e 基线权重冻结一档
        + λ_12·Huber(ret_12) + λ_24·Huber(ret_24)   # 默认 λ_24=0.15, λ_12=0.10
        + λ_part·BCE(participation_* , ideal_participate_*)
        + λ_leg_dir·CE(direction | align_direction)   # 仅在 confirmed leg mask 上
        + λ_consist·|sum(ret_5) - ret_12_short|       # 防止多头互相矛盾
```

**采样**：

- `confirmed leg` bar **过采样** 或 loss mask 加权（`w_leg=3.0`）
- `CHOP_HARD` bar 降权（与 022 chop guard 一致）

**训练配方（初值）**：

| 项 | 值 |
|----|-----|
| 起点 checkpoint | `market_state_best.pt`（0062e） |
| 实验编号 | **0065a**（leg-align 分支） |
| 数据 | BTCUSDT 1h 365d，split 与 v023 一致 |
| 早停 | valid `leg_participation_auc` + `cum_return_ic` 双指标 |

### 6.4 WP3：参与分数推理与校准（P1，1~2 周）

**方案 A（推荐）**：扩展 `TradingSignal`，不破坏现有 `edge` 契约：

```python
# trading_system/signal.py 增量字段
participate_score_long: float = 0.0   # sigmoid(participation_logit_long)
participate_score_short: float = 0.0
edge_long_hz: float = 0.0             # 来自 24h head 或加权组合
effective_edge_long: float = 0.0      # 校准后供规则读取（可选）
```

**`ModelSignalProvider` 映射**：

```text
effective_edge_long = w5·edge_5 + w24·edge_24 + w_part·logit_part_long
默认 w5=0.35, w24=0.45, w_part=0.20（valid 调参）
```

**方案 B（备选）**：仅在 `TrendEntryQualifier` 消费 `participate_score_*`，不改标准 `OPEN_*_SIGNAL` 阈值（与 023 phase3.1 同思路，但分数来自模型而非规则乘子）。

**校准**：valid 上 Platt / isotonic，分 regime（UP leg / DOWN leg / crash）各一套；**禁止用 test 调参**。

### 6.5 WP4：模型级 + 全链路验收（P0，1 周）

**模型级**（先于规则）：`examples/eval_model_participation.py`

| 指标 | 说明 |
|------|------|
| `confirmed_leg_flat_edge_p50/p90` | 空仓 confirmed leg bar 的 edge 分位 |
| `participation_auc` | ideal_participate vs participate_score |
| `leg_entry_recall@k` | top-k edge bar 对 ideal entry 的召回 |
| `hz_direction_acc_24` | 24 bar 方向准确率（leg 子集） |

**全链路**：冻结 `configs/trading_rule_v023_phase1c_0062e.json`，仅换 checkpoint：

| 实验 | 模型 |
|------|------|
| A0 | 0062e baseline |
| A1 | 0065a leg-align（仅新损失） |
| A2 | A1 + participation head + 校准 |
| A3 | A2 + RuleEngine 读 `effective_edge_*`（若启用） |

**硬门槛**：与 §5.1 相同；**任一项不达标则不替换 prod checkpoint**。

### 6.6 与 017 best-point 的协同（P2，可选）

若 WP2 后 `participation_auc` 仍 < 0.58：

- 增加 **联合头**：`p_entry_aligned = f(ms_hidden, bp_hidden)`，训练标签来自 WP1
- 规则层：`TrendEntryQualifier` 在 `require_best_point=true` 时改读 `p_entry_aligned`
- **不**在 024 主路径重训 full best-point；仅 fusion 最后一层

## 11. 023 阶段性结论（转入 024 的前置裁定）

1. **规则栈可合并项**：phase1b + phase1c（`regime_threshold.enabled=false`），test return **7.77%**，架构合规。
2. **规则栈不可合并项**：phase3 标准开仓放宽、teq ceiling 极端门槛——覆盖略升但 PnL 不可接受。
3. **调参不足以闭合 §13**：teq ceiling 已将 teq 开仓从 1 提至 6，leg_count_coverage 仍仅 **20%**，return **1.42%**。
4. **下一瓶颈在模型**：confirmed leg 空仓窗口内 edge 分位数不足；需 024 从训练与推理修正。

## 13. 结论

023 证明了趋势 **识别层与规则资格层** 可以安全叠加，但无法突破 **0062e 模型在趋势腿时间尺度上的表达能力上限**。024 将 KPI 前移至训练与模型验收，以 **腿对齐标签 + 多视野头 + 参与分数** 三板斧，使 `TrendEntryQualifier` 在 confirmed leg 上获得 **方向正确、幅度足够、可校准** 的模型输入。

**建议批准 024 立项**；023 规则工作 **阶段冻结**，checkpoint 替换以 024 Phase 3 A/B 全门槛 PASS 为唯一投产条件。
