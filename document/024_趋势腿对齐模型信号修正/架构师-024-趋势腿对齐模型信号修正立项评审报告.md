## 2. 总体结论

024 的立项方向 **业务上必要、架构上正确、技术上可行但复杂度高**。相对 023「规则层参与度修正」，024 将瓶颈前移至 **0062e 短周期信号与趋势腿时间尺度错配**，与 teq ceiling 诊断叙事一致，是合理的下一阶段。

023 规则栈已证明：

| 路径 | 典型 test 表现 | 含义 |
|------|----------------|------|
| v022 baseline | return 9.31%，leg 覆盖 ~10%（023 口径） | 模型强、参与弱 |
| phase1c（TEQ + 规则栈） | return ~7.77%，leg_count_coverage ~16.7%，teq_open≈1 | 架构合规，覆盖略升、收益降 |
| teq ceiling（极端 teq 门槛） | return ~1.42%，coverage ~20%，teq_open≈6 | **调参无法同时闭合 coverage≥35% 与 return≥9.31%** |

因此 **「规则调参边际收益趋零 → 需模型侧腿对齐」** 的业务判断 **成立**。

但 v1.0 立项材料仍存在四类会直接影响实施正确性的问题：

1. **双硬门槛（return≥9.31% 且 leg_count_coverage≥35%）一次性验收风险极高，缺少分阶段出口与 fallback 契约。**
2. **`ideal_participate_*` 与 `effective_edge` 推理契约未冻结，标签—损失—规则消费链路存在口径断裂。**
3. **方案 A（改 `TradingSignal` / effective_edge）与方案 B（仅 TEQ 读 participate_score）并存，Phase 3 默认路径未选定，可能误伤标准 `OPEN_*_SIGNAL`。**
4. **回测依据路径（`backtest/v023_phase1c/`、`CEILING_REPORT.md`）未入库，立项数据可复现性不足。**

评审结论：

```text
024 可以立项；
建议有条件批准；
进入 Phase 0（标签管线）前完成 v1.1 修订；
Phase 3 全链路 A/B 前不得替换 prod checkpoint。
```

## 3. 业务可行性评审

### 3.1 问题定位与 023 衔接 — **通过**

024 正确继承 023 §11 裁定：

- **可合并**：phase1b + phase1c（`regime_threshold` 对标准开仓关闭）
- **不可合并**：phase3 全局放宽、teq ceiling 极端门槛

**评审意见：通过。**

---

### 3.2 业务 KPI 方向 — **通过，但需分阶段**

§5.1 四维目标（return、coverage、leg_pnl_capture、teq_pnl）与 023 §13 对齐，方向正确。

需警惕的是 **目标同时偏 ambitious**：

```text
phase1c 最好：coverage 16.7%，return 7.77%（较 v022 -1.54pp）
teq ceiling：coverage 20.0%，return 1.42%
024 要求：coverage ≥35%，return ≥9.31%（较 phase1c +1.54pp 且 coverage ×2+）
```

这在单 checkpoint 替换下 **可能无法一次达成**。建议立项修订为 **分层验收**：

| 阶段 | 最低要求 | 用途 |
|------|----------|------|
| Phase 1 模型 valid | `participation_auc≥0.55`，`confirmed_leg_flat_edge_p50` 方向正确 | 证明信号可学 |
| Phase 2 模型+校准 valid | teq 触发率 ≥2×0062e，precision@open 不劣于 ceiling | 证明可参与 |
| Phase 3 全链路 test | return ≥8.84%（95% v022）**且** coverage ≥28% | 探索线 |
| Phase 4 推荐 checkpoint | return ≥9.31% **且** coverage ≥35% | 投产线 |

**评审意见：有条件通过；修订分层门槛。**

## 4. 架构合理性评审

### 4.1 与 021 / 023 的边界 — **通过**

方案明确：

```text
不重写 TrendBias / TrendEntryQualifier / RiskBudget
冻结 configs/trading_rule_v023_phase1c_0062e.json
024 仅替换 checkpoint + ModelSignalProvider 映射
```

这与当前 `TrendEntryQualifier` 设计一致：资格来自趋势模块，**模型只提供执行分数**。架构分层清晰：

```text
TrendSegment / TrendSignal → TrendEntryQualifier（资格）
ModelSignalProvider → edge / participate_score（强度）
TrendBias + RiskBudget → RuleEngine（动作）
```

**评审意见：通过。**

---

### 4.2 方案 A vs 方案 B 未决 — **阻塞**

§6.4 同时给出：

- **方案 A（推荐）**：扩展 `TradingSignal`，`effective_edge_*` 加权组合，A3 让 RuleEngine 读取
- **方案 B（备选）**：仅 TEQ 消费 `participate_score_*`

风险：

| 路径 | 风险 |
|------|------|
| A3 全局 effective_edge | 可能改变 **标准 `OPEN_*_SIGNAL`**、reverse、continue 等全链路行为，归因困难 |
| B | 仅 teq 改善，**crash / slow_up / 标准开仓** 仍用旧 edge，coverage 提升有上限 |

**必须修正（阻塞项 B3）**：

1. **Phase 3 默认采用方案 B**（仅 `_try_trend_qualified_open` 读 `participate_score_*` / `edge_long_hz`）
2. 方案 A 作为 **A3 实验分支**，且 `effective_edge` **不得**替换 `TradingSignal.edge` 全局字段，应新增 `teq_edge_long` 等 TEQ 专用字段
3. `RuleEngine._trend_qualified_thresholds()` 文档化：比较的是 `teq_edge` 还是 `edge`

**评审意见：架构方向对，消费边界未冻结。**

---

### 4.3 标签源与 022 冻结 — **需澄清**

WP1 标签来自 `TrendSegmentEngine + TrendSignalProvider`，但 022 模块仍在演进（Chop Guard 等）。若训练标签与回测 segment 版本不一致，会出现 **模型学 A 腿、规则用 B 腿**。

**建议修订**：

- 复用 023 §5.5 思路：`segment_module_version` + `trend_signal_config_hash` 写入 label parquet 元数据
- Phase 0 对齐 `eval_participation.py` 的 leg 集合定义（`CONFIRMED_LEG_TYPES`、aligned/counter）

**评审意见：架构合理，需版本 gate。**

---

### 4.4 与 TEQ 腿类型不对称 — **细节缺口**

当前 `TrendEntryQualifier._SHORT_LEG_TYPES` **仅含 `FAST_DOWN_LEG`**，不含 `SLOW_DOWN_LEG`（`trend_entry_qualifier.py`）。

WP1 标签若对称生成 `ideal_participate_short` 于 `SLOW_DOWN_LEG`，会出现 **标签正样本多、规则永远不消费** 的结构性浪费。

**建议修订**：

- 要么 TEQ 扩展 `SLOW_DOWN_LEG`（023 后续小改，024 文档声明依赖）
- 要么标签 short 正样本 **仅 FAST_DOWN_LEG**，与 TEQ 对齐

**评审意见：需在 v1.1 明确。**
