### 2.3 核心判断


## 5. 总体方案

022 拆为五个工作包（WP），按依赖顺序实施：

```text
WP-A  TrendSignal 记忆与确认分层
WP-B  震荡市守卫（Chop Guard）
WP-C  TrendSegment 性能与增量腿指标
WP-D  TrendBias 偏置再平衡与阻断审计
WP-E  趋势模块评估验收体系（Trend Module Eval Harness）
        ↓
联动回测验收（复用 v016/v021 配置，验证交易层指标改善）
```

原则：

1. **方向与确认分离**：`direction` 表示当前倾向，`is_confirmed` 表示可驱动开仓/升级；`is_sustained` 表示可维持已有趋势仓。
2. **is_sustained 硬约束**（架构师 §3.2）：`is_sustained` **仅**用于持仓维持 / runner 容忍；**不得**用于新开仓或首次趋势升级，否则震荡市假趋势将增多。
3. **记忆优先于单 bar**：用迟滞状态机消化回撤，避免 2 根反向即 `broken`。
4. **性能是功能的一部分**：分段引擎不能成为回测瓶颈；bounded buffer 须先设计绝对索引映射。
5. **召回与精度成对验收**：Teacher coverage 不是唯一真理；须同时监控 precision 与 RANGE 误报。
6. **先量化验收，再调交易规则**：PnL 不是模块级硬门槛，但联动回测有最低业务底线。

## 6. 详细方案

### 6.1 WP-A：TrendSignal 记忆与确认分层

#### 6.1.1 问题

当前逻辑（`trend_signal.py`）：

```text
单 bar score 未达 confirmed_score → invalid_count += 1
invalid_count >= invalid_confirm_bars(2) → direction 记忆清零、is_broken=true
```

导致：

- 趋势中段正常回撤 1~2 根即丢失 `trend_age`
- OOS `broken` 占 70%，确认仅 23.7%
- 方向 raw match 高（73~87%），confirmed match 低（48~54%）

#### 6.1.2 方案：双轨确认 + 迟滞记忆

新增配置（`TrendSignalConfig` 扩展）：

```python
@dataclass
class TrendSignalConfig:
    # 现有字段保留 ...
    hold_confirm_score: int = 3          # 维持已确认趋势的最低分（低于 confirmed_score）
    invalid_confirm_bars: int = 3        # 默认从 2 提高到 3（可配置）
    invalid_reset_bars: int = 4          # 连续失效多少根才清零 trend_age
    direction_margin: int = 2            # 已有：score 差值门槛
    min_trend_age_for_hold: int = 3      # trend_age 达标后启用 hold_confirm_score
```

状态机修订：

```text
若 memory.trend_age >= min_trend_age_for_hold 且 score_abs >= hold_confirm_score
  且 direction 与 memory.current_direction 同向：
    → 维持 confirmed，invalid_count 不增加（或慢速衰减）

若 direction 反向且 score 差值 >= direction_margin：
    → invalid_count 快速增加

仅当 invalid_count >= invalid_reset_bars：
    → 清零 trend_age，标记 REVERSAL_RISK / broken
```

输出字段（向后兼容；新字段须有默认值）：

```python
from enum import Enum

class ConfirmTier(str, Enum):
    NONE = "NONE"
    CANDIDATE = "CANDIDATE"
    CONFIRMED = "CONFIRMED"
    SUSTAINED = "SUSTAINED"

@dataclass
class TrendSignal:
    is_confirmed: bool       # 开仓/升级用（高门槛）；语义不变
    is_sustained: bool = False   # 持仓维持用（低门槛）；默认 False
    confirm_tier: ConfirmTier = ConfirmTier.NONE
```

**消费边界（硬约束）**：

| 字段 | 允许用途 | 禁止用途 |
|------|----------|----------|
| `is_confirmed` | 新开仓、趋势升级、`ADD_TREND_CONTINUATION` 候选 | — |
| `is_sustained` | 持仓维持、runner 容忍、退出投票宽限 | 新开仓、首次 `UPGRADE_TO_TREND_*` |
| `confirm_tier` | 日志、评估、调试 | 直接替代 `is_confirmed` 做开仓门控 |

`is_sustained` 仅在 RuleEngine **full_bias** 持仓逻辑中启用（§7 与 018 分工）。

#### 6.1.3 预期效果

| 指标 | 当前 OOS | 022 目标 |
|------|----------|----------|
| 确认趋势占比 | 23.7% | 30%~45%（可调） |
| `broken` 占比 | 70.0% | < 45% |
| Teacher 趋势覆盖（M1，合并口径） | 51.5% | **≥ 65%** |
| Teacher 覆盖（M1a，`confirmed_only`） | — | 报告必出；开仓/升级质量主指标 |
| Teacher 覆盖（M1b，`confirmed_or_sustained`） | — | 报告必出；持仓维持质量参考 |
| Teacher DOWN confirmed match | 53.7% | **≥ 65%** |
| `confirmed_precision_vs_teacher`（M9） | — | **≥ 55%**（初期） |
| `false_confirm_on_range_teacher`（M10） | — | **≤ 20%** |
| `confirmed_direction_macro_f1`（M11） | 基线 | 不低于 022 前基线 |

### 6.2 WP-B：震荡市守卫（Chop Guard）

#### 6.2.1 问题

`test_choppy_market_outputs_none` 失败：小幅交替涨跌仍满足 `close < ema_fast` 等 DOWN 投票项。

019/020 已证明慢速上涨趋势常具备低波动、低 ATR 扩张、短窗口收益不强等特征；若 Chop Guard 只看 `chop_range_atr` 与 `ret_24_atr`，可能误杀 slow uptrend。

#### 6.2.2 方案：投票前守卫 + 强/弱分级

**实现顺序**（须在投票前执行，避免 reason_codes 与分数语义混乱）：

```text
1. 计算 chop 所需的 range / flip / efficiency
2. 评估 slow-trend 例外（见下）；若命中例外 → 跳过 CHOP_HARD
3. 若强 CHOP → direction=NONE，confirm_tier=NONE
4. 若弱 CHOP → 保留 direction，但禁止 is_confirmed（仅允许 CANDIDATE 或 NONE）
5. 非 CHOP → 走原投票与确认逻辑
```

**Chop 判定条件**（任一可触发，经 slow-trend 例外过滤后生效）：

```text
chop_range_atr = (rolling_high_24 - rolling_low_24) / atr
若 chop_range_atr < 3.0 且 abs(ret_24_atr) < 1.0 → CHOP_RANGE 候选

chop_flip_count = 过去 12 根 close 与 ema_fast 交叉次数
若 chop_flip_count >= 4 → CHOP_FLIP 候选

efficiency_12 = abs(close - close[-12]) / sum(abs(diff(close[-12:])))
若 efficiency_12 < 0.25 → CHOP_INEFFICIENT 候选
```

**分级**：

| 级别 | 条件 | 行为 |
|------|------|------|
| `CHOP_HARD` | ≥2 项 chop 候选同时满足，且未命中 slow-trend 例外 | `direction=NONE`，`is_confirmed=false`，`reason_codes+=["CHOP_HARD"]` |
| `CHOP_SOFT` | 仅 1 项 chop 候选满足 | 不清空方向；`is_confirmed=false`；`reason_codes+=["CHOP_SOFT"]` |

**Slow-trend 例外**（满足任一即不触发 `CHOP_HARD` 强制 NONE）：

```text
Phase 1 在 TrendSignalProvider 内部可计算的代理条件（优先）：
1. 48/72 bar EMA slope 为正且持续（可配置窗口）
2. persistence_above_ema_mid 高于阈值
3. efficiency_48 高于阈值（长窗口效率补偿短窗口 CHOP）

评估 / 二次确认（不作为 Phase 1 compute() 输入）：
4. SlowUptrendDetector.stable_slow_uptrend — 仅 Eval Harness 对照统计
```

**计算顺序约束**（`engine.py` 当前顺序）：

```text
TrendSignalProvider → CrashRegimeDetector → SlowUptrendDetector → TrendSegmentEngine → TrendBiasBuilder
```

`TrendSignalProvider.compute()` **不得**在 Phase 1 直接依赖 `SlowUptrendDetector` 输出，否则形成循环依赖。慢涨保护在信号层仅用上述 1~3 项代理；`stable_slow_uptrend` 用于评估报告中的「代理 vs  detector」一致性检查。

配置项：

```python
chop_guard_enabled: bool = True
chop_range_atr_max: float = 3.0
chop_flip_max: int = 4
chop_efficiency_min: float = 0.25
chop_slow_trend_exception_enabled: bool = True
chop_efficiency_48_min: float = 0.35
chop_ema_slope_bars: int = 48
```

#### 6.2.3 CHOP_SOFT 与 micro_direction（Phase 3 落地）

`CHOP_SOFT` 保留 `direction` 但禁止 `is_confirmed`。进入 `TrendBiasBuilder` 后，`TrendSignal.direction` 仍可能影响 `micro_direction` 与 `alignment_score_*`，弱 chop 方向可能继续放大开仓偏置。

**分阶段处理**：

| 阶段 | 行为 |
|------|------|
| Phase 1 | `reason_codes` 含 `CHOP_SOFT`；Eval Harness 统计 CHOP_SOFT bar 上的 `micro_direction` 暴露（观察项） |
| Phase 3（WP-D） | Builder 识别 `CHOP_SOFT` 时追加 `CHOP_SOFT_MICRO_WEAK`；`micro_direction` **降权或仅作诊断**，不参与 `open_bias` 放大 |

```text
CHOP_SOFT 时：
  micro_direction 权重 × chop_soft_micro_weight（默认 0.5，可配置）
  或 micro_direction 仅写入 reason_codes，不计入 alignment_score
```

#### 6.2.4 验收

- `test_choppy_market_outputs_none` 必须通过
- OOS 上 `CHOP_HARD` 触发 bar 中，Teacher 标为 TRANSITION/RANGE 的比例应 **> 70%**
- `choppy_false_confirm_rate`（M2）≤ 5%
- slow uptrend 回归用例：代理条件 1~3 命中时，不因 `CHOP_HARD` 清零方向（新增或扩展现有单测）
- Eval Harness 输出 `slow_trend_proxy_hit_rate` 与 `stable_slow_uptrend` 对照（不要求 Phase 1 一致）

### 6.3 WP-C：TrendSegment 性能与增量腿指标

#### 6.3.1 问题

`TrendSegmentEngine.update` 每根 bar 对当前腿执行：

```text
_leg_efficiency(start, end)      # O(leg_length) 循环
_max_adverse_ratio(start, end)   # O(leg_length) 循环
_close_hist / _high_hist 无限 append
```

8745 bar 总复杂度近似 **O(n²)**，实测 ~401s。

#### 6.3.2 方案

**C0. 绝对索引与有界缓冲映射**（实施 deque 前必须先设计）

```python
buffer_start_bar_idx: int   # deque[0] 对应的绝对 bar index

def abs_to_local(abs_idx: int) -> int:
    return abs_idx - buffer_start_bar_idx
```

规定：

```text
active_leg.start_bar_idx 为绝对 index；腿仍 active 时其起点不得被 deque 淘汰
若 active_leg 跨度超过 max buffer → 切换到增量累加器，不再依赖历史切片
previous_leg 仅保留汇总指标（efficiency、pullback 等），不依赖 buffer 重算
golden snapshot 对比须使用相同 abs_idx 语义
```

**C1. 有界历史缓冲**

```text
_close_hist / _high_hist / _low_hist / _atr_hist 改为 maxlen = max(leg_lookback, 512) 的 deque
仅保留分段与 swing 所需窗口；与 C0 映射联动
```

**C2. 增量腿指标**

为 `_active_leg` 维护滚动累加器，每 bar O(1) 更新：

```text
leg_net_return
leg_gross_move
leg_max_favorable
leg_max_adverse
leg_tr_sum
```

推导：

```text
leg_efficiency = abs(leg_net_return) / max(leg_gross_move, eps)
leg_pullback_ratio = leg_max_adverse / max(abs(leg_net_return), eps)
```

**C3. Swing 检测窗口化**

### 6.4 WP-D：TrendBias 偏置再平衡与阻断审计

#### 6.4.1 问题

OOS 上 `allow_open_long=False` 达 578 bars，而 `allow_open_short=False` 仅 130 bars。宏观偏置 DOWN:UP = 821:491。

多头被多重来源叠加阻断：

```text
CrashRegimeDetector → 硬阻断开多
legacy TrendRegimeFilter.is_downtrend → 硬阻断开多
macro_direction = DOWN → 提高逆势成本
```

BTC OOS 区间可能处于下跌 regime，开多被阻断多于开空在部分时段合理；验收不能机械追求多空 2:1。

#### 6.4.2 方案

**D1. 硬阻断来源可追溯（不修改 021 冻结 schema）**

采用架构师 **方案 B**：

```text
TrendBiasContext 不新增 block_reason_long/short 字段
TrendBiasContext.reason_codes 继续承载阻断原因
eval_trend_modules.py 从 reason_codes / 日志派生：
  block_long_by_reason, block_short_by_reason, block_reason_entropy
logger 可选输出派生 JSON 字段，不进入核心 dataclass
```

**D2. 阻断降级策略与优先级表**

`legacy_down_hard_block` 默认改为 `False`，但须遵守：

| 优先级 | 条件 | 对开多的影响 |
|--------|------|--------------|
| P1 | `crash` active | 仍 **hard_block** 开多 |
| P2 | confirmed DOWN leg + `should_avoid_counter` | 仍 **hard_block** 开多 |
| P3 | legacy `is_downtrend`（无 crash、无 confirmed down leg） | 仅 **soft** size_bias，不单独 hard_block |
| P4 | `macro_direction=DOWN` | 仅 soft bias；RANGE/TRANSITION leg 上不触发 macro 硬阻断 |

```text
当 trend_signal.is_sustained 且 direction=DOWN 但 crash 已退出：
  legacy is_downtrend 仅影响 size_bias（见上表 P3）

当 segment.leg_type in (RANGE_LEG, TRANSITION_LEG)：
  不触发 macro_direction 硬阻断，仅软偏置

is_sustained 不参与开仓门控（§6.1.2）
```

**D3. 对称性配置与分层验收**

```python
@dataclass
class TrendBiasConfig:
    max_hard_block_ratio: float = 0.35   # 单侧硬阻断 bar 占比上限（评估告警，非全局硬门槛）
    legacy_down_hard_block: bool = False
    crash_hard_block_long: bool = True
```

**D4. 按 regime 分层审计输出**

评估脚本输出：

```text
block_long_by_reason: {CRASH: n, LEGACY: n, SEGMENT: n, MACRO: n, ...}
block_short_by_reason: {...}
block_reason_entropy

hard_block_symmetry_by_regime:
  RANGE_TRANSITION: long_block / short_block 不应严重失衡
  confirmed_DOWN_leg: long_block 高可接受，须由 CRASH/SEGMENT 解释占比
  confirmed_UP_leg: short_block 应对称生效
```

#### 6.4.3 预期效果

| 指标 | 当前 OOS | 022 目标 |
|------|----------|----------|
| 硬阻断开多占比（全局 M5） | 44.1%（578/1312） | **< 30%**（参考线；须结合 regime 解读） |
| RANGE/TRANSITION 区间多空 hard block 比（M8） | 严重失衡 | 比值 **< 3 : 1** |
| confirmed UP leg 上 short hard block | — | 与 long 在 confirmed DOWN 上对称生效 |
| `block_reason_entropy` | — | 可解释；CRASH+LEGACY 占主导须标注 |

## 9. 风险与约束

| 风险 | 缓解 |
|------|------|
| 放宽确认导致震荡市假阳性上升 | Chop Guard + M2/M9/M10 上限 |
| Chop Guard 误杀慢涨 | slow-trend 代理 1~3 + 019 回归用例 |
| SlowUptrendDetector 与 Signal 循环依赖 | Phase 1 仅用内部代理；detector 仅 Eval 对照 |
| CHOP_SOFT 弱方向放大偏置 | Phase 3 `CHOP_SOFT_MICRO_WEAK` 降权 |
| Phase 1 混入交易层改动难以归因 | 硬边界：不改 RuleEngine / trend_bias 行为 |
| 放宽确认与 016 tuned2 参数冲突 | 单独 `trading_rule_v022_trend_quality_0062e.json` |
| 分段增量实现与全量重算不一致 | C0 索引规范 + golden snapshot |
| deque 淘汰 active_leg 起点 | C0 规则：active 腿不切历史切片 |
| 降低 hard_block 增加逆势亏损 | P1/P2 优先级表；crash 保留 hard_block |
| 机械追求多空对称在熊市放松多头 | M8 分层验收替代全局 2:1 |
| Teacher 标签与线上一致性有限 | 固定版本 + precision 成对指标；M1 不作唯一真理 |
| 修改 021 冻结 schema | 方案 B：仅派生审计字段 |

**明确不做**：

- 不在 022 内重训 Transformer / 市场状态模型
- 不引入新的趋势模块替代 `TrendSignalProvider`
- 不为了覆盖率强行降低 `confirmed_score` 到 ≤2（避免退化为噪声）
- 不向 `TrendBiasContext` 新增 `block_reason_*` 字段

## 11. 立项结论

022 项目 **立项复审通过，可进入 Phase 1 实施**。

理由：

1. 016~021 已完成架构铺底，当前瓶颈是**工程可用性**而非缺模块。

5. 初审八项修订与复审三项澄清均已纳入 v1.2；成功标准明确为「模块 + 联动 + 性能」三维。
6. 架构师确认**不需要再次立项层大评审**；最终以 Phase 3 联动回测与业务底线为准。
