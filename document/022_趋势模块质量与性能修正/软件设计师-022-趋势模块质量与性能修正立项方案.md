## 5. 总体方案

022 拆为五个工作包（WP），按依赖顺序实施：

```text
WP-A  TrendSignal 记忆与确认分层
WP-B  震荡市守卫（Chop Guard）
WP-C  TrendSegment 性能与增量腿指标
WP-D  TrendBias 偏置再平衡与阻断审计
WP-E  趋势模块评估验收体系（Trend Module Eval Harness）
        ↓
联动回测验收（复用 v016/v021 配置，验证交易层指标改善）
```

原则：

1. **方向与确认分离**：`direction` 表示当前倾向，`is_confirmed` 表示可驱动持仓升级；允许「方向对但未确认」用于观察，「已确认」用于持仓。
2. **记忆优先于单 bar**：用迟滞状态机消化回撤，避免 2 根反向即 `broken`。
3. **性能是功能的一部分**：分段引擎不能成为回测瓶颈。
4. **先量化验收，再调交易规则**：022 不以 PnL 为唯一成功标准。

## 6. 详细方案

### 6.1 WP-A：TrendSignal 记忆与确认分层

#### 6.1.1 问题

当前逻辑（`trend_signal.py`）：

```text
单 bar score 未达 confirmed_score → invalid_count += 1
invalid_count >= invalid_confirm_bars(2) → direction 记忆清零、is_broken=true
```

导致：

- 趋势中段正常回撤 1~2 根即丢失 `trend_age`
- OOS `broken` 占 70%，确认仅 23.7%
- 方向 raw match 高（73~87%），confirmed match 低（48~54%）

#### 6.1.2 方案：双轨确认 + 迟滞记忆

新增配置（`TrendSignalConfig` 扩展）：

```python
@dataclass
class TrendSignalConfig:
    # 现有字段保留 ...
    hold_confirm_score: int = 3          # 维持已确认趋势的最低分（低于 confirmed_score）
    invalid_confirm_bars: int = 3        # 默认从 2 提高到 3（可配置）
    invalid_reset_bars: int = 4          # 连续失效多少根才清零 trend_age
    direction_margin: int = 2            # 已有：score 差值门槛
    min_trend_age_for_hold: int = 3      # trend_age 达标后启用 hold_confirm_score
```

状态机修订：

```text
若 memory.trend_age >= min_trend_age_for_hold 且 score_abs >= hold_confirm_score
  且 direction 与 memory.current_direction 同向：
    → 维持 confirmed，invalid_count 不增加（或慢速衰减）

若 direction 反向且 score 差值 >= direction_margin：
    → invalid_count 快速增加

仅当 invalid_count >= invalid_reset_bars：
    → 清零 trend_age，标记 REVERSAL_RISK / broken
```

输出字段（向后兼容，可选新增）：

```python
is_confirmed: bool       # 开仓/升级用（高门槛）
is_sustained: bool       # 持仓维持用（低门槛，新增）
confirm_tier: str        # "NONE" | "CANDIDATE" | "CONFIRMED" | "SUSTAINED"
```

#### 6.1.3 预期效果

| 指标 | 当前 OOS | 022 目标 |
|------|----------|----------|
| 确认趋势占比 | 23.7% | 30%~45%（可调） |
| `broken` 占比 | 70.0% | < 45% |
| Teacher 趋势覆盖 | 51.5% | **≥ 65%** |
| Teacher DOWN confirmed match | 53.7% | **≥ 65%** |

### 6.2 WP-B：震荡市守卫（Chop Guard）

#### 6.2.1 问题

`test_choppy_market_outputs_none` 失败：小幅交替涨跌仍满足 `close < ema_fast` 等 DOWN 投票项。

#### 6.2.2 方案

在 `TrendSignalProvider.compute` 投票前增加 **chop 判定**（任一满足即强制 `direction=NONE`，且不计入 confirmed）：

```text
chop_range_atr = (rolling_high_24 - rolling_low_24) / atr
若 chop_range_atr < 3.0 且 abs(ret_24_atr) < 1.0 → CHOP_RANGE

chop_flip_count = 过去 12 根 close 与 ema_fast 交叉次数
若 chop_flip_count >= 4 → CHOP_FLIP

efficiency_12 = abs(close - close[-12]) / sum(abs(diff(close[-12:])))
若 efficiency_12 < 0.25 → CHOP_INEFFICIENT
```

当 `chop_guard` 触发：

```text
score_up = score_down = 0（或保留原始分但不产出方向）
direction = NONE
is_confirmed = false
reason_codes += ["CHOP_GUARD"]
```

配置项：

```python
chop_guard_enabled: bool = True
chop_range_atr_max: float = 3.0
chop_flip_max: int = 4
chop_efficiency_min: float = 0.25
```

#### 6.2.3 验收

- `test_choppy_market_outputs_none` 必须通过
- OOS 上 CHOP_GUARD 触发 bar 中，Teacher 标为 TRANSITION/RANGE 的比例应 **> 70%**

### 6.3 WP-C：TrendSegment 性能与增量腿指标

#### 6.3.1 问题

`TrendSegmentEngine.update` 每根 bar 对当前腿执行：

```text
_leg_efficiency(start, end)      # O(leg_length) 循环
_max_adverse_ratio(start, end)   # O(leg_length) 循环
_close_hist / _high_hist 无限 append
```

8745 bar 总复杂度近似 **O(n²)**，实测 ~401s。

#### 6.3.2 方案

**C1. 有界历史缓冲**

```text
_close_hist / _high_hist / _low_hist / _atr_hist 改为 maxlen = max(leg_lookback, 512) 的 deque
仅保留分段与 swing 所需窗口
```

**C2. 增量腿指标**

为 `_active_leg` 维护滚动累加器，每 bar O(1) 更新：

```text
leg_net_return
leg_gross_move
leg_max_favorable
leg_max_adverse
leg_tr_sum
```

推导：

```text
leg_efficiency = abs(leg_net_return) / max(leg_gross_move, eps)
leg_pullback_ratio = leg_max_adverse / max(abs(leg_net_return), eps)
```

**C3. Swing 检测窗口化**

`SwingDetector` 不再扫描 `0..current` 全历史，仅在 `[candidate - left, current]` 窗口内比较。

**C4. 性能基准测试**

新增 `tests/test_trend_segment_perf.py`：

```text
8745 bar update 耗时 < 30s（CI 宽松阈值 60s）
结果与优化前抽样 500 bar 输出一致（golden snapshot）
```

#### 6.3.3 预期效果

| 指标 | 当前 | 022 目标 |
|------|------|----------|
| 8745 bar 全链路评估 | ~401s | **< 30s** |
| 确认腿占比 | 17.1% | 不劣化（±2%） |
| 腿类型分布 | 大量 TRANSITION/RANGE | 通过 WP-A/B 间接改善，022 不硬改分类阈值 |

### 6.4 WP-D：TrendBias 偏置再平衡与阻断审计

#### 6.4.1 问题

OOS 上 `allow_open_long=False` 达 578 bars，而 `allow_open_short=False` 仅 130 bars。宏观偏置 DOWN:UP = 821:491。

多头被多重来源叠加阻断：

```text
CrashRegimeDetector → 硬阻断开多
legacy TrendRegimeFilter.is_downtrend → 硬阻断开多
macro_direction = DOWN → 提高逆势成本
```

#### 6.4.2 方案

**D1. 硬阻断来源可追溯**

`TrendBiasContext` 新增：

```python
block_reason_long: list[str]   # 如 ["CRASH", "LEG_COUNTER", "LEGACY_DOWN"]
block_reason_short: list[str]
```

**D2. 阻断降级策略**

```text
当 trend_signal.is_sustained 且 direction=DOWN 但 crash 已退出：
  legacy is_downtrend 仅影响 size_bias，不再单独 hard_block 开多
  （crash 活跃时仍 hard_block）

当 segment.leg_type in (RANGE_LEG, TRANSITION_LEG)：
  不触发 macro_direction 硬阻断，仅软偏置
```

**D3. 对称性配置**

```python
@dataclass
class TrendBiasConfig:
    max_hard_block_ratio: float = 0.35   # 单侧硬阻断 bar 占比上限（评估告警）
    legacy_down_hard_block: bool = False  # 默认改为软偏置
    crash_hard_block_long: bool = True    # 保留
```

**D4. 审计输出**

评估脚本输出：

```text
block_long_by_reason: {CRASH: n, LEGACY: n, ...}
block_short_by_reason: {...}
hard_block_long_ratio / hard_block_short_ratio
```

#### 6.4.3 预期效果

| 指标 | 当前 OOS | 022 目标 |
|------|----------|----------|
| 硬阻断开多占比 | 44.1%（578/1312） | **< 30%** |
| 硬阻断开空占比 | 9.9% | < 25% |
| 多空硬阻断比 | 4.4 : 1 | **< 2 : 1** |

## 9. 风险与约束

| 风险 | 缓解 |
|------|------|
| 放宽确认导致震荡市假阳性上升 | Chop Guard + M2 上限 |

## 11. 立项结论

022 项目建议 **批准立项**。

理由：

1. 016~021 已完成架构铺底，当前瓶颈是**工程可用性**而非缺模块。
