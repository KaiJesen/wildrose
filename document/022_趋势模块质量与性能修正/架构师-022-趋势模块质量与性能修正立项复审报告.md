## 2. 总体结论

软件设计师已对 022 立项方案完成实质性修订。初审提出的关键风险基本关闭：

```text

2. Chop Guard 已加入 CHOP_HARD / CHOP_SOFT 与 slow-trend 例外；
3. Teacher 评估已补 precision / false positive；

```

从量化业务角度看，修订后方案避免了“只追求趋势召回率”的问题，开始同时约束：

```text
趋势召回、趋势精度、横盘误报、慢涨保护、交易联动不退化。
```

从架构角度看，修订后方案没有继续扩张 021 冻结的 `TrendBiasContext` schema，而是把阻断原因作为 `reason_codes` 的派生审计项处理，这是正确选择。

因此，022 可以进入 Phase 1：

```text
WP-A TrendSignal 记忆与确认分层
WP-B Chop Guard

```

但实施前仍需澄清 3 个细节，避免程序员在实现时产生歧义。

## 4. 量化业务复审意见

### 4.1 业务有效性：通过

022 修订后，业务目标更加稳健。它不是简单追求：

```text
更多 confirmed trend bar
```

而是同时要求：

```text
teacher_trend_coverage 上升；
confirmed_precision_vs_teacher 不低；
false_confirm_on_range_teacher 受控；
choppy_false_confirm_rate 受控；
交易层收益 / 回撤 / 交易次数不退化。
```

这符合量化研究中的基本原则：**提高信号召回必须同时约束误报，否则交易层可能变差。**

尤其是新增的业务底线：

```text
total_return 不低于 v021 full_bias 的 70%
max_drawdown 不劣化超过 0.2% 绝对值或 20% 相对值
trade_count 不超过基线 1.5×
```

可以防止模块指标好看但交易系统退化。

### 4.2 慢涨保护：通过，但需实现时特别关注

Chop Guard 加入 slow-trend 例外后，已经吸收初审意见。该设计能避免把 019 项目关注的低波动慢涨误判为横盘。

不过实际实现中要注意：当前 `engine.py` 的计算顺序是：

```text
TrendSignalProvider
CrashRegimeDetector
SlowUptrendDetector
TrendSegmentEngine
TrendBiasBuilder
```

如果 `TrendSignalProvider` 的 Chop Guard 直接依赖 `SlowUptrendDetector` 的输出，会形成计算顺序冲突。

建议实现时优先使用 `TrendSignalProvider` 内部可计算的 slow-trend 代理条件：

```text
EMA 48/72 slope；
persistence_above_ema_mid；
efficiency_48。
```

`SlowUptrendDetector.stable_slow_uptrend` 可作为评估或后续二次确认，不建议 Phase 1 直接作为 `TrendSignalProvider.compute()` 的输入。

## 5. 技术路径复审意见

### 5.1 与 021 架构契约一致：通过

修订版明确：

```text
不向 TrendBiasContext 新增 block_reason_* 字段；
阻断审计从 reason_codes / logger 派生；
WP-D 不修改 021 冻结 schema。
```

这解决了初审中最大的架构风险。022 作为“质量与性能修正项目”，不应重新打开 021 的核心 schema，当前处理方式合理。

### 5.2 分段性能路径：通过

修订版已补充：

```python
buffer_start_bar_idx: int

def abs_to_local(abs_idx: int) -> int:
    return abs_idx - buffer_start_bar_idx
```

并规定 active leg 仍在进行时其起点不得被淘汰，previous leg 只保留汇总指标。这可以避免 deque 化后索引错位。

实施建议：

```text
先实现增量腿指标；
再替换有界 buffer；
最后做 golden snapshot。
```

不要先上 deque 再追 bug。

### 5.3 Eval Harness 路径：通过

修订版已补报告元数据：

```text
symbol / interval / start / end
split
data_source / cache_path
teacher_label_version
teacher_params_sha256
config_sha256
run_timestamp
```

这能满足趋势模块评估的可复现要求。

建议补充一个小点：如果无法取得某天 Binance Vision 数据，报告中应把缺失日期写入 `missing_data_dates`，避免静默缩短评估窗口。

## 6. 实施前澄清项

以下 3 项不构成立项阻塞，但建议在 Phase 1 PR 描述或实现注释中明确。

### 澄清一：M1 指标需拆分 confirmed-only 与 sustained-inclusive

当前 M1 定义为：

```text
Teacher 确认趋势 bar 中被同向 confirmed/sustained 覆盖的比例
```

由于 `is_sustained` 门槛更低，把 confirmed 与 sustained 合在一个指标里，可能掩盖真正可开仓确认能力不足。

建议 Eval Harness 同时输出：

```text
teacher_coverage_confirmed_only
teacher_coverage_confirmed_or_sustained
```

其中：

```text
confirmed_only 用于开仓 / 升级质量；
confirmed_or_sustained 用于持仓维持质量。
```

M1 可以继续采用合并口径，但报告必须展示拆分结果。

### 澄清二：CHOP_SOFT 的 direction 是否可进入 micro_direction

修订版规定：

```text
CHOP_SOFT 不清空 direction；is_confirmed=false。
```

这合理。但进入 `TrendBiasBuilder` 后，`TrendSignal.direction` 仍可能影响 `micro_direction`，进而影响 `alignment_score_*`。

建议实现中增加 reason 或字段：

```text
CHOP_SOFT_MICRO_WEAK
```

并在 Builder 中降低 micro_direction 权重或只作为诊断，避免弱 chop 方向继续放大开仓偏置。

### 澄清三：Phase 1 是否改交易层

修订版说 Phase 1 交付 `TrendSignal`、Chop Guard 和 Eval Harness。建议明确：

```text
Phase 1 不改 RuleEngine 交易行为；
只允许新增字段、日志和评估；
交易联动回测只作为观察，不作为 Phase 1 必须达标项。
```

否则 Phase 1 同时改信号和交易层，回测变化会难以归因。

## 7. 复审结论

修订后的 022 立项文档已经吸收初审意见，业务目标更稳健，技术边界也更清晰。

复审结论：

```text
022 立项复审通过；
可进入 Phase 1 实施；
不需要再次进行立项层大评审。
```

Phase 1 建议范围：

```text
1. TrendSignal 增加 ConfirmTier / is_sustained；
2. 实现 CHOP_HARD / CHOP_SOFT；
3. 增加 slow-trend 代理例外；
4. 初版 eval_trend_modules.py；
5. 单测修复与新增慢涨回归用例。
```

进入 Phase 2 前，需要确认：

```text
M1 拆分指标已输出；
CHOP_SOFT 对 micro_direction 的影响已明确；
Phase 1 未混入交易规则改动。
```

最终上线仍需以 Phase 3 后的联动回测为准，遵守：

```text
模块指标达标 + 交易联动不退化 + 性能可日常运行
```
