# 022 趋势模块质量与性能修正立项评审报告

## 1. 评审说明

| 项 | 内容 |
|----|------|
| 角色 | 架构师 / 量化业务评审 |
| 流水号 | 022 |
| 评审对象 | `软件设计师-022-趋势模块质量与性能修正立项方案.md` |
| 评审维度 | 量化业务有效性、技术路径合理性、与 016~021 的架构一致性 |
| 参考材料 | 018/020/021 设计文档、021 回测报告、当前 `trading_system/` 趋势模块实现 |
| 评审结论 | **建议批准立项，但需补充若干业务验收与架构边界后进入实施** |

本文不重复 022 立项方案内容，重点判断：

```text
这个项目在量化业务上是否真正解决交易系统痛点；
技术路径是否能在不破坏 021 趋势偏置框架的前提下落地。
```

---

## 2. 总体结论

022 的立项方向 **业务上有效，技术上必要**。

当前系统已经通过 020/021 建立了趋势分段、趋势偏置和风险预算框架，但 021 回测仍暴露出明显瓶颈：

```text
趋势覆盖不足；
趋势升级仍少；
慢涨机会仍未转化；
分段引擎性能过慢；
bias 层对多头存在明显压制。
```

022 选择不继续新增交易规则，而是回到趋势模块本身做：

```text
质量校准 + 性能修正 + 模块级验收体系
```

这个方向是正确的。它能避免继续在交易层堆补丁，也能为 018/021 的持仓生命周期和趋势偏置提供更可信输入。

但当前方案仍有几个需要修正的风险点：

1. 部分指标可能鼓励“过度确认趋势”，需要增加趋势误报 / 精度类指标；
2. Teacher 标签不能作为单一真理，应固定版本并加入多窗口稳健性评估；
3. 硬阻断多空对称性不能只看比例，应按市场 regime / 机会分布归因；
4. `block_reason_long/short` 会改动 021 冻结字段契约，需要明确是扩展 schema 还是仅日志字段；
5. Chop Guard 可能误杀低波动慢涨趋势，需要增加 slow-trend 例外条件；
6. 分段性能优化涉及索引和历史窗口，必须先设计“绝对 bar index 与 bounded buffer”的映射。

因此评审结论为：

```text
022 可立项；
建议先做一次小幅立项修订，补齐业务验收和架构边界；
修订后可进入 Phase 1 实施。
```

## 3. 量化业务有效性评审

### 3.1 业务问题定位准确

022 把问题定义为：

```text
趋势不是完全看不见，而是确认太碎、记忆太短、分段太慢、偏置不平衡。
```

这个判断符合目前回测结果。021 full_bias 在 test OOS 上已经能降低回撤，但趋势侧仍存在：

| 指标 | 表现 | 业务含义 |
|------|------|----------|
| `trend_upgrade_count = 0` | 趋势识别未稳定转化为趋势仓 | 趋势模块没有真正驱动持仓生命周期 |
| `long_trend_capture_ratio = 0%` | 多头长趋势仍未捕捉 | 系统存在下跌补救强、多头趋势弱的问题 |
| `short_coverage_downtrend_ratio` 仍低 | 下跌趋势覆盖不足 | crash 能赚钱，但趋势腿利用率仍不充分 |
| `avg_bars_held` 偏短 | 长趋势收益吃不满 | 与 018 的核心目标仍冲突 |

022 不以 PnL 为唯一成功指标，而先看趋势模块质量，这是成熟量化研发流程中更合理的拆分。

**评审意见：通过。**

---

### 3.2 “方向与确认分离”在业务上合理

022 提出：

```text
direction 表示当前倾向；
is_confirmed 表示可驱动开仓 / 升级；
is_sustained 表示可维持已有趋势仓。
```

这个拆分非常关键。真实交易中，“能继续持有”和“能新开仓”不是同一个问题：

| 状态 | 交易含义 |
|------|----------|
| 方向倾向存在 | 可观察，可作为弱信号 |
| 趋势确认 | 可用于升级 / 顺势建仓 |
| 趋势维持 | 可用于不因正常 pullback 过早退出 |

这能直接解决当前趋势仓过早退出的问题。

**但必须加一条硬约束：**

```text
is_sustained 只能用于持仓维持 / runner 容忍；
不能直接用于新开仓或首次趋势升级。
```

否则系统可能把“低门槛维持信号”误用为“开仓确认信号”，导致震荡市假趋势增多。

---

### 3.3 Chop Guard 业务方向正确，但需防止误杀慢涨

震荡市误判方向是当前趋势模块的真实问题。引入 Chop Guard 合理，尤其是：

```text
range_atr 低；
EMA 频繁交叉；
efficiency 低。
```

这些都是量化上常见的横盘过滤条件。

但 019/020 已经证明：慢速上涨趋势往往具备：

```text
波动低；
单 bar 斜率低；
ATR 扩张不明显；
短窗口收益不强。
```

如果 Chop Guard 只看 `chop_range_atr < 3.0` 和 `abs(ret_24_atr) < 1.0`，可能把慢涨长段误判为 CHOP。

**建议补充 slow-trend 例外：**

```text
若满足以下任一条件，不触发 CHOP 强制 NONE：
1. 48/72 bar EMA slope 为正且持续；
2. persistence_above_ema_mid 高；
3. efficiency_48 高于阈值；
4. SlowUptrendDetector 输出 stable_slow_uptrend。
```

否则 022 可能修复震荡误判，却重新伤害 019 要解决的慢涨捕捉。

---

### 3.4 Teacher 覆盖指标有价值，但不能作为唯一业务真理

022 使用 020 Teacher 标签衡量：

```text
teacher_trend_coverage >= 65%
```

这是合理的模块级指标，因为 Teacher 代表 hindsight 主波段结构。

但 Teacher 标签本身来自规则和参数，不是市场真理。若只优化 coverage，容易出现：

```text
为了覆盖 Teacher 趋势，模块在更多 bar 上输出 confirmed；
趋势误报上升；
交易层逆向亏损增加。
```

**建议增加成对指标：**

| 指标 | 目的 |
|------|------|
| `teacher_trend_coverage` | 召回趋势 |
| `confirmed_precision_vs_teacher` | 防止乱确认 |
| `false_confirm_on_range_teacher` | 防止把 RANGE/TRANSITION 判为趋势 |
| `confirmed_direction_macro_f1` | 防止只优化 UP 或 DOWN |

建议补充达标线：

```text
confirmed_precision_vs_teacher >= 55%（初期）
false_confirm_on_range_teacher <= 20%
confirmed_direction_macro_f1 不低于 022 前基线
```

---

### 3.5 偏置平衡目标有效，但不能机械追求多空对称

022 发现：

```text
allow_open_long=False 578 bars
allow_open_short=False 130 bars
```

这说明系统对多头压制显著更强，业务上值得修正。

但 BTC OOS 区间本身可能处于明显下跌 regime。若市场真实下跌更多，开多被阻断多于开空是合理现象。直接要求：

```text
多空硬阻断比 < 2:1
```

可能在单边熊市中过度放松多头。

**建议改为分 regime 验收：**

```text
在 Teacher/Segment 判定为 RANGE 或 TRANSITION 的区间：
  多空 hard block 不应严重失衡；

在 confirmed DOWN leg：
  long block 高是合理的，但应由 SEGMENT/CRASH 解释；

在 confirmed UP leg：
  short block 应对称生效。
```

并增加：

```text
block_long_by_reason
block_short_by_reason
block_reason_entropy
```

让回测师能判断“多头被压制”到底来自 crash、legacy，还是 macro 派生过强。

## 4. 技术架构合理性评审

### 4.1 工作包拆分合理

022 的五个工作包边界清晰：

| WP | 技术评价 |
|----|----------|
| WP-A TrendSignal 记忆与确认 | 直接修复当前 `is_broken` 过高问题 |
| WP-B Chop Guard | 修复震荡误判，适合作为单测驱动 |
| WP-C TrendSegment 性能 | 必要，当前实现确有 O(n²) 风险 |
| WP-D TrendBias 再平衡 | 与 021 full_bias 直接衔接 |
| WP-E Eval Harness | 必要，避免靠主观图形调参 |

依赖顺序也合理：

```text
先修信号质量；
并行修分段性能；
再修 bias 平衡；
最后联动回测。
```

**评审意见：通过。**

---

### 4.2 WP-A 技术路径可行，但需保护向后兼容

当前 `TrendSignal` dataclass 没有：

```python
is_sustained
confirm_tier
```

022 建议新增字段。技术上可行，但会影响所有构造 `TrendSignal(...)` 的地方，尤其是早期 `min_len` 不足返回分支和测试 fixture。

**建议：**

1. 新字段必须有明确默认；
2. 先以可选字段加入，避免大面积破坏；
3. `is_confirmed` 语义不变，保持开仓/升级高门槛；
4. `is_sustained` 仅在 RuleEngine full-bias 持仓逻辑中启用。

建议定义：

```python
class ConfirmTier(str, Enum):
    NONE = "NONE"
    CANDIDATE = "CANDIDATE"
    CONFIRMED = "CONFIRMED"
    SUSTAINED = "SUSTAINED"
```

不要用裸字符串，避免日志与配置拼写漂移。

---

### 4.3 WP-B 应作为 TrendSignal 的前置守卫，而不是事后覆盖

文档写：

```text
投票前增加 chop 判定；
触发后 direction=NONE。
```

这个顺序合理。若先投票再覆盖，会导致 reason_codes 和分数含义混乱。

**建议实现顺序：**

```text
1. 计算 chop 所需的 range / flip / efficiency；
2. 若强 CHOP，直接返回 NONE；
3. 若弱 CHOP，只降低 score 或提高 confirmed_score；
4. 非 CHOP 再走原投票。
```

建议区分：

```text
CHOP_HARD：强制 NONE；
CHOP_SOFT：不清空方向，但禁止 confirmed。
```

这样可以减少对慢涨和早期趋势的误杀。

---

### 4.4 WP-C 性能优化必要，但当前方案需补索引设计

当前 `TrendSegmentEngine` 确实使用：

```python
self._close_hist: list[float] = []
self._high_hist: list[float] = []
self._low_hist: list[float] = []
self._atr_hist: list[float] = []
```

并且 `_leg_efficiency`、`_max_adverse_ratio` 会按腿区间循环。022 判断性能瓶颈合理。

但将历史改成 bounded `deque` 会引入一个架构风险：

```text
TrendLeg.start_bar_idx / end_bar_idx 是绝对 bar index；
deque 内部索引是滑动窗口相对 index。
```

如果不先设计映射，会导致：

- 腿指标取错窗口；
- active_leg 起点被 deque 淘汰；
- previous_leg 复盘字段丢失；
- golden snapshot 难以一致。

**建议补充设计：**

```python
buffer_start_bar_idx: int
abs_idx -> local_idx = abs_idx - buffer_start_bar_idx
```

并规定：

```text
active_leg 起点不得在腿仍 active 时被淘汰；
若 active_leg 超过 max buffer，应切换到增量指标，不再依赖历史切片；
previous_leg 仅保留汇总指标，不依赖 buffer。
```

022 的“增量腿指标”方向正确，应优先使用增量指标解决长腿问题，而不是仅靠 deque。

---

### 4.5 WP-D 存在 021 字段契约冲突

022 提出在 `TrendBiasContext` 新增：

```python
block_reason_long: list[str]
block_reason_short: list[str]
```

但 021 刚冻结 `TrendBiasContext` schema，并要求所有文档和代码采用冻结字段。022 又直接新增字段，会破坏 021 契约。

**建议二选一：**

方案 A：正式扩展 021 schema

```text
TrendBiasContext v2 增加 block_reason_long/short；
同步更新 021 字段契约、logger、A/B 验收。
```

方案 B：不改 context，只做评估/日志字段

```text
TrendBiasContext.reason_codes 继续承载原因；
eval_trend_modules.py 从 reason_codes 聚合 block_long_by_reason；
logger 输出 block_reason_long/short 作为派生字段，不进入核心 dataclass。
```

评审建议采用 **方案 B**。因为 022 目标是质量修正，不应轻易打开 021 冻结契约。

---

### 4.6 WP-D 不能把 legacy down hard block 直接降级为默认

022 建议：

```python
legacy_down_hard_block: bool = False
```

业务上方向合理，但技术上要确保 crash 与 confirmed down leg 仍能强约束多头。

建议规则：

```text
legacy down 不再单独 hard block；
但 confirmed DOWN leg + should_avoid_counter 仍可 hard block；
crash active 仍 hard block long；
macro DOWN 仅做 soft bias。
```

这应写成优先级表，否则实现时可能过度放开下跌中的逆势多头。

---

### 4.7 Eval Harness 是必要交付，但需固定数据与 Teacher 版本

022 提出：

```text
examples/eval_trend_modules.py
默认 BTCUSDT 1h 365d
对齐 020 Teacher 标签
```

这是正确方向。但为了可复现，需要补充：

1. 数据起止时间；
2. Binance Vision 数据延迟处理；
3. Teacher 标签版本和参数 hash；
4. 配置 hash；
5. OOS 切分规则；
6. 报告中记录 git commit 或文件摘要。

否则 022 的验收结果会因数据更新或 Teacher 参数变化而不可比较。

建议报告元数据包含：

```json

  "symbol": "BTCUSDT",
  "interval": "1h",
  "start": "...",
  "end": "...",
  "split": "test_oos_last_15pct",
  "teacher_label_version": "020_major_legs_vX",
  "config_path": "...",
  "config_sha256": "..."
}
```

## 5. 对 022 验收指标的修订建议

### 5.1 模块级指标建议改为“召回 + 精度 + 业务约束”

当前指标偏召回和性能：

```text
teacher_trend_coverage >= 65%
broken_ratio <= 45%
runtime <= 30s
hard_block_long_ratio <= 30%
```

建议改成：

| 类别 | 指标 | 建议 |
|------|------|------|
| 趋势召回 | `teacher_trend_coverage` | ≥ 65% |
| 趋势精度 | `confirmed_precision_vs_teacher` | ≥ 55% 初始线 |
| 横盘误报 | `false_confirm_on_range_teacher` | ≤ 20% |
| 震荡守卫 | `choppy_false_confirm_rate` | ≤ 5% |
| 破坏率 | `broken_ratio` | ≤ 45% |
| 性能 | `segment_runtime_8745` | ≤ 30s，本地；CI ≤ 60s |
| 多空偏置 | `hard_block_symmetry_by_regime` | 分 regime 统计，不只看总比值 |
| 交易联动 | `missed_confirmed_trend_bars` | 下降 ≥ 20% |

### 5.2 联动交易验收需要保留收益回撤底线

022 说 PnL 不是硬门槛，这合理。但交易系统最终服务业务，不能只看模块指标。

建议增加最低业务底线：

```text
若模块指标达标但：
  total_return 明显劣化；
  max_drawdown 超过基线；
  trade_count 异常上升；
则不得进入 prod 候选，只能继续作为模块实验分支。
```

建议：

```text
total_return 不低于 v021 full_bias 的 70%
max_drawdown 不劣化超过 0.2% 绝对值或 20% 相对值
trade_count 不超过基线 1.5x，除非收益/回撤显著改善
```

## 6. 需要补充的立项修订项

进入实施前，建议补充以下内容：

1. 明确 `is_sustained` 只能用于持仓维持，不能用于新开仓；
2. Chop Guard 增加慢涨例外规则，避免误杀 slow uptrend；
3. Teacher 评估增加 precision / false positive 指标；
4. hard block 平衡指标改为按 regime / leg_type 分层统计；
5. `block_reason_long/short` 不直接加入冻结 `TrendBiasContext`，优先从 `reason_codes` 派生；
6. TrendSegment bounded buffer 增加绝对索引到局部索引的设计；
7. `eval_trend_modules.py` 固定数据窗口、Teacher 版本、配置 hash；
8. 联动交易验收增加最低收益 / 回撤 / 交易次数底线。

## 7. 最终评审结论

022 在量化业务上是有效的：

```text
它瞄准的是趋势交易系统的输入质量，而不是继续在交易层堆规则。
```

022 在技术路径上总体合理：

```text
先修 TrendSignal，再修 TrendSegment 性能，再修 TrendBias 审计，最后建立评估 harness。
```

但为了避免“提高覆盖率但增加假趋势”“修复震荡但错杀慢涨”“审计字段破坏 021 契约”等问题，建议在批准实施前完成一次小修订。

评审意见：

```text
建议批准 022 立项；
建议以“有条件通过”进入实施准备；
完成 §6 修订项后，可进入 Phase 1。
```

建议项目经理将 022 的成功标准定义为：

```text
模块指标达标 + 交易联动不退化 + 性能可日常运行
```

而不是单独追求 PnL 或单独追求趋势覆盖率。
