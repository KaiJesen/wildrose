## 7. 问题分析与架构师待决事项

### 7.1 return_ic 与 cum_return_ic 的结构性 trade-off

现象：

- 训练全程 **valid return_ic 多为负**（约 -0.01 ~ -0.02），但 **test cum_return_ic 可达 0.13+**。
- 增大 `return_weight`（0.35 → 0.45）、降低 `direction_weight`、关闭 direction class weights、从 0060/0062c 不同起点重训，**test return_ic 最高仅 0.006**。

推测：

- `HorizonReturnHead` 与 `cum_return_pred` 主导了 trunk 表征，优化的是 **累计收益排序**，而非逐步收益排序。
- `return_consistency_weight` 已降至 0.01–0.02，仍不足以拉回 step IC。
- 这不是单次随机波动：0062c–f 可复现。

### 7.2 需架构师确认的问题

1. **验收指标是否调整**：是否接受以 `cum_return_ic` 替代 step `return_ic` 作为收益排序主指标？若坚持 `return_ic >= 0.035`，可能需要结构层方案（见 §8）。
2. **cum_direction_head 何时重启**：0062 阶段已证明 `sign(cum_return_pred)` 方向准确率可达 60%+；独立 `cum_direction_logit` head 在 0062e 上 test acc 为 60.8%，可暂缓强化。
3. **投产候选**：训练师建议 **`0062e`** 为当前最优折中；checkpoint 见 `recommended/market_state_best.pt`。

## 8. 若需继续提升 return_ic 的结构/训练建议（供参考）

架构师-008 §10 暂不建议加结构；但若 `return_ic` 仍为硬门槛，可考虑：

1. **双路径表征**：step return head 与 cum return head 使用不同 pooled 特征或中期 detach cum 分支。
2. **阶段性 detach**（已有 `--detach-risk-vol-after-epoch`，可对称增加 cum 分支 detach）。
3. **分阶段训练**：先冻结 cum head 恢复 step return_ic，再联合微调 cum。
4. **降低 cum_return_weight**（如 0.10–0.12）换取 step IC，需验证 cum_return_ic 是否仍 ≥ 0.08。

以上均未在本系列中验证，需架构师拍板后执行 **0063**。
